# 9.3 runner-multi.ts 多工作区隔离

## 9.3.1 现有多工作区模型

`runner-multi.ts`（位于 `mvp-runner/src/`）管理多个工作区的并发执行。每个工作区是独立的 Trae Chat 会话，有独立的 CDP 连接：

```typescript
// 简化的现有结构
class RunnerMulti {
  private workspaces: Map<string, WorkspaceRunner> = new Map();
  
  async start() {
    for (const [name, config] of this.workspaceConfigs) {
      const runner = new WorkspaceRunner(name, config);
      await runner.start();
      this.workspaces.set(name, runner);
    }
  }
}
```

每个 `WorkspaceRunner` 内部有自己的 `taskMachine` 和 `chatMutexMachine`。

## 9.3.2 独立性原则

心跳必须满足"**一个工作区的异常不影响另一个工作区**"。这意味着：

| 维度 | 要求 |
|------|------|
| Timer 独立 | PMCLI 的 Layer 1 Timer 和 DEVCLI 的 Layer 1 Timer 是两个 setInterval |
| 状态独立 | PMCLI 进入 model-stalled 不影响 DEVCLI 的状态判定 |
| 配置独立 | PMCLI 用 keep 策略，DEVCLI 用 delete 策略，互不干扰 |
| 通知独立 | PMCLI 的冷却期不影响 DEVCLI 的通知发送 |
| 日志独立 | PMCLI 写入 `workspaces/PMCLI/runs/heartbeat.jsonl`，DEVCLI 写各自的 |

## 9.3.3 改造点：WorkspaceRunner 持有独立 HeartbeatDaemon

每个 `WorkspaceRunner` 构造时创建自己的 `HeartbeatDaemon` 实例：

```typescript
// mvp-runner/src/workspace-runner.ts

class WorkspaceRunner {
  private name: string;
  private cdpClient: CdpClient;
  private chatMutex: Interpreter<typeof chatMutexMachine>;
  private heartbeatDaemon: HeartbeatDaemon;   // 新增
  
  constructor(name: string, config: WorkspaceConfig) {
    this.name = name;
    this.cdpClient = new CdpClient(config.cdpUrl);
    this.chatMutex = interpret(chatMutexMachine).start();
    
    // 心跳独立实例
    this.heartbeatDaemon = new HeartbeatDaemon({
      workspace: name,
      cdpClient: this.cdpClient,          // 共享 CDP 连接
      chatMutex: this.chatMutex,          // 共享互斥锁（用于协调）
      policy: config.heartbeatPolicy,     // 独立策略
      metricsPath: `./workspaces/${name}/runs/heartbeat.jsonl`,
    });
  }
  
  async start() {
    await this.cdpClient.connect();
    await this.heartbeatDaemon.start();
    // ... 原有逻辑
  }
}
```

**关键点**：心跳共享 CDP 连接和互斥锁（因为物理上确实是同一个 Chat），但策略、日志、Timer 全部独立。

## 9.3.4 共享资源的使用规则

虽然心跳独立，但几个资源是全局共享的，必须有使用规则：

### 共享资源一：飞书客户端

所有工作区共用一个 `LarkClient` 实例（避免多 WebSocket 连接）。通知发送时带上 workspace 标识，确保消息能区分来源。

### 共享资源二：Admin Server

整个 `runner-multi` 进程只启动一个 `HeartbeatAdminServer`（9090 端口），它内部持有所有 `HeartbeatDaemon` 的引用：

```typescript
const adminServer = new HeartbeatAdminServer(
  Array.from(runner.workspaces.values()).map(w => w.heartbeatDaemon)
);
```

Admin API 的 workspace 参数用来路由到具体的 Daemon 实例。

### 共享资源三：WikiBot 联动

所有工作区的告警汇聚到**同一个** `wiki/todo.md`（项目级），而不是每个工作区一个。这样 WikiBot 蒸馏时能看到跨工作区的模式（比如"所有 Trae CN 工作区在晚上都容易排队"）。

## 9.3.5 故障隔离验证

独立性必须可验证。在测试中人工触发 PMCLI 的 `task-interrupted`，同时观察 DEVCLI：

| 验证点 | 期望结果 |
|-------|---------|
| DEVCLI 的心跳 Timer | 继续正常跑 |
| DEVCLI 的日志文件 | 不会被污染 |
| DEVCLI 的告警 | 不会因 PMCLI 的问题延迟 |
| DEVCLI 的恢复动作 | 不会被 PMCLI 的 mutex 阻塞 |

这些验证点写入 Part 10 的测试用例，作为**集成测试的必过项**。
