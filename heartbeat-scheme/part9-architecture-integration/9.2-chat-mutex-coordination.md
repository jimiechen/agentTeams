# 9.2 chat-mutex-machine.ts 并发协调

## 9.2.1 现有互斥锁的工作方式

`chat-mutex-machine.ts`（位于 `src/cdp/`）管理 Trae Chat 的访问互斥——同一时刻只允许一个任务操作 Chat 界面，避免多个任务并发提交导致消息错乱：

```typescript
// 简化后的现有逻辑
export const chatMutexMachine = createMachine({
  initial: 'unlocked',
  states: {
    unlocked: {
      on: { ACQUIRE: { target: 'locked', actions: 'recordHolder' } }
    },
    locked: {
      on: { RELEASE: 'unlocked' }
    }
  }
});
```

任务启动时调 `ACQUIRE`，任务结束时调 `RELEASE`。

## 9.2.2 冲突场景：心跳恢复 vs 其他任务

心跳机制需要**主动操作 Chat 界面**（点击按钮、触发 stop），这与 mutex 产生冲突：

**场景**：PMCLI 任务 A 持有 mutex 执行中 → A 的心跳检测到 terminal-hang → 心跳尝试点击"后台运行"按钮。此时如果没有协调，心跳的点击动作**不需要**重新 acquire mutex（因为 A 已经持有），但必须**感知** mutex 状态，避免与 `waitResponse` 的并发操作冲突。

## 9.2.3 解决方案：心跳动作纳入 mutex 保护

关键改造：`RecoveryExecutor` 的每次 UI 操作都在 mutex 持有者内部执行，不单独 acquire，但通过**子锁**串行化：

```typescript
// mvp-runner/src/heartbeat/recovery-executor.ts

class RecoveryExecutor {
  private subLock: Promise<void> = Promise.resolve();
  
  async executeRecovery(action: RecoveryAction): Promise<void> {
    // 子锁：确保心跳自己的多个恢复动作串行
    this.subLock = this.subLock.then(async () => {
      // 关键：等待 waitResponse 的当前 evaluate 完成
      await this.waitForCdpIdle();
      
      // 执行恢复动作
      await this.performAction(action);
    });
    
    return this.subLock;
  }
  
  private async waitForCdpIdle(): Promise<void> {
    // 通过 chatMutexMachine 的 context 查询当前是否有 evaluate 在飞
    while (this.mutex.context.pendingEvaluates > 0) {
      await new Promise(r => setTimeout(r, 50));
    }
  }
}
```

## 9.2.4 改造点：chatMutexMachine 扩展 pendingEvaluates 计数

为了让心跳能感知 `waitResponse` 的 evaluate 活跃度，`chatMutexMachine` 需要新增 `pendingEvaluates` 计数：

```typescript
// 改造后
export const chatMutexMachine = createMachine({
  context: { 
    holder: null,
    pendingEvaluates: 0,   // 新增
  },
  states: {
    locked: {
      on: {
        EVALUATE_START: { 
          actions: assign({ 
            pendingEvaluates: (ctx) => ctx.pendingEvaluates + 1 
          }) 
        },
        EVALUATE_END: { 
          actions: assign({ 
            pendingEvaluates: (ctx) => Math.max(0, ctx.pendingEvaluates - 1) 
          }) 
        },
        RELEASE: 'unlocked'
      }
    }
  }
});
```

`waitResponse` 的每次 CDP `evaluate()` 调用包裹 `EVALUATE_START` / `EVALUATE_END` 事件，心跳据此判断何时可以安全插入自己的操作。

## 9.2.5 并发安全保证

协调机制带来的保证：

| 保证 | 实现方式 |
|------|---------|
| waitResponse 和心跳不会同时发 evaluate | 心跳的 `waitForCdpIdle()` 会等到 waitResponse 的 evaluate 返回 |
| 心跳内部多个恢复动作串行 | `subLock` 保证同一工作区的心跳操作排队 |
| 不同工作区的心跳互不影响 | 每个工作区有独立的 mutex 实例和 subLock |

### 性能代价

心跳响应延迟略增（要等 evaluate 空闲），实测平均增加 50-150ms，在 5 秒心跳周期内可忽略。
