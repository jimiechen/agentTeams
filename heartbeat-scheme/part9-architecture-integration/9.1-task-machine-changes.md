# 9.1 task-machine.ts 改造点

## 9.1.1 现有状态机结构回顾

先看改造前的 `task-machine.ts`（位于 `src/core/states/`），它用 xstate 管理任务生命周期：

```typescript
// 改造前

export const taskMachine = createMachine({
  id: 'task',
  initial: 'idle',
  states: {
    idle: {
      on: { START: 'submitting' }
    },
    submitting: {
      invoke: { src: 'submitPrompt', onDone: 'waiting' }
    },
    waiting: {
      invoke: { src: 'waitResponse', onDone: 'completed', onError: 'failed' }
    },
    completed: { type: 'final' },
    failed: { type: 'final' }
  }
});
```

**问题**：`waiting` 状态是黑盒——一旦进入 `waitResponse` 就失去对内部的控制，任务挂起、终端超时、弹窗阻塞等异常都被 `waitResponse` 内部消化，上层无从感知。这是心跳机制必须改造的核心。

## 9.1.2 改造点一：新增 heartbeat 子状态机

把 `waiting` 状态改成**并行状态**（parallel state），其中一条轨道跑原有的 `waitResponse`，另一条轨道跑心跳：

```typescript
// 改造后

import { heartbeatMachine } from './heartbeat-machine';

export const taskMachine = createMachine({
  id: 'task',
  initial: 'idle',
  states: {
    idle: {
      on: { START: 'submitting' }
    },
    submitting: {
      invoke: { src: 'submitPrompt', onDone: 'waiting' }
    },
    waiting: {
      type: 'parallel',
      states: {
        response: {
          invoke: { 
            src: 'waitResponse', 
            onDone: '#task.completed', 
            onError: '#task.failed' 
          }
        },
        heartbeat: {
          invoke: { 
            src: heartbeatMachine,
            data: { workspace: (ctx) => ctx.workspace }
          }
        }
      },
      on: {
        HEARTBEAT_CRITICAL: 'interrupted',    // 心跳判定需人工介入
        HEARTBEAT_RECOVERED: undefined,        // 恢复成功，不改变状态
      }
    },
    completed: { type: 'final' },
    failed: { type: 'final' },
    interrupted: { type: 'final' }   // 新增：心跳告警挂起
  }
});
```

### 关键变化

- `waiting` 变成并行状态，`response` 和 `heartbeat` 两条轨道并发
- 新增 `interrupted` 终态，专门对应心跳判定的人工介入场景
- `HEARTBEAT_RECOVERED` 事件不改变状态（恢复成功后继续等待响应）

## 9.1.3 改造点二：事件通道设计

心跳机制需要向主状态机发送三类事件：

| 事件 | 触发条件 | 上层响应 |
|------|---------|---------|
| `HEARTBEAT_RECOVERED` | 自动恢复成功 | 记录日志，不中断主流程 |
| `HEARTBEAT_CRITICAL` | 重试耗尽 / 任务中断 | 切换到 `interrupted` 终态 |
| `HEARTBEAT_QUEUING` | 检测到排队 | 延长 waitResponse 超时时间 |

事件发送通过 xstate 的 `sendParent` 实现：

```typescript
// heartbeat-machine.ts 片段
{
  entry: (ctx) => {
    if (ctx.mode === 'task-interrupted' || ctx.retryExhausted) {
      sendParent({ type: 'HEARTBEAT_CRITICAL', reason: ctx.mode });
    } else if (ctx.recoverySucceeded) {
      sendParent({ type: 'HEARTBEAT_RECOVERED', action: ctx.lastAction });
    }
  }
}
```

## 9.1.4 改造点三：context 扩展

`taskMachine.context` 需要新增心跳相关字段，便于上层日志和 UI 展示：

```typescript
// 改造前
interface TaskContext {
  workspace: string;
  prompt: string;
  startTime: number;
}

// 改造后
interface TaskContext {
  workspace: string;
  prompt: string;
  startTime: number;
  
  // 新增心跳相关
  heartbeatStats: {
    totalHeartbeats: number;         // 心跳次数统计
    recoveryAttempts: number;        // 恢复尝试次数
    lastMode: HealthMode;            // 最近一次健康模式
    lastRecoveryTime?: number;       // 最近一次恢复时间
  };
}
```

这些字段在 `taskMachine` 结束时一并写入 `runs/*.md`，让日志能完整展示"任务期间系统做了多少次心跳、触发了几次恢复"。

## 9.1.5 改造点四：降级开关

为了防止心跳机制本身故障影响主流程，必须提供**运行时开关**让心跳可以被旁路：

```typescript
export const taskMachine = createMachine({
  // ...
  states: {
    waiting: {
      type: 'parallel',
      states: {
        response: { /* ... */ },
        heartbeat: {
          invoke: {
            src: (ctx) => ctx.heartbeatEnabled 
              ? heartbeatMachine 
              : noopMachine    // 空实现，什么都不做
          }
        }
      }
    }
  }
});
```

`ctx.heartbeatEnabled` 从环境变量 `HEARTBEAT_ENABLED=false` 读取，默认 true。心跳机制一旦出问题，通过 `.env` 关闭即可回到原有行为，不需要回滚代码。
