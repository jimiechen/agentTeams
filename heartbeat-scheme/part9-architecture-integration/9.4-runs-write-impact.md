# 9.4 runs/*.md 写入逻辑影响

## 9.4.1 现有写入逻辑

当前 `runs/*.md` 由 `waitResponse` 完成后一次性写入，文件名格式 `{timestamp}.md`，内容包含用户 prompt 和 AI 响应：

```markdown
# Task Run: 2026-04-29T10-15-00

## User Prompt
...

## AI Response  
...

## Metadata
- Duration: 45s
- Workspace: PMCLI
```

**问题**：这是"成功完成"的视角，如果任务被心跳打断（`interrupted` 终态），现有逻辑不会写任何文件，导致**心跳介入的任务没有执行记录**。

## 9.4.2 改造点一：写入时机扩展

三个终态都要写 `runs/*.md`，不只是 `completed`：

```typescript
// mvp-runner/src/run-writer.ts

class RunWriter {
  async writeOnFinal(
    status: 'completed' | 'failed' | 'interrupted',
    ctx: TaskContext
  ): Promise<void> {
    const filename = `${ctx.startTime}-${status}.md`;
    const content = this.buildContent(status, ctx);
    writeFileSync(this.pathFor(filename), content);
  }
  
  private buildContent(status: string, ctx: TaskContext): string {
    return `# Task Run: ${ctx.startTime}

**Status**: ${status}
**Workspace**: ${ctx.workspace}
**Duration**: ${(Date.now() - ctx.startTime) / 1000}s

## User Prompt
${ctx.prompt}

## AI Response
${ctx.response ?? 'No response captured'}

## Heartbeat Stats
- Total heartbeats: ${ctx.heartbeatStats.totalHeartbeats}
- Recovery attempts: ${ctx.heartbeatStats.recoveryAttempts}
- Last mode: ${ctx.heartbeatStats.lastMode}

${status === 'interrupted' ? this.buildInterruptionSection(ctx) : ''}
`;
  }
  
  private buildInterruptionSection(ctx: TaskContext): string {
    return `

## Interruption Details

This task was interrupted by the heartbeat mechanism.

**Reason**: ${ctx.heartbeatStats.lastMode}
**Last detected mode**: ${ctx.heartbeatStats.lastMode}
**Diagnostic log**: \`workspaces/${ctx.workspace}/runs/heartbeat.jsonl\`

See heartbeat log for full signal history.
`;
  }
}
```

## 9.4.3 改造点二：心跳元信息嵌入

所有 `runs/*.md` 都追加心跳统计段——即使任务正常完成，也让用户知道"这次任务期间系统做了几次心跳、是否触发过恢复"。这是**可观测性的基础**。

### 正常完成的任务示例

```markdown
## Heartbeat Stats
- Total heartbeats: 47 (Layer1: 36, Layer2: 9, Layer3: 2)
- Recovery attempts: 0
- Last mode: healthy
- Uptime with recovery: 100%
```

### 被打断的任务示例

```markdown
## Heartbeat Stats
- Total heartbeats: 23 (Layer1: 18, Layer2: 4, Layer3: 1)
- Recovery attempts: 3 (terminal-hang: 2, model-stalled: 1)
- Last mode: task-interrupted
- Uptime with recovery: 67%

## Interruption Details

This task was interrupted by the heartbeat mechanism.

**Reason**: model-stalled retry exhausted (3/3)
**Last detected mode**: task-interrupted
**Diagnostic log**: `workspaces/PMCLI/runs/heartbeat.jsonl`

See heartbeat log for full signal history.
```

## 9.4.4 改造点三：双文件协同

当前方案：

| 文件 | 视角 | 内容 |
|------|------|------|
| `runs/{timestamp}-{status}.md` | 任务视角 | 每个任务一个文件，包含 prompt + response + 心跳摘要 |
| `runs/heartbeat.jsonl` | 心跳视角 | 按时间 append，包含所有心跳信号 |

两个文件的联动通过**时间戳 + workspace** 对齐。

### 查问题的典型路径

```
1. 看 runs/{timestamp}-interrupted.md 知道"这次任务被打断了，原因是 model-stalled"
2. grep heartbeat.jsonl 对应时间窗口，看"心跳在这个任务期间的完整信号流"
3. 定位到具体心跳事件，看原始 DOM 信号（lastSignal 字段）
```

这个工作流在 Part 10 的"排查场景测试"里会用具体案例验证。

## 9.4.5 对现有下游消费的兼容性

现有的 WikiBot 蒸馏、周报生成等下游流程都读取 `runs/*.md`。改造必须保证**向后兼容**：

### 兼容性检查清单

| 变化点 | 影响范围 | 兼容方案 |
|-------|---------|---------|
| 文件名变化 | 从 `{timestamp}.md` 改为 `{timestamp}-{status}.md` | glob 模式 `runs/*.md` 不受影响；精确正则需要更新 |
| 新增字段 | `## Heartbeat Stats` 和 `## Interruption Details` | 不影响已有 section 的解析 |
| 语义变化 | `interrupted` 是新的状态值 | 下游状态聚合统计需把 `interrupted` 视为"非成功完成" |

### 涉及的下游代码

改造落地前需要搜索所有消费 `runs/*.md` 的代码，确认兼容性：

| 文件 | 改动量 |
|------|-------|
| `lark-monitor.js` | < 10 行 |
| `wiki-distiller` | < 10 行 |
| `gen-overview.js` | < 10 行 |
