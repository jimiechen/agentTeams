# 1.3 心跳机制的三条核心设计原则

## 原则一：分层采样

按"发现时效性 × 采集成本"矩阵，把心跳分成三个频率层：

| 层级 | 频率 | 采集内容 | 成本 | 时效性 |
|------|------|---------|------|--------|
| Layer 1 | 5秒 | 按钮存在性、任务状态 | 低 | 高（5秒内） |
| Layer 2 | 15秒 | 文本增量、排队识别 | 中 | 中（30秒内） |
| Layer 3 | 60秒 | 深度指标、健康度 | 高 | 低（可延迟） |

**设计意图**：
- 高频层只采轻量信号（按钮是否存在）
- 低频层才做重量级采集（完整按钮快照、token 估算）
- 避免资源浪费，同时保证关键异常的及时发现

## 原则二：有限自治

心跳只在**白名单内的异常模式**上执行自动恢复动作，且每种动作有**重试上限**。

### 自治边界
- ✅ **允许自动恢复**：terminal-hang、modal-blocking、model-stalled
- ❌ **禁止自动恢复**：model-queuing（等待即可）、task-interrupted（终态）

### 重试上限
- terminal-hang：最多重试 2 次
- modal-blocking：最多重试 1 次（弹窗通常只出现一次）
- model-stalled：最多重试 3 次

### 与 Watchdog 的区别

| 特性 | Watchdog | 心跳机制 |
|------|----------|---------|
| 目标 | 追求自愈率 | 追求可预测性 |
| 边界 | 尽可能恢复 | 只恢复白名单内场景 |
| 失败处理 | 继续尝试 | 立即转人工告警 |

**核心差异**：心跳机制追求"做什么都可控"，而不是"尽可能救活"。

## 原则三：可观测优先于可恢复

任何一次心跳决策都必须满足：

### 可追溯性
- 判定为某模式 → 记录判定依据（原始信号）
- 执行某动作 → 记录动作类型、时间、结果
- 写入 `runs/heartbeat.jsonl`，包含完整上下文

### 排查效率
- 人工排查时必须能在 **10 秒内**看清：
  - 系统看到了什么（原始信号）
  - 系统做了什么（执行的动作）
  - 为什么失败（错误信息）

### 日志格式要求
```json
{
  "timestamp": "2024-04-29T10:00:00.000Z",
  "workspace": "PMCLI",
  "mode": "terminal-hang",
  "signals": {
    "hasBackgroundButton": true,
    "taskStatus": "running"
  },
  "action": {
    "type": "click_button",
    "buttonLabel": "后台运行",
    "success": true
  }
}
```

---

## 原则总结

```
分层采样 ── 解决资源与时效的矛盾
有限自治 ── 保证行为可预测、可控制
可观测优先 ── 出问题时能快速定位根因
```
