# 1.4 与现有脚本能力的复用关系

## 复用关系总览

`BUTTON_SCRIPTS_README.md` 里的四个检测函数是心跳机制的**底层能力**，不需要重写：

| 现有脚本 | 心跳机制中的角色 | 使用层级 |
|---------|----------------|---------|
| `checkTaskStatus()` | Layer 1 采集项之一 | Layer 1 |
| `checkTerminalTimeout()` | Layer 1 采集项之一 | Layer 1 |
| `checkModelResponse()` | Layer 2 采集项之一 | Layer 2 |
| `checkUserPrompt()` | Layer 1 采集项之一 | Layer 1 |

## 能力对比

### 现有脚本特征
- **独立使用**：每次手动调用一个函数
- **单一职责**：每个函数只检测一种状态
- **即时返回**：调用后立即返回结果
- **无记忆**：不保存历史状态

### 心跳机制特征
- **组合使用**：把多个检测函数组合成分层采样管道
- **状态记忆**：保存历史采样数据用于增量分析
- **模式判定**：在上层增加模式判定逻辑
- **自动恢复**：根据判定结果执行恢复动作

## 复用方式

### Layer 1 复用
```
Layer 1 (5秒)
    │
    ├── checkTaskStatus()    ──→ 任务状态文本
    ├── checkTerminalTimeout() ──→ 后台运行/取消按钮
    └── checkUserPrompt()    ──→ 保留/删除按钮
```

### Layer 2 复用
```
Layer 2 (15秒)
    │
    └── checkModelResponse() ──→ 排队状态、文本内容
```

## 改造点

### 现有脚本的问题
- 返回结构不一致
- 没有统一的错误处理
- 无法批量调用

### 改造策略
- **封装适配层**：统一返回结构
- **批量执行**：一次 CDP evaluate 返回所有 Layer 1 信号
- **错误处理**：统一的异常捕获和日志记录

## 复用收益

| 收益类型 | 量化效果 |
|---------|---------|
| 代码复用率 | 80% 的底层检测逻辑可复用 |
| 开发时间 | 节省约 2 天开发时间 |
| 测试覆盖 | 继承现有测试用例 |
| 维护成本 | 单点维护，多处受益 |
