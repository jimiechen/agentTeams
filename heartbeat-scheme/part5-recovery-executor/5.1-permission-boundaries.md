# 5.1 RecoveryExecutor 的权限边界

## 设计原则

恢复执行器的权限设计遵循**最小特权原则**——只能点击白名单内的按钮，且每次恢复都必须有明确的模式触发。

## 类定义

```typescript
class RecoveryExecutor {
  constructor(cdp, lark, policy, notifyChatId);

  // 根据信号中的 mode 分派到对应的恢复动作
  recover(signal: HeartbeatSignal): Promise<RecoveryResult>;

  // 每个任务开始时重置重试计数
  reset(): void;
}

interface RecoveryResult {
  attempted: boolean;    // 是否尝试了恢复
  action: string;        // 具体动作名（click:后台运行 / stop+retry / alert-only）
  success: boolean;      // 恢复是否成功执行（点击成功，不等同于任务恢复）
  error?: string;
}
```

## 按钮白名单

**硬编码在代码中，不读配置**：

| 允许点击 | aria-label 匹配 | 触发模式 |
|---------|---------------|---------|
| 后台运行 | 含"后台" | terminal-hang |
| 取消（终端） | 等于"取消" | terminal-hang（策略开关） |
| 保留 | 含"保留" | modal-blocking |
| 删除 | 含"删除" | modal-blocking（策略开关） |
| 停止生成 | 含"停止" 或 含"Stop" | model-stalled |
| 重试 | 含"重试" 或 含"Retry" | model-stalled |

## 禁止操作

**明确禁止点击**：
- 复制按钮
- 插入到光标处按钮
- 添加到新文件按钮
- 赞/踩按钮
- 其他内容操作按钮

这些由业务逻辑主动调用，不属于恢复范畴。

## 权限检查流程

```
收到 recover 请求
    │
    ▼
检查 mode 是否在允许恢复的列表中
    │
    ├─→ terminal-hang / modal-blocking / model-stalled → 继续
    └─→ model-queuing / task-interrupted → 返回 { attempted: false }
    │
    ▼
根据 mode 确定允许点击的按钮列表
    │
    ▼
检查按钮是否在白名单内
    │
    ├─→ 在白名单 → 执行点击
    └─→ 不在白名单 → 返回错误
    │
    ▼
记录恢复动作到日志
```

## 安全保障

### 多层防护
1. **模式防护**：只有特定模式才能触发恢复
2. **白名单防护**：只有白名单内的按钮才能点击
3. **策略防护**：不同工作区有不同的策略配置
4. **重试限制**：每种模式有重试次数上限

### 审计日志
每次恢复动作都记录到 `heartbeat.jsonl`，包含：
- 时间戳
- 工作区
- 模式
- 动作类型
- 成功/失败状态
