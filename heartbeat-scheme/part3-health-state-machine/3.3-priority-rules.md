# 3.3 模式判定的优先级规则

## 优先级总览

同一次心跳可能采集到多个异常信号，必须按固定优先级裁决：

```
优先级从高到低：

1. task-interrupted   （终态优先，不浪费恢复资源）
2. modal-blocking     （弹窗挡住一切交互，必须先处理）
3. terminal-hang      （终端不释放，模型也无法继续）
4. model-queuing      （排队是服务端问题，不要误判为停滞）
5. model-stalled      （排除前面几种后才能判停滞）
6. completed          （正常完成）
7. healthy            （正常执行中）
8. unknown            （信号缺失）
```

## 优先级规则详解

### 规则 1：终态优先

**原理**：task-interrupted 和 completed 是终态，无需继续检测其他信号。

```javascript
// 伪代码
if (taskStatus === 'interrupted') return 'task-interrupted';
if (taskStatus === 'completed') return 'completed';
```

**原因**：
- 避免在终态上浪费恢复资源
- 简化判定逻辑
- 保证状态一致性

### 规则 2：弹窗优先

**原理**：modal-blocking（弹窗阻塞）会阻止所有后续交互，必须优先处理。

```javascript
// 伪代码
if (hasKeepButton || hasDeleteButton) return 'modal-blocking';
```

**场景**：任务完成后出现"保留/删除"弹窗，如果不处理，后续的复制操作都无法进行。

### 规则 3：终端优先

**原理**：terminal-hang（终端挂起）会占用资源，阻止模型继续执行。

```javascript
// 伪代码
if (hasBackgroundButton) return 'terminal-hang';
```

**场景**：终端执行超时出现"后台运行"按钮，如果不点击，模型会一直等待终端完成。

### 规则 4：排队优先于停滞

**原理**：model-queuing 和 model-stalled 的征兆都是"文本停止增长"，但排队是正常现象，不应误判为停滞。

```javascript
// 伪代码
if (isQueuing) return 'model-queuing';  // 先判排队
if (stalledSeconds >= 30) return 'model-stalled';  // 再判停滞
```

**场景**：模型显示"排在第 5 位"，此时文本停止增长是正常的排队等待，不是停滞。

### 规则 5：停滞最后判定

**原理**：model-stalled 是隐式异常，需要排除所有其他可能性后才能判定。

**场景**：只有在排除了排队、终端挂起、弹窗阻塞后，连续 30 秒无增长才能判定为停滞。

## 多信号冲突处理

### 冲突场景示例

**场景 1**：同时检测到"后台运行"按钮和"排在第 5 位"文本

```
判定结果：terminal-hang（终端优先级高于排队）
原因：终端挂起会阻止任务继续，必须先处理
```

**场景 2**：同时检测到"保留"按钮和"后台运行"按钮

```
判定结果：modal-blocking（弹窗优先级高于终端）
原因：弹窗会阻止一切交互，必须先处理
```

**场景 3**：同时检测到"中断"状态和"后台运行"按钮

```
判定结果：task-interrupted（终态优先）
原因：任务已中断，无需处理按钮
```

## 优先级规则总结

| 优先级 | 模式 | 判定条件 | 处理策略 |
|-------|------|---------|---------|
| 1 | task-interrupted | 任务状态为"中断" | 告警，停止心跳 |
| 2 | modal-blocking | 出现"保留/删除"按钮 | 按策略点击按钮 |
| 3 | terminal-hang | 出现"后台运行"按钮 | 点击"后台运行" |
| 4 | model-queuing | 文本含"排在第 X 位" | 被动等待 |
| 5 | model-stalled | 连续 30s 无增长且非排队 | stop + retry |
| 6 | completed | 任务状态为"完成" | 停止心跳 |
| 7 | healthy | 无异常信号 | 持续监控 |
| 8 | unknown | 信号缺失或过期 | 跳过本轮 |

## 规则设计原理

### 为什么弹窗优先于终端？
- 弹窗会阻止所有交互，包括点击"后台运行"按钮
- 必须先处理弹窗，才能进行其他操作

### 为什么终端优先于排队？
- 终端挂起是本地问题，排队是服务端问题
- 终端不释放，即使不排队，模型也无法继续
- 处理终端是解决问题的第一步

### 为什么排队优先于停滞？
- 排队是正常现象，停滞是异常现象
- 误判排队为停滞会导致不必要的重试
- 重试会浪费资源，甚至影响其他任务
