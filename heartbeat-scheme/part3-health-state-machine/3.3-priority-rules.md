# 3.3 模式判定优先级规则

## 3.3.1 优先级设计原则

当多个检测器同时返回信号时，需要**优先级裁决**确定最终模式。设计原则：

1. **终态优先**：`task-interrupted` > 其他所有模式
2. **显式异常优先**：`terminal-hang` / `modal-blocking` > 隐式异常
3. **可恢复优先**：`model-stalled` > `unknown`
4. **正常状态最低**：`healthy` / `completed` 优先级最低

## 3.3.2 优先级层级表

| 优先级 | 模式 | 判定来源 | 说明 |
|-------|------|---------|------|
| 1（最高） | `task-interrupted` | Layer 1 | 终态，立即告警 |
| 2 | `modal-blocking` | Layer 3 | 显式 UI 阻塞 |
| 3 | `terminal-hang` | Layer 3 | 显式 UI 阻塞 |
| 4 | `model-queuing` | Layer 2 | 有明确排队提示 |
| 5 | `model-stalled` | Layer 2 | 隐式异常，需恢复 |
| 6 | `completed` | Layer 1 | 正常终态 |
| 7 | `healthy` | 默认 | 正常运行 |
| 8（最低） | `unknown` | 兜底 | 无法判断 |

## 3.3.3 裁决算法

```typescript
function resolveMode(signals: DetectedSignals): HealthMode {
  const { taskStatus, charsDelta, domSignals, lastMessage } = signals;
  
  // 优先级 1: task-interrupted（终态）
  if (taskStatus === 'interrupted' || taskStatus === 'cancelled') {
    return 'task-interrupted';
  }
  
  // 优先级 2: modal-blocking
  if (domSignals.modal?.exists && domSignals.modal?.isBlocking) {
    return 'modal-blocking';
  }
  
  // 优先级 3: terminal-hang
  if (domSignals.backgroundRunButton?.exists) {
    return 'terminal-hang';
  }
  
  // 优先级 4: model-queuing
  if (lastMessage.includes('排在第') || lastMessage.includes('排队')) {
    return 'model-queuing';
  }
  
  // 优先级 5: model-stalled
  if (charsDelta === 0 && signals.stallDuration >= 30_000) {
    return 'model-stalled';
  }
  
  // 优先级 6: completed
  if (taskStatus === 'completed') {
    return 'completed';
  }
  
  // 优先级 7: healthy
  if (charsDelta > 0 || signals.stallDuration < 15_000) {
    return 'healthy';
  }
  
  // 优先级 8: unknown（兜底）
  return 'unknown';
}
```

## 3.3.4 冲突场景裁决

### 3.3.4.1 场景一：排队 + 弹窗同时出现

**信号**：
- Layer 2 检测到"排在第 5 位"
- Layer 3 检测到模态弹窗

**裁决**：`modal-blocking`（优先级 2 > 优先级 4）

**理由**：弹窗需要立即处理，否则任务无法继续。排队提示在处理弹窗后可能已失效。

### 3.3.4.2 场景二：停滞 + 后台运行按钮同时出现

**信号**：
- Layer 2 检测到 30 秒无文本增长
- Layer 3 检测到"后台运行"按钮

**裁决**：`terminal-hang`（优先级 3 > 优先级 5）

**理由**：后台运行按钮是显式异常，有明确的恢复路径。先处理终端阻塞，再观察 AI 响应。

### 3.3.4.3 场景三：中断 + 弹窗同时出现

**信号**：
- Layer 1 检测到任务状态为"中断"
- Layer 3 检测到模态弹窗

**裁决**：`task-interrupted`（优先级 1 > 优先级 2）

**理由**：中断是终态，任务已无法继续，处理弹窗无意义。

### 3.3.4.4 场景四：排队 + 停滞信号冲突

**信号**：
- Layer 2 检测到 30 秒无文本增长
- Layer 2 同时检测到"排在第 3 位"

**裁决**：`model-queuing`（优先级 4 > 优先级 5）

**理由**：排队是正常等待，不是异常。优先按排队处理，延长超时。

## 3.3.6 优先级配置化

优先级规则支持通过 Admin API 动态调整：

```typescript
interface PriorityConfig {
  task_interrupted: number;   // 默认 1
  modal_blocking: number;      // 默认 2
  terminal_hang: number;       // 默认 3
  model_queuing: number;       // 默认 4
  model_stalled: number;       // 默认 5
  completed: number;           // 默认 6
  healthy: number;             // 默认 7
  unknown: number;             // 默认 8
}
```

**使用场景**：
- 测试环境可调整优先级验证特定路径
- 生产环境发现优先级不合理时可热修复

## 3.3.7 裁决日志

每次裁决记录详细日志，用于问题排查：

```typescript
interface ResolutionLog {
  timestamp: string;
  signals: DetectedSignals;
  candidates: Array<{
    mode: HealthMode;
    priority: number;
    reason: string;
  }>;
  winner: HealthMode;
  resolutionTimeMs: number;
}
```

日志写入 `logs/heartbeat-resolution-{date}.jsonl`。
