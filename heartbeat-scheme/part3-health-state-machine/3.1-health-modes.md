# 3.1 七种健康模式定义

## 3.1.1 模式概述

心跳机制的核心是**健康状态分类**。通过多层检测器收集的信号，最终归类为 7 种模式之一：

| 模式 | 含义 | 严重程度 | 是否需要恢复 |
|------|------|---------|-------------|
| `healthy` | 任务正常运行 | 无 | 否 |
| `completed` | 任务已完成 | 无 | 否 |
| `model-stalled` | AI 响应停滞 | 中 | 是 |
| `terminal-hang` | 终端命令阻塞 | 中 | 是 |
| `modal-blocking` | 弹窗等待用户输入 | 中 | 是 |
| `model-queuing` | AI 排队中 | 低 | 否 |
| `task-interrupted` | 任务被中断 | 高 | 否（终态） |
| `unknown` | 无法判断 | 待定 | 否 |

## 3.1.2 healthy 模式

**定义**：任务处于活跃运行状态，AI 正在响应或刚响应完毕。

**判定条件**（满足任一）：
- Layer 2 检测到 `charsDelta > 0`（文本增长）
- Layer 2 检测到 `charsDelta == 0` 但持续时间 < 15 秒
- Layer 1 检测到任务状态为 `running` 且无异常信号

**语义**：一切正常，无需任何干预。

**状态转移**：
- `healthy` → `model-stalled`：文本 30 秒无增长
- `healthy` → `terminal-hang`：检测到"后台运行"按钮
- `healthy` → `modal-blocking`：检测到模态弹窗
- `healthy` → `completed`：任务状态变为 `completed`

## 3.1.3 completed 模式

**定义**：任务已成功完成，进入终态。

**判定条件**：
- Layer 1 检测到任务状态为 `completed`
- 任务列表中该任务项消失或标记为完成

**语义**：任务结束，心跳监控停止。

**状态转移**：
- `completed` 为终态，不再转移

## 3.1.4 model-stalled 模式

**定义**：AI 响应停滞，文本内容长时间无增长。

**判定条件**：
- Layer 2 连续 2 次检测（间隔 15 秒）`charsDelta == 0`
- 累计停滞时间 ≥ 30 秒
- 排除 `model-queuing` 情况（无"排在第 X 位"文本）

**语义**：AI 可能卡死或网络断开，需要尝试恢复。

**恢复策略**：
1. 点击停止按钮
2. 等待 3 秒
3. 点击重试按钮
4. 进入 `verifying` 验证窗口

**状态转移**：
- `model-stalled` → `verifying`：执行恢复动作后
- `model-stalled` → `healthy`：恢复成功（验证窗口内文本增长）
- `model-stalled` → `task-interrupted`：重试次数耗尽

## 3.1.5 terminal-hang 模式

**定义**：终端命令执行时间过长，出现"后台运行"按钮。

**判定条件**：
- Layer 3 检测到 DOM 中存在"后台运行"按钮
- 按钮文本包含"后台运行"或"background"

**语义**：终端命令阻塞，需要用户决策是否后台运行。

**恢复策略**：
1. 点击"后台运行"按钮
2. 发送 info 级别通知
3. 任务继续执行

**状态转移**：
- `terminal-hang` → `healthy`：点击"后台运行"后
- `terminal-hang` → `task-interrupted`：用户点击"取消"

## 3.1.6 modal-blocking 模式

**定义**：Trae 弹出模态对话框等待用户选择。

**判定条件**：
- Layer 3 检测到模态弹窗 DOM 元素
- 常见场景：文件保留/删除选择、权限确认

**语义**：任务被 UI 阻塞，需要自动处理弹窗。

**恢复策略**：
- PMCLI 工作区：点击"保留"
- DEVCLI 工作区：点击"删除"
- 其他工作区：点击默认选项

**状态转移**：
- `modal-blocking` → `healthy`：处理弹窗后
- `modal-blocking` → `task-interrupted`：弹窗无法自动处理

## 3.1.7 model-queuing 模式

**定义**：AI 服务繁忙，请求正在排队。

**判定条件**：
- Layer 2 检测到最后一条消息包含"排在第 X 位"或类似排队提示
- 文本无增长但有排队标识

**语义**：正常排队等待，不是异常，不需要恢复。

**特殊处理**：
- 延长 `waitResponse` 超时时间
- 不触发恢复动作
- 不发送告警通知

**状态转移**：
- `model-queuing` → `healthy`：排队结束，AI 开始响应
- `model-queuing` → `model-stalled`：排队提示消失但文本仍无增长

## 3.1.8 task-interrupted 模式

**定义**：任务被外部中断，进入终态。

**判定条件**：
- Layer 1 检测到任务状态文本包含"中断"
- 任务列表中该任务标记为中断状态

**语义**：任务无法继续，需要人工介入。

**处理策略**：
- 立即发送 critical 级别告警
- 不尝试自动恢复
- 任务机进入 `interrupted` 终态

**状态转移**：
- `task-interrupted` 为终态，不再转移

## 3.1.9 unknown 模式

**定义**：检测器无法确定当前状态。

**判定条件**：
- 所有检测器返回空或冲突
- DOM 结构不符合预期

**语义**：需要人工检查。

**处理策略**：
- 记录详细日志
- 发送 warning 级别通知
- 持续监控，等待状态明朗

**状态转移**：
- `unknown` → 其他模式：检测到明确信号后
- `unknown` → `task-interrupted`：长时间无法确定（> 5 分钟）
