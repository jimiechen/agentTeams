# 3.4 状态回落机制

## 回落流程概览

异常恢复后不是直接回到 healthy，而是走**验证窗口**：

```
异常模式 ──恢复动作执行──▶ verifying（临时状态）
                                │
                          等待 10 秒
                                │
            ┌───────────────────┴───────────────────┐
            ▼                                       ▼
     信号恢复正常                           异常信号仍存在
     转 healthy                             重试恢复（计数+1）
                                            超过上限转 task-interrupted
```

## 验证窗口设计

### 验证窗口时长

**10 秒** - 足够观察模型是否真正恢复：

| 场景 | 所需时间 |
|------|---------|
| 模型重新开始流式输出 | 2~5 秒 |
| 按钮消失（弹窗/终端） | 1~3 秒 |
| CDP 响应延迟 | 最多 2 秒 |
| 预留缓冲 | 剩余时间 |

### 验证条件

不同模式的验证条件不同：

| 模式 | 验证条件 |
|------|---------|
| terminal-hang | "后台运行"按钮消失 |
| modal-blocking | "保留/删除"按钮消失 |
| model-stalled | 文本重新开始增长（charsDelta > 0） |

### 验证代码

```javascript
async function verifyRecovery(mode) {
    await sleep(10000); // 等待 10 秒验证窗口
    
    // 刷新最新信号
    await layer1FastProbe(cdp);
    await layer2ContentProbe(cdp);
    
    switch (mode) {
        case 'terminal-hang':
            return !lastSignal.layer1.hasBackgroundButton;
        case 'modal-blocking':
            return !lastSignal.layer1.hasKeepButton && !lastSignal.layer1.hasDeleteButton;
        case 'model-stalled':
            return lastSignal.layer2.charsDelta > 0;
        default:
            return true;
    }
}
```

## 重试计数策略

### 独立计数
每种模式独立计数，计数器在转为 healthy 时重置：

```javascript
const recoveryAttempts = {
    modelStalled: 0,     // 最多重试 3 次
    terminalHang: 0,     // 最多重试 2 次
    modalBlocking: 0     // 最多重试 1 次
};
```

### 重试上限
| 模式 | 重试上限 | 原因 |
|------|---------|------|
| model-stalled | 3 次 | 模型问题可能需要多次重试 |
| terminal-hang | 2 次 | 终端问题重试 2 次足够 |
| modal-blocking | 1 次 | 弹窗通常只出现一次 |

### 计数逻辑

```javascript
// 执行恢复动作前检查重试次数
function canRetry(mode) {
    const limits = {
        'model-stalled': 3,
        'terminal-hang': 2,
        'modal-blocking': 1
    };
    const attempts = lastSignal.recoveryAttempts[mode.replace(/-/g, '')];
    return attempts < limits[mode];
}

// 执行恢复动作后增加计数
function recordRecoveryAttempt(mode) {
    const key = mode.replace(/-/g, '');
    lastSignal.recoveryAttempts[key]++;
}

// 恢复成功后重置计数
function resetRecoveryAttempts() {
    lastSignal.recoveryAttempts = {
        modelStalled: 0,
        terminalHang: 0,
        modalBlocking: 0
    };
}
```

## 回落流程代码

```javascript
async function handleRecovery(mode) {
    // 检查是否可以重试
    if (!canRetry(mode)) {
        console.log(`⚠️ ${mode} 重试次数已耗尽，转为 task-interrupted`);
        lastSignal.currentMode = 'task-interrupted';
        await sendAlert('critical', `${mode} 重试次数耗尽`);
        return;
    }
    
    // 记录重试
    recordRecoveryAttempt(mode);
    
    // 执行恢复动作
    console.log(`🔄 执行 ${mode} 恢复动作 (第 ${lastSignal.recoveryAttempts[mode.replace(/-/g, '')]} 次)`);
    const success = await executeRecoveryAction(mode);
    
    if (!success) {
        console.log(`⚠️ ${mode} 恢复动作执行失败`);
        return;
    }
    
    // 进入验证窗口
    console.log(`⏳ 进入验证窗口 (10秒)`);
    lastSignal.currentMode = 'verifying';
    
    // 等待验证
    const recovered = await verifyRecovery(mode);
    
    if (recovered) {
        console.log(`✅ ${mode} 恢复成功`);
        resetRecoveryAttempts();
        lastSignal.currentMode = 'healthy';
    } else {
        console.log(`⚠️ ${mode} 恢复验证失败`);
        // 保持当前模式，下次心跳继续尝试
    }
}
```

## 特殊处理

### verifying 状态的意义

**为什么需要 verifying 状态？**

1. **防止表面成功**：点击了按钮但实际没效果
2. **等待副作用**：某些操作需要时间生效
3. **减少状态抖动**：避免快速切换状态

### 验证失败后的处理

验证失败后不立即转为 task-interrupted，而是保持当前模式，让下次心跳继续判定：

```
验证失败 ──→ 保持当前模式 ──→ 下次心跳重新判定 ──→ 决定是重试还是升级
```

### 计数累积规则

只有执行了恢复动作才计数，验证失败不计入：

```javascript
// 只有成功执行了恢复动作才计数
if (actionExecuted) {
    recordRecoveryAttempt(mode);
}
```

## 回落机制总结

### 流程

```
异常检测 ──→ 执行恢复动作 ──→ 进入 verifying ──→ 验证成功 ──→ healthy
                                              ──→ 验证失败 ──→ 保持原模式
                                              ──→ 重试耗尽 ──→ task-interrupted
```

### 关键设计点

1. **验证窗口**：10 秒等待，确保恢复真正生效
2. **独立计数**：每种模式独立计数，互不影响
3. **条件验证**：不同模式用不同条件验证
4. **计数重置**：恢复成功后重置所有计数器

### 优势

- **防止误判**：验证窗口避免表面成功
- **资源保护**：重试上限防止无限循环
- **状态稳定**：减少状态抖动
- **可追溯**：每次重试都有记录
