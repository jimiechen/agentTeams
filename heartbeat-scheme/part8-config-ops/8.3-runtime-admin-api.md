# 8.3 运行时动态调整接口

## 8.3.1 动态调整的必要场景

有些参数需要**不重启**就能调整，典型场景：

1. **紧急降频**：发现心跳本身压力过大，临时把 layer1 从 5s 调到 10s
2. **临时扩容重试**：某类任务当天特别不稳定，把 `stalled_retry_limit` 从 2 调到 5
3. **屏蔽告警**：已知某工作区在维护中，临时关闭它的告警发送

这些场景都要求分钟级响应，重启流程太慢。

## 8.3.2 接口设计原则

### 原则一：只读和写操作分离
- GET /status、GET /metrics 无副作用，任何人可访问
- POST /policy、POST /toggle 需要认证（如 token）

### 原则二：原子性变更
每次接口只能变更一个小范围参数，如"只改 layer1 频率"或"只改一个工作区的策略"，避免大配置替换导致混乱。

### 原则三：配置落地双写
接口变更后：
1. 立即应用到内存中的 Daemon（生效时间 <1s）
2. 写入 `config/heartbeat.yaml.override`（落地持久化）
3. Daemon 启动时优先读 override，再读基础配置

双写确保重启后仍保持调整后的状态，不需要手动重写配置。

## 8.3.3 完整接口列表

### GET /status
返回所有 Daemon 的当前状态（运行/暂停/已停止）。

### GET /policy/:workspace
返回指定工作区的当前策略配置（JSON）。

### POST /policy/:workspace
更新单个工作区的策略，请求体：

```json
{
  "modal_strategy": "keep",
  "stalled_retry_limit": 3
}
```

### POST /toggle/:workspace/:action
:action 可选：start、stop、pause、resume。

### POST /global/frequency
更新全局频率配置（不影响已有 Timer，下次启动生效）。

### GET /heartbeat/stream
WebSocket 端点，实时推送心跳事件（调试用）。

## 8.3.4 安全性设计

### 认证机制
通过 HTTP Header `Authorization: Bearer ${HEARTBEAT_ADMIN_TOKEN}` 认证，token 在环境变量里配置，默认不开启（HEARTBEAT_ADMIN_ENABLED=true 才启用）。

### 限流保护
写接口（POST）有 IP 级别的限流（如 10s 内最多 5 次），防误操作和攻击。

### 日志审计
所有写接口调用都写入 `admin-audit.jsonl`，包含：
- 调用时间
- 调用者 IP
- 变更前后的配置 diff
- 操作结果
