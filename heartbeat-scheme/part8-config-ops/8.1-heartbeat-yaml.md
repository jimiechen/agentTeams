# 8.1 heartbeat.yaml 配置文件

## 8.1.1 配置分层设计

配置文件分三个层次：**global**（全局参数）、**policies**（按工作区的策略）、**notifications**（通知相关）。

分层的原因是**变更频率不同**——global 几乎不变，policies 按工作区差异化，notifications 随告警接收人变化而调整。

## 8.1.2 完整配置示例

```yaml
# config/heartbeat.yaml

global:
  # 三层心跳频率（毫秒）
  layer1_interval_ms: 5000
  layer2_interval_ms: 15000
  layer3_interval_ms: 60000
  
  # 通知频率控制
  notify_cooldown_ms: 300000  # 5 分钟
  
  # metrics 轮转
  metrics_max_size_mb: 10
  metrics_retention_days: 7

policies:
  PMCLI:
    modal_strategy: keep          # 弹窗默认保留（保守）
    terminal_strategy: background # 终端默认后台
    stalled_retry_limit: 2        # 停滞最多重试 2 次
    alert_on_first_stall: false   # 第一次停滞不告警
    queuing_alert_threshold_min: 5 # 排队超 5 分钟告警
    
  DEVCLI:
    modal_strategy: delete        # 允许覆盖（激进）
    terminal_strategy: background
    stalled_retry_limit: 3        # 更激进
    alert_on_first_stall: false
    queuing_alert_threshold_min: 10

notifications:
  heartbeat_chat_id: "oc_xxxxxxxxxxxxx"
  critical_alert_users:
    - "ou_xxxxxxxxxxxxx"
  wiki_todo_integration: true
```

## 8.1.3 工作区策略差异化

PMCLI 和 DEVCLI 的策略不同，反映的是**工作区性质差异**：

- **PMCLI**（产品管理类任务）：结果不可覆盖，每次产出都是增量，所以 `modal_strategy: keep`；任务通常较短，`stalled_retry_limit: 2` 够用。
- **DEVCLI**（开发类任务）：代码生成经常需要重跑覆盖，所以 `modal_strategy: delete`；任务复杂度高，重试次数提高到 3。

差异化策略通过工作区 key 直接查表，不需要复杂的策略引擎。

## 8.1.4 热加载与版本兼容

### 当前状态

当前版本的配置文件在 Daemon 启动时读取一次，**不支持热加载**。原因是热加载增加复杂度（需要处理正在执行的恢复动作与新配置的冲突），收益有限（配置变更通常伴随重启）。

### 未来热加载设计

如果需要热加载，可以在 `HeartbeatDaemon` 里增加 `fs.watch()` 监听配置文件变化，但必须遵守：

"已开始的恢复动作按旧配置走完，新动作按新配置启动"。

### 版本兼容

版本兼容通过 `config_version` 字段实现——配置文件顶部声明 `config_version: "1.0"`，未来 schema 变更时递增版本号，代码按版本号选择解析逻辑。
