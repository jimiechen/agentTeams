# 8.2 环境变量管理

## 8.2.1 环境变量 vs 配置文件的分工

两者分工的核心原则是**敏感度**：

| 内容 | 存放位置 | 原因 |
|------|---------|------|
| 心跳频率、策略参数 | `heartbeat.yaml` | 非敏感，需要版本管理 |
| 飞书 chat_id、用户 ID | `heartbeat.yaml` | 半敏感，但需要按环境差异化 |
| 飞书 App Secret、LLM API Key | `.env` | 高敏感，禁止进 Git |
| 工作区物理路径 | `.env` | 环境差异，不应硬编码 |

### 原则总结

凡是泄露会直接造成损失的（API Key、Secret），必须走环境变量；凡是需要多人协作 review 的参数（频率、策略），必须走配置文件。

## 8.2.2 `.env` 模板

```bash
# .env.example（版本控制中保留此文件，.env 本身 gitignore）

# 飞书通知
HEARTBEAT_LARK_APP_ID=cli_xxxxxxxxxxxxx
HEARTBEAT_LARK_APP_SECRET=xxxxxxxxxxxxx
HEARTBEAT_LARK_CHAT_ID=oc_xxxxxxxxxxxxx
HEARTBEAT_CRITICAL_ALERT_USERS=ou_xxx,ou_yyy

# LLM API（用于 WikiBot 蒸馏）
OPENAI_API_KEY=sk-xxxxxxxxxxxxx
DISTILL_MODEL=gpt-4o-mini

# 工作区路径
WORKSPACE_PMCLI_PATH=./workspaces/PMCLI
WORKSPACE_DEVCLI_PATH=./workspaces/DEVCLI

# Admin 接口
HEARTBEAT_ADMIN_PORT=9090
HEARTBEAT_ADMIN_ENABLED=true
```

### Git 规则

版本控制中只提交 `.env.example`（模板，值为占位符），真正的 `.env` 通过 `.gitignore` 排除。

新成员上手时 `cp .env.example .env` 并填入自己的凭据。

## 8.2.3 覆盖优先级

配置加载时的优先级（从高到低）：

```
1. 命令行参数（如 --layer1_interval_ms=3000）
2. 环境变量（HEARTBEAT_LAYER1_INTERVAL_MS=3000）
3. heartbeat.yaml
4. 代码内置默认值
```

这个优先级让**临时调试**（命令行）、**环境差异**（环境变量）、**常规配置**（yaml）、**兜底保护**（默认值）各司其职。

## 8.2.4 敏感信息保护

环境变量虽然不进 Git，但仍可能通过日志泄露。心跳模块的日志输出必须遵守：

- **禁止直接输出 process.env**（会暴露所有环境变量）
- **涉及 API Key 的日志必须打码**（如 `sk-xxxx...xxxx` 只显示首尾 4 位）
- **错误堆栈打印前必须过滤**（通过 `sanitizeError()` 函数移除敏感字段）
