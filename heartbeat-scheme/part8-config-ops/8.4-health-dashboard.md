# 8.4 心跳健康度仪表盘

## 8.4.1 仪表盘的三个目标

仪表盘不是用来监控 Trae 任务本身，而是用来监控**心跳系统自身**：

1. **发现心跳是否漏报**（如明明有异常但没发通知）
2. **评估自动恢复的效果**（恢复率、重试次数）
3. **识别需要优化的模式**（哪些异常高频但恢复成功？可以做成更激进的自动处理）

## 8.4.2 数据来源

仪表盘唯一数据来源是 `heartbeat.jsonl`——不依赖额外的数据库，不要求 daemon 一直运行，即使服务重启后仍然可以聚合历史数据。

聚合逻辑写在独立的脚本 `scripts/heartbeat-dashboard.js` 里，可以手动跑，也可以用 cron 每 5 分钟自动跑。

## 8.4.3 仪表盘视图（HTML 单页）

### 顶部概览卡片

```
┌───────────────────────────────────────────────────────────────┐
│  🟢 总体健康度: 87%              🟡 告警中: 1 个工作区       │
│  💾 数据时间范围: 过去 7 天          📊 总心跳次数: 24,576   │
└───────────────────────────────────────────────────────────────┘
```

### 三个核心图表

#### 图表一：异常模式分布（饼图）
显示各模式在 7 天内的占比，一眼看出哪类问题最多。

#### 图表二：自动恢复成功率趋势（折线图）
按天显示恢复率变化，如果某天恢复率突然下降，说明 Trae 服务或环境可能有异常。

#### 图表三：告警分布热力图
x 轴是小时，y 轴是工作日，看告警主要集中在什么时候（如凌晨模型服务不稳定）。

### 底部工作区列表

每个工作区一行，显示：
- 最近一次健康检查
- 活跃异常模式（如果有）
- 7 天恢复率
- 快速链接（打开 Trae IDE、打开日志）

## 8.4.4 异常高亮规则

如果满足以下任一条件，工作区卡片变成红色/黄色（红色比黄色严重）：

- **红色**：连续 3 次 critical 告警未解决、自动恢复率 <50%
- **黄色**：有 active warning、排队时长 >10 分钟、恢复率 50%~70%

## 8.4.5 实现思路（非代码）

1. 脚本先读取 `heartbeat.jsonl`，按时间过滤出最近 7 天的数据
2. 用简单的 Canvas 库（如 Chart.js）画图表
3. 生成静态 HTML 文件，放在 `public/heartbeat-dashboard.html`
4. 用简单的静态文件服务器（如 `python -m http.server 8080`）对外访问

不引入复杂的 Web 框架，保持运维简单。
