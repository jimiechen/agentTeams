## 二、详细评审意见

### 2.1 架构设计评审

#### ✅ 优点

1. **三层心跳架构设计合理**
   - Layer 1（5s）：快速检测显式异常，成本可控
   - Layer 2（15s）：文本增量检测，避免误判
   - Layer 3（60s）：深度健康评估，支持可观测性

2. **状态机设计完整**
   - 7种健康模式定义清晰，互斥关系明确
   - 状态转移矩阵覆盖所有场景
   - 优先级规则避免多信号冲突

3. **信号传递机制高效**
   - 共享内存避免重复采集
   - 三层信号复用，保证数据一致性

#### ⚠️ 建议改进

| 问题 | 位置 | 建议 |
|------|------|------|
| **缺少心跳自身的健康检查** | Part 6 | 建议增加"心跳看门狗"机制，检测心跳是否正常执行 |
| **Layer 3 采集项可能遗漏** | Part 2.3 | 建议增加"未知按钮告警"，当发现非白名单按钮时主动通知 |

**改进建议代码**：

```typescript
// 心跳看门狗：检测心跳是否正常执行
class HeartbeatWatchdog {
  private lastHeartbeatTime: Map<string, number> = new Map();
  private readonly WATCHDOG_THRESHOLD = 30000; // 30秒
  
  check(workspace: string): boolean {
    const lastTime = this.lastHeartbeatTime.get(workspace) ?? 0;
    const elapsed = Date.now() - lastTime;
    
    if (elapsed > this.WATCHDOG_THRESHOLD) {
      console.warn(`⚠️ ${workspace} 心跳已停滞 ${elapsed}ms`);
      return false;
    }
    return true;
  }
  
  feed(workspace: string): void {
    this.lastHeartbeatTime.set(workspace, Date.now());
  }
}
```

## 四、评审结论与建议

### 4.1 总体结论

**通过，有条件建议改进**

方案设计整体优秀，架构清晰、职责单一、扩展性好。主要改进点集中在：
1. 补充 Part 10-11 文档
2. 增加心跳自身的健康检查
3. 完善操作审计的持久化

### 4.2 必须改进项（阻塞上线）

| 序号 | 改进项 | 原因 |
|------|-------|------|
| 1 | 补充 Part 10 测试与验证文档 | 无法验证方案正确性 |
| 2 | 增加心跳看门狗机制 | 心跳自身故障无法检测 |
| 3 | 明确下游代码兼容性清单 | 无法评估改动影响 |

### 4.3 建议改进项（可后续迭代）

| 序号 | 改进项 | 优先级 |
|------|-------|-------|
| 1 | 操作审计持久化 | 中 |
| 2 | CDP 调用超时机制 | 中 |
| 3 | 心跳状态持久化 | 低 |
| 4 | 验证窗口动态调整 | 低 |
| 5 | 按钮选择器缓存 | 低 |

### 4.4 实施建议

1. **分阶段上线**
   - 第一阶段：只观测不干预，验证信号采集准确性
   - 第二阶段：激活恢复动作，但限制重试次数
   - 第三阶段：完整功能上线

2. **灰度策略**
   - 先在 DEVCLI 工作区验证
   - 验证通过后再推广到 PMCLI

3. **监控指标**
   - 心跳执行成功率
   - 自动恢复成功率
   - 误判率（人工确认）
