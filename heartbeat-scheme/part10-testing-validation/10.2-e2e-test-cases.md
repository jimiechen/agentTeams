# 10.2 端到端测试用例设计

## 10.2.1 测试分层

心跳机制的测试分三层：

| 层级 | 范围 | 工具 | 运行时长 |
|------|------|------|---------|
| **单元测试** | 单个函数（如 `classifyMode()`） | Vitest + mock DOM | < 10s |
| **集成测试** | Detector + Executor 联动 | Vitest + 真实 CDP + 触发工具 | 1-5min |
| **端到端测试** | 完整任务 + 心跳全流程 | 真实 Trae + 真实任务 | 5-30min |

本节聚焦**端到端测试用例**，因为前两层较标准。

## 10.2.2 核心测试用例清单

设计 12 个必过用例，覆盖正常 + 五种异常 + 边界场景：

| 用例 ID | 场景 | 预期结果 |
|---------|------|---------|
| E2E-001 | 任务正常完成 | 无心跳介入，runs/*.md 含心跳摘要 |
| E2E-002 | terminal-hang 自动恢复 | 点击"后台运行"，任务继续完成，发 info 通知 |
| E2E-003 | modal-blocking（PMCLI）自动保留 | 点击"保留"，发 info 通知 |
| E2E-004 | modal-blocking（DEVCLI）自动删除 | 点击"删除"，发 info 通知 |
| E2E-005 | model-stalled 首次重试成功 | stop+retry，回到 healthy，发 warning 通知 |
| E2E-006 | model-stalled 重试耗尽 | 进入 interrupted 终态，发 critical 告警 |
| E2E-007 | model-queuing 被动等待 | 不触发恢复，延长 waitResponse 超时 |
| E2E-008 | task-interrupted 立即告警 | 进入 interrupted 终态，发 critical 告警 |
| E2E-009 | 多工作区并发，一个异常 | 异常工作区走告警，其他工作区不受影响 |
| E2E-010 | 心跳降级开关关闭 | 行为与改造前完全一致 |
| E2E-011 | Admin API 动态调整 | 调整后新阈值立即生效 |
| E2E-012 | Daemon 崩溃自愈 | Daemon 自动重启，主任务不中断 |

## 10.2.3 测试用例模板

所有用例遵循统一模板（Given-When-Then 风格）：

```typescript
// test/e2e/heartbeat-stalled-recovery.test.ts

describe('E2E-005: model-stalled 首次重试成功', () => {
  let runner: WorkspaceRunner;
  let triggers: HeartbeatTriggers;
  
  beforeAll(async () => {
    runner = await setupTestRunner('PMCLI-test');
    triggers = new HeartbeatTriggers(runner.cdpClient);
  });
  
  afterAll(async () => {
    await triggers.cleanup();
    await runner.stop();
  });
  
  it('should recover after 30s stall and reach completed', async () => {
    const taskPromise = runner.submitTask('请写一段 hello world');
    await waitForTaskState(runner, 'waiting');
    
    await sleep(10_000);
    await triggers.stalled(40_000);
    
    const heartbeatEvents = await captureHeartbeatEvents(runner, 60_000);
    
    expect(heartbeatEvents).toContainEqual(
      expect.objectContaining({
        mode: 'model-stalled',
        action: 'stop+retry'
      })
    );
    
    const result = await taskPromise;
    expect(result.status).toBe('completed');
    
    const notifications = await readLarkNotifications();
    expect(notifications).toContainEqual(
      expect.objectContaining({ level: 'warning', mode: 'model-stalled' })
    );
    
    const runFile = await readLatestRunFile();
    expect(runFile).toMatch(/Recovery attempts: 1/);
    expect(runFile).toMatch(/model-stalled: 1/);
  });
});
```

**设计要点**：
- 每个用例都验证**多个维度**（任务状态 + 心跳事件 + 通知 + 日志）
- 验证"发生了什么"而不是"内部怎么做的"（避免测试代码和实现耦合）
- 超时设置留足缓冲（实际需要 30 秒的用例给 60 秒超时）

## 10.2.4 E2E-009 多工作区隔离测试

隔离性验证是最关键也最容易被忽略的用例。专门设计：

```typescript
describe('E2E-009: 多工作区隔离', () => {
  it('PMCLI interrupted should not affect DEVCLI', async () => {
    const pmcliTask = pmcliRunner.submitTask('task A');
    const devcliTask = devcliRunner.submitTask('task B');
    
    await Promise.all([
      waitForTaskState(pmcliRunner, 'waiting'),
      waitForTaskState(devcliRunner, 'waiting')
    ]);
    
    await pmcliTriggers.interrupted();
    
    const pmcliResult = await pmcliTask;
    expect(pmcliResult.status).toBe('interrupted');
    
    const devcliResult = await devcliTask;
    expect(devcliResult.status).toBe('completed');
    
    const devcliHeartbeats = await readHeartbeatJsonl('DEVCLI');
    const anomalies = devcliHeartbeats.filter(e => e.mode !== 'healthy');
    expect(anomalies).toHaveLength(0);
  });
});
```

这个用例专门验证 Part 9.3 设计的独立性原则，是集成测试的**必过项**。

## 10.2.5 测试环境准备

E2E 测试需要真实的 Trae 环境，但**不应该污染生产账号**。规范：

1. **专用测试 Trae 账号**：公司内申请 `trae-test-{name}` 账号，只用于 E2E 测试
2. **专用工作区命名**：所有测试工作区以 `-test` 后缀命名（PMCLI-test、DEVCLI-test），与生产隔离
3. **专用飞书群**：测试通知发到 `trae-heartbeat-test` 群，不打扰生产监控群
4. **测试后清理**：每个 describe 的 `afterAll` 必须调 `triggers.cleanup()` 和删除测试生成的文件
