# 10.4 回滚与降级策略

## 10.4.1 三级降级机制

心跳机制一旦故障，必须有明确的降级路径，**不能让心跳的故障拖垮主任务**：

| 降级级别 | 触发条件 | 行为 |
|---------|---------|------|
| **L1 单动作降级** | 某次恢复动作失败 | 本次恢复跳过，记录错误，下次继续 |
| **L2 Daemon 降级** | 连续 5 次 CDP 失败 | Daemon 停止心跳，保留任务运行 |
| **L3 全局降级** | 环境变量 `HEARTBEAT_ENABLED=false` | 完全禁用心跳，回到改造前行为 |

### L1 单动作降级

每次恢复动作失败时，记录错误但不中断心跳：

```typescript
async recover(signal: HeartbeatSignal): Promise<RecoveryResult> {
  try {
    // ... 执行恢复动作
  } catch (error) {
    this.logger.error(`Recovery failed for ${signal.mode}: ${error.message}`);
    this.recoveryFailures++;
    
    return {
      attempted: true,
      action: signal.mode,
      success: false,
      error: error.message
    };
  }
}
```

**验证点**：单次失败不影响后续心跳，错误信息写入 `heartbeat-errors.log`。

### L2 Daemon 降级

连续 5 次 CDP 调用失败时，Daemon 自动停止：

```typescript
class HeartbeatDaemon {
  private cdpFailureCount = 0;
  private readonly MAX_CDP_FAILURES = 5;
  
  private async runHeartbeat(): Promise<void> {
    try {
      await this.detector.heartbeat();
      this.cdpFailureCount = 0;
    } catch (error) {
      this.cdpFailureCount++;
      
      if (this.cdpFailureCount >= this.MAX_CDP_FAILURES) {
        this.logger.error('连续 5 次 CDP 失败，停止心跳守护');
        this.stop();
        await this.notifyAlert('heartbeat-stopped', '心跳守护因 CDP 故障停止');
      }
    }
  }
}
```

**验证点**：CDP 恢复后需要手动重启 Daemon，或配置自动重启策略。

### L3 全局降级

通过环境变量 `HEARTBEAT_ENABLED=false` 完全禁用心跳：

```typescript
// task-machine.ts
export const taskMachine = createMachine({
  states: {
    waiting: {
      type: 'parallel',
      states: {
        response: { /* ... */ },
        heartbeat: {
          invoke: {
            src: (ctx) => ctx.heartbeatEnabled 
              ? heartbeatMachine 
              : noopMachine
          }
        }
      }
    }
  }
});
```

**验证点**：设置后行为与改造前完全一致，可作为紧急回滚手段。

## 10.4.2 回滚场景与操作

### 场景一：心跳机制引入 Bug

**问题**：心跳修改了某个状态机逻辑，导致任务无法正常完成。

**回滚操作**：

```bash
# 方式一：临时禁用（最快）
export HEARTBEAT_ENABLED=false
./start.sh

# 方式二：代码回滚
git revert <heartbeat-commit-hash>
git push origin main
```

**优先级**：先试方式一（1分钟内生效），确认问题后再做代码回滚。

### 场景二：心跳频繁误判

**问题**：心跳把正常任务误判为异常，频繁触发恢复动作。

**回滚操作**：

```bash
# 临时降低心跳频率
export HEARTBEAT_LAYER1_INTERVAL_MS=30000  # 从 5s 改为 30s
./start.sh
```

同时紧急修复误判逻辑。

### 场景三：通知风暴

**问题**：心跳发送过多通知，打扰用户。

**回滚操作**：

```bash
# 临时关闭通知
export HEARTBEAT_NOTIFICATIONS_ENABLED=false
./start.sh
```

同时检查频率控制逻辑是否生效。

## 10.4.3 回滚后的验证

回滚后必须验证以下指标，确保系统回到稳定状态：

| 指标 | 验证方式 | 预期结果 |
|------|---------|---------|
| 心跳是否停止 | 查看 heartbeat.jsonl 是否继续写入 | 停止写入 |
| 任务是否正常 | 提交一个测试任务 | 正常完成 |
| 通知是否停止 | 查看飞书群 | 无通知 |
| CPU 占用是否下降 | `top` 命令 | 回到改造前水平 |

## 10.4.4 灰度发布策略

为了降低上线风险，推荐采用灰度发布：

**阶段一（10%）**：仅在 PMCLI-test 工作区启用心跳，观察 24 小时

**阶段二（50%）**：在 PMCLI 生产工作区启用，DEVCLI 保持禁用，观察 48 小时

**阶段三（100%）**：全量启用，持续监控

每个阶段都设置回滚阈值，如：

- 恢复成功率 < 80% → 立即回滚
- 告警误报率 > 10% → 立即回滚
- CPU 占用 > 5% → 立即回滚

## 10.4.5 监控告警配置

生产环境必须配置以下监控告警：

| 告警项 | 阈值 | 告警级别 | 通知渠道 |
|--------|------|---------|---------|
| 心跳停止 | 连续 5 分钟无心跳 | critical | 飞书 + 电话 |
| 恢复成功率下降 | < 80% | warning | 飞书 |
| 告警误报率上升 | > 10% | warning | 飞书 |
| CPU 占用过高 | > 5% | warning | 飞书 |
| 内存泄漏 | 持续增长 > 50MB | critical | 飞书 + 电话 |

### 告警通知规则

- **critical 级告警**：立即通知，同时发送到飞书群和值班人员电话
- **warning 级告警**：延迟 5 分钟发送（防止抖动），只发飞书群
- **info 级通知**：汇总到每日报告，不单独发送

### 监控仪表盘

实时监控仪表盘需要展示以下指标：

1. **实时状态**：各工作区的当前健康模式
2. **恢复统计**：今日恢复次数、成功率
3. **告警统计**：今日告警数量、级别分布
4. **资源占用**：CPU、内存、文件句柄

### 日志保留策略

| 日志类型 | 保留期限 | 存储方式 |
|---------|---------|---------|
| heartbeat.jsonl | 30 天 | 本地文件 |
| heartbeat-errors.log | 90 天 | 本地文件 |
| 飞书通知记录 | 永久 | WikiBot 蒸馏 |

### 降级后恢复流程

当降级被触发后，恢复流程：

1. **确认问题已修复**：检查 CDP 连接、修复代码 bug
2. **手动重启 Daemon**：通过 Admin API 或重启进程
3. **验证恢复**：提交测试任务，确认心跳正常工作
4. **恢复通知**：发送 info 级通知告知用户心跳已恢复

### 回滚演练

每季度进行一次回滚演练，确保降级机制有效：

1. **模拟场景**：人工触发 CDP 连接失败
2. **验证降级**：确认 Daemon 自动停止
3. **验证恢复**：修复后确认 Daemon 能正常重启
4. **记录问题**：总结演练中发现的问题并改进
