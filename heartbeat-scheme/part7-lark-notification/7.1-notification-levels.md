# 7.1 三级通知分级体系

## 7.1.1 分级原则

心跳机制的通知系统采用**三级分级体系**，确保用户在合适的时机收到合适的信息：

| 级别 | 含义 | 典型场景 | 用户感知 | 发送频率 |
|------|------|---------|---------|---------|
| `info` | 信息通知 | 恢复成功、状态变更 | 低打扰 | 每次事件 |
| `warning` | 警告通知 | 自动恢复失败、未知状态 | 中等关注 | 5分钟/次 |
| `critical` | 严重告警 | 任务中断、恢复耗尽 | 高优先级 | 立即发送 |

## 7.1.2 info 级定义

**语义**：任务正常运行过程中的状态变化，无需用户立即关注。

**触发条件**：
- 自动恢复成功（terminal-hang → healthy）
- 自动恢复成功（modal-blocking → healthy）
- 任务正常完成
- 排队状态变化（进入/退出 model-queuing）

**用户期望**：
- 作为"心跳"信号，确认任务仍在运行
- 可忽略，不影响正常工作
- 支持聚合显示（如"今日已自动处理 3 次终端阻塞"）

**发送策略**：
- 每次事件独立发送
- 消息内容简洁，突出关键信息
- 不@用户，不触发强提醒

## 7.1.3 warning 级定义

**语义**：检测到潜在问题，系统正在尝试处理，建议用户关注。

**触发条件**：
- 自动恢复失败，进入重试
- 检测到 `unknown` 状态
- 单次恢复动作执行超时
- 连续多次检测到同一异常

**用户期望**：
- 了解任务可能存在问题
- 决定是否需要人工介入
- 不要求立即响应

**发送策略**：
- 频率控制：同类警告 5 分钟内不重复发送
- 包含问题描述和当前处理状态
- 提供"查看详情"链接

## 7.1.4 critical 级定义

**语义**：严重问题发生，任务无法继续，需要用户立即介入。

**触发条件**：
- 任务进入 `task-interrupted` 终态
- 恢复重试次数耗尽
- 检测到不可恢复的系统错误
- 用户权限不足导致恢复失败

**用户期望**：
- 立即收到通知
- 清晰了解问题原因
- 获得下一步操作建议

**发送策略**：
- 立即发送，无频率限制
- @相关用户，触发强提醒
- 包含告警 ID 便于追踪
- 提供快速操作入口（如"重新启动任务"）

## 7.1.5 级别判定流程

```
┌─────────────────────────────────────────────────────────────┐
│                      事件发生                                │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
                    ┌─────────────────┐
                    │ 是否为终态？     │
                    └─────────────────┘
                     │            │
                    是            否
                     │            │
                     ▼            ▼
            ┌──────────────┐  ┌─────────────────┐
            │ critical     │  │ 恢复是否成功？   │
            └──────────────┘  └─────────────────┘
                               │           │
                              是           否
                               │           │
                               ▼           ▼
                      ┌────────────┐  ┌─────────────────┐
                      │ info       │  │ 重试次数 > 0？   │
                      └────────────┘  └─────────────────┘
                                       │           │
                                      是           否
                                       │           │
                                       ▼           ▼
                              ┌────────────┐  ┌────────────┐
                              │ warning    │  │ critical   │
                              └────────────┘  └────────────┘
```

## 7.1.6 级别与恢复策略的关联

| 健康模式 | 恢复动作 | 通知级别 | 说明 |
|---------|---------|---------|------|
| `terminal-hang` | 点击"后台运行" | info | 自动处理成功 |
| `modal-blocking` | 点击弹窗按钮 | info | 自动处理成功 |
| `model-stalled` | stop + retry | info/warning | 首次成功为 info，失败为 warning |
| `model-queuing` | 无 | info | 仅通知排队状态 |
| `task-interrupted` | 无 | critical | 终态，需人工介入 |
| `unknown` | 持续监控 | warning | 无法自动处理 |

## 7.1.7 通知渠道配置

| 级别 | Lark 机器人 | 邮件 | 短信 | 站内信 |
|------|------------|------|------|--------|
| `info` | ✅ | ❌ | ❌ | ✅ |
| `warning` | ✅ | ❌ | ❌ | ✅ |
| `critical` | ✅ | ✅ | ✅（值班人员） | ✅ |

**配置原则**：
- Lark 机器人作为主渠道，承载所有级别通知
- 邮件仅用于 critical 级别，避免通知疲劳
- 短信仅通知值班人员，确保响应时效
- 站内信作为兜底，支持历史查询
