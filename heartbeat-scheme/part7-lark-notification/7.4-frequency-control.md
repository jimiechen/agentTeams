# 7.4 通知频率控制

## 7.4.1 控制目标：防告警风暴

心跳高频采样带来一个副作用——同一个异常在连续几次心跳里都会被检测到。如果每次都发通知，飞书会在几分钟内收到十几条相同告警，这就是"告警风暴"。

频率控制的目标是**同类异常合并为一条通知**。

## 7.4.2 `shouldNotify()` 实现

```typescript
private lastNotifyTime: Map<string, number> = new Map();

private shouldNotify(
  level: 'info' | 'warning' | 'critical', 
  workspace: string,
  mode: string
): boolean {
  const key = `${level}:${workspace}:${mode}`;
  const now = Date.now();
  const lastTime = this.lastNotifyTime.get(key) ?? 0;
  
  const minInterval = {
    info: 5 * 60 * 1000,       // 5 分钟
    warning: 5 * 60 * 1000,    // 5 分钟
    critical: 0,               // 立即发送
  }[level];
  
  if (now - lastTime < minInterval) {
    return false;
  }
  
  this.lastNotifyTime.set(key, now);
  return true;
}
```

key 的设计是 `level:workspace:mode` 三维组合——相同级别、相同工作区、相同模式才算"同类"。PMCLI 的 terminal-hang 和 DEVCLI 的 terminal-hang 是两个独立的频控通道。

## 7.4.3 状态升级的例外规则

频率控制有一个重要例外：**状态升级时不受冷却期限制**。

### 场景举例

```
10:15:00  model-stalled 首次检测 → 发 warning
10:15:30  model-stalled 重试成功 → 节流跳过（距离上次不到 5 分钟）
10:17:00  model-stalled 再次出现 → 节流跳过
10:18:30  model-stalled 升级 task-interrupted → 必须发 critical（不受冷却限制）
```

### 升级规则判断标准

如果当前通知级别 > 最近一次同工作区通知级别，强制发送。这确保问题**恶化**时用户一定能收到消息。

## 7.4.4 冷却期重置规则

两种情况触发冷却期重置：

### 情况一：模式彻底恢复

连续 3 次 Layer 3 心跳确认工作区处于 healthy，清除该工作区所有模式的 `lastNotifyTime`。下次异常发生时即使距离上次通知不到 5 分钟，也重新发送。

### 情况二：Daemon 重启

`lastNotifyTime` 存储在内存，Daemon 重启后自动清空。这是容错设计——重启本身说明状态可能不一致，重发通知比漏发更安全。
