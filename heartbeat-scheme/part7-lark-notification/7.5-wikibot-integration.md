# 7.5 与 WikiBot 的知识联动

## 7.5.1 联动目的：告警 → 知识

心跳告警是一次性事件，单独看价值有限。但如果把**一段时间内的告警**聚合分析，就能发现规律——哪些异常模式高频、哪些工作区更脆弱、哪些恢复动作成功率低。

WikiBot 联动的目的是把**心跳实时信号**转化为**长期知识积累**。

## 7.5.2 告警写入 `wiki/todo.md`

所有 critical 告警和 warning 通知（不含 info）在发送飞书消息的同时，追加一行到 `wiki/todo.md`：

```typescript
private async notifyAlert(
  level: 'warning' | 'critical',
  mode: HealthMode, 
  detail: string
): Promise<void> {
  // 发送飞书消息
  await this.larkClient.send(this.formatMessage(level, mode, detail));
  
  // 写入 wiki/todo.md
  const timestamp = new Date().toISOString();
  const todoEntry = `- [ ] [${level}] 心跳告警 ${timestamp}: ${mode} - ${detail}\n`;
  appendFileSync(this.todoPath, todoEntry);
}
```

### 为什么用 Markdown 任务项格式？

用 `- [ ]` 格式的原因：WikiBot 蒸馏时可以通过 `grep "- \[ \]"` 提取所有待处理告警，处理完后改为 `- [x]` 标记闭环。

## 7.5.3 蒸馏规则：晋升与淘汰

WikiBot 每日蒸馏时对告警记录应用两条规则：

### 晋升规则

同一模式（如 `PMCLI:terminal-hang`）在 7 天内出现 ≥3 次 → 晋升为 Layer 2 核心知识的"已知陷阱"，写入 `wiki/known-issues.md`：

```markdown
### PMCLI 终端挂起高频问题

**观察频次**：7 天内 4 次
**首次发生**：2026-04-22
**最近发生**：2026-04-29
**自动恢复成功率**：75%
**建议**：长任务（>10min）开启"后台运行"默认策略
```

### 淘汰规则

同一模式连续 14 天未出现 → 从 Layer 2 淘汰回 Layer 3（归档），说明问题已解决或场景消失。淘汰时保留历史记录，不删除，以防未来再次出现时能快速关联。

## 7.5.4 蒸馏输出给用户的价值

蒸馏后的知识以**周报**形式发送给用户（非心跳通道，走独立的周报机器人）：

```
📊 心跳周报 2026-W18

本周告警总数：47
自动恢复成功率：83%（40/47 恢复）

Top 3 异常模式：
1. PMCLI terminal-hang × 18（100% 恢复）
2. DEVCLI model-stalled × 9（67% 恢复）
3. PMCLI modal-blocking × 6（100% 恢复）

需要关注：
⚠️ DEVCLI model-stalled 恢复率较低，建议排查模型服务稳定性
```

周报让用户从**事件驱动**切换到**趋势驱动**的视角——不关注每一次告警，而关注**告警分布的变化**。
