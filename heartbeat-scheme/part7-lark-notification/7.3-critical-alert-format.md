# 7.3 critical 级告警格式

## 7.3.1 设计目标：30 秒定位

critical 级的目标是让用户在收到通知后的 **30 秒内**完成三个判断：

1. **发生了什么**（异常模式和严重程度）
2. **在哪看**（日志文件、IDE 会话）
3. **该怎么处理**（具体操作步骤）

超过 30 秒用户还在搜索信息，说明消息格式失败。

## 7.3.2 消息模板的四要素

```
🚨 心跳告警 - 需人工介入

工作区：PMCLI
异常模式：model-stalled
详情：模型停滞 30 秒，已执行 stop+retry 共 3 次，均未恢复
时间：2026-04-29 10:15:23

📍 快速定位
- 查看 Trae IDE：直接打开当前会话
- 查看心跳日志：workspaces/PMCLI/runs/heartbeat.jsonl
- 查看最近执行：workspaces/PMCLI/runs/2026-04-29T10-00-00.md

🔧 建议操作
1. 检查 Trae IDE 是否卡死（尝试手动点击停止按钮）
2. 检查网络连接（模型服务可能不可用）
3. 如确认 IDE 异常，重启后重新触发任务

---
告警 ID: HB-20260429-101523-PMCLI-stalled
```

四要素分别对应：**基础事实**（工作区/模式/详情/时间）+ **快速定位**（日志路径）+ **建议操作**（处理步骤）+ **告警 ID**（追溯凭证）。

## 7.3.3 告警 ID 的三个用途

告警 ID 格式：`HB-{YYYYMMDD}-{HHMMSS}-{workspace}-{mode}`，每次 critical 告警生成全局唯一 ID。

### 用途一：飞书消息去重
网络抖动可能导致通知重复发送，接收端通过 ID 去重。

### 用途二：日志追溯
用户拿到 ID 后，可以在 `heartbeat.jsonl` 里 `grep "HB-20260429-101523"` 直接定位到该次告警的完整上下文。

### 用途三：WikiBot 蒸馏
每日蒸馏时，WikiBot 以告警 ID 为 key 聚合告警记录，统计频次、分析模式，产出"本周 Top 3 告警模式"之类的知识条目。

## 7.3.4 @相关人员的策略

critical 告警默认 @ 工作区 owner（在 `heartbeat.yaml` 的 `critical_alert_users` 里配置）。

如果 owner 在 30 分钟内未回复任何消息，自动升级 @ 备选联系人。这个升级逻辑通过飞书机器人的 read-receipt 接口实现，不在心跳核心代码里。
