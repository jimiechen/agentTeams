# 7.2 info 级消息格式

## 7.2.1 设计目标：非打扰

info 级的唯一目标是"让用户知道系统在正常工作"。用户看到通知的理想反应是：**扫一眼 → 确认没问题 → 继续原本的工作**。整个过程不超过 3 秒。

这意味着消息必须满足：

- 信息密度高（单屏可见完整内容）
- 无冗余格式（不用分隔线、不用多级标题）
- 无操作指引（因为不需要操作）

## 7.2.2 消息模板

```
🔧 心跳自动恢复

工作区：PMCLI
异常模式：terminal-hang
恢复动作：点击"后台运行"
时间：2026-04-29 10:15:23
```

模板只包含四个字段：**工作区 / 异常模式 / 恢复动作 / 时间**。不加分隔线、不加"此类通知 5 分钟最多 1 条"之类的元信息——这些信息在第一次发送时解释过，后续就是噪音。

## 7.2.3 发送的三个条件

info 通知不是每次恢复成功都发，必须同时满足三个条件：

1. **恢复动作执行成功**（按钮点击 API 返回成功）
2. **下一次心跳检测确认模式回到 healthy**（避免"点击成功但实际未恢复"的误报）
3. **距离上次同类 info 通知超过 5 分钟**（频率节流）

第二个条件尤其重要——点击按钮成功不等于恢复成功，Trae IDE 可能接收了点击但内部状态没变。必须通过下一次心跳的客观观察来闭环确认。

## 7.2.4 发送后的状态登记

每次 info 通知发送后，在 `RecoveryExecutor` 内部的 `lastNotifyTime` Map 里登记时间戳，key 为 `${level}:${workspace}:${mode}`。

这个登记是 7.4 节频率控制的基础数据。
