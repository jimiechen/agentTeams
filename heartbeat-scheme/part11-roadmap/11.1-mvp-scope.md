# 11.1 MVP 范围与 3 天落地计划

## 11.1.1 MVP 核心目标

**先跑起来，不求全。**

MVP 阶段的目标是验证心跳机制的可行性，而非追求完整覆盖。核心原则：

1. **最小可用**：能检测、能恢复、能告警
2. **风险可控**：只处理明确可恢复的场景
3. **快速迭代**：3 天内上线，边跑边修

## 11.1.2 MVP 覆盖范围

### 检测层：仅 Layer 1

| 检测层 | MVP 状态 | 说明 |
|--------|---------|------|
| Layer 1 - 快速检测 | ✅ 接入 | 5 秒轮询，检测 UI 显式状态 |
| Layer 2 - 内容检测 | ❌ 暂缓 | 需要文本 diff 计算，复杂度高 |
| Layer 3 - 深度检测 | ❌ 暂缓 | 需要 CDP 深度注入，风险高 |

**Layer 1 检测项**：

- terminal-hang：检测"后台运行"按钮出现
- modal-blocking：检测"保留/删除"弹窗出现
- task-interrupted：检测任务状态"中断"

### 恢复层：terminal-hang + modal-blocking

| 中断模式 | MVP 状态 | 恢复动作 |
|---------|---------|---------|
| terminal-hang | ✅ 支持 | 点击"后台运行"按钮 |
| modal-blocking | ✅ 支持 | 按策略点击"保留"或"删除" |
| model-stalled | ❌ 暂缓 | 需要 Layer 2 检测 |
| model-queuing | ❌ 不处理 | 正常排队，非异常 |
| task-interrupted | ❌ 不自动恢复 | 仅告警，人工介入 |

### 告警层：基础通知

- 仅支持飞书文本消息
- 区分 info（恢复成功）和 critical（需要人工）
- 不做频率控制（MVP 阶段可接受重复告警）

## 11.1.3 3 天拆解计划

### Day 1：底座接入

**目标**：心跳检测框架跑起来

| 任务 | 预计时间 | 交付物 |
|------|---------|--------|
| HeartbeatDetector 类骨架 | 2h | `HeartbeatDetector.ts` |
| Layer 1 检测逻辑 | 3h | `detectLayer1()` 方法 |
| 与 HeartbeatDaemon 集成 | 2h | 定时调度跑通 |
| 单元测试 | 1h | 检测逻辑测试用例 |

**验收标准**：
- 能在 5 秒内检测到 terminal-hang 和 modal-blocking
- 检测结果正确输出 mode 和 confidence

### Day 2：恢复执行

**目标**：自动恢复动作跑通

| 任务 | 预计时间 | 交付物 |
|------|---------|--------|
| RecoveryExecutor 类骨架 | 1h | `RecoveryExecutor.ts` |
| terminal-hang 恢复动作 | 2h | 点击"后台运行"按钮 |
| modal-blocking 恢复动作 | 2h | 点击策略（默认"保留"） |
| 幂等性保证 | 2h | 同一 mode 5 秒内不重复执行 |
| 飞书通知集成 | 1h | 恢复成功发 info 消息 |

**验收标准**：
- terminal-hang 能自动点击"后台运行"，任务继续
- modal-blocking 能自动点击"保留"，任务继续
- 恢复成功后发送飞书通知

### Day 3：稳定性

**目标**：可上线状态

| 任务 | 预计时间 | 交付物 |
|------|---------|--------|
| 错误处理与重试 | 2h | 恢复失败时的 fallback |
| 日志与指标 | 2h | 关键路径埋点 |
| E2E 测试 | 2h | 手动触发验证 |
| 灰度配置 | 1h | 开关与降级能力 |
| 文档与交接 | 1h | 使用说明 |

**验收标准**：
- 连续运行 24 小时无崩溃
- 手动触发 terminal-hang 和 modal-blocking 均能自动恢复
- 有开关可随时关闭心跳功能

## 11.1.4 MVP 不做的事

明确边界，避免范围蔓延：

| 不做 | 原因 |
|------|------|
| Layer 2/3 检测 | 复杂度高，风险大 |
| model-stalled 恢复 | 依赖 Layer 2 |
| 智能策略选择 | MVP 用固定策略即可 |
| 可观测大盘 | 先跑通再优化 |
| 知识沉淀 | 需要数据积累 |

## 11.1.5 MVP 风险与止损

### 风险点

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 误判正常状态为异常 | 中 | 任务被错误中断 | 增加置信度阈值 |
| 恢复动作失败 | 中 | 任务卡住 | 告警人工介入 |
| CDP 注入影响 Trae | 低 | Trae 不稳定 | 灰度开关快速关闭 |

### 止损点

```yaml
mvp_stop_loss:
  detection_confidence_threshold: 0.9
  recovery_max_attempts: 2
  recovery_cooldown_seconds: 5
  global_kill_switch: "HEARTBEAT_ENABLED=false"
```

**止损触发条件**：

1. 连续 3 次恢复失败
2. 误判导致正常任务被中断
3. Trae 出现异常行为且疑似心跳引起

触发止损后立即设置 `HEARTBEAT_ENABLED=false`，心跳停止工作。
