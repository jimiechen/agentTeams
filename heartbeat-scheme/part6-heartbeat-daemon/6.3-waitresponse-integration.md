# 6.3 与 waitResponse 的集成点

## 集成方案

现有 `wait-response.ts` 的改造只需要两处：

```typescript
// mvp-runner/src/actions/wait-response.ts

export async function waitResponse(ctx: ActionContext): Promise<void> {
  const daemon = new HeartbeatDaemon(
    new HeartbeatDetector(ctx.cdp, ctx.workspace),
    new RecoveryExecutor(ctx.cdp, ctx.lark, ctx.policy, ctx.notifyChatId),
    ctx.workspace,
    ctx.runsDir,
  );

  daemon.start();  // ← 新增：启动心跳

  try {
    // ... 原有的等待响应逻辑 ...
    await waitForCompletion(ctx);
  } finally {
    daemon.stop(); // ← 新增：确保停止心跳
  }
}
```

## 关键设计

### finally 块的重要性
无论任务成功、失败、异常，心跳必须停止，否则会持续占用 CDP 连接触发空转。

### 守护进程生命周期绑定
心跳守护进程的生命周期与一次 `waitResponse` 调用绑定：
- 调用开始时启动
- 调用结束时停止

## 集成流程图

```
waitResponse() 调用
    │
    ▼
创建 HeartbeatDaemon 实例
    │
    ▼
daemon.start()
    │
    ▼
执行原有等待逻辑（waitForCompletion）
    │
    ├─→ 成功
    │       │
    │       ▼
    │   finally 块
    │       │
    │       ▼
    │   daemon.stop()
    │
    ├─→ 失败
    │       │
    │       ▼
    │   finally 块
    │       │
    │       ▼
    │   daemon.stop()
    │
    └─→ 异常
            │
            ▼
        finally 块
            │
            ▼
        daemon.stop()
```

## 上下文传递

### 需要传递的上下文

| 参数 | 类型 | 用途 |
|------|------|------|
| `cdp` | CDPClient | CDP 连接 |
| `workspace` | string | 工作区名称 |
| `lark` | LarkBot | 飞书客户端 |
| `policy` | Policy | 策略配置 |
| `notifyChatId` | string | 通知群聊 ID |
| `runsDir` | string | 日志目录 |

### 上下文获取

从 `ActionContext` 中获取：

```typescript
interface ActionContext {
  cdp: CDPClient;
  workspace: 'PMCLI' | 'DEVCLI';
  lark: LarkBot;
  policy: Policy;
  notifyChatId: string;
  runsDir: string;
  // ... 其他字段
}
```

## 无侵入改造

### 为什么不修改核心逻辑
- 最小化改动范围
- 保持原有逻辑不变
- 降低风险

### 改造范围
只在 `waitResponse` 函数的开头和结尾添加代码，不修改中间的等待逻辑。

## 测试考虑

### 测试场景
1. 正常流程：启动 → 等待完成 → 停止
2. 异常流程：启动 → 异常 → finally 停止
3. 中断流程：启动 → 任务中断 → finally 停止

### 测试验证
- 验证心跳在各种情况下都能正确停止
- 验证 CDP 连接不会被持续占用
