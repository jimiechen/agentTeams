## 3. 根本原因分析

### 3.1 直接原因：心跳检测误判任务状态

日志显示（第 5337 行）：
```
Heartbeat #4: btn=disabled, task=interrupted, terminal=false, delete=false
```

**问题**: 任务实际上正在正常执行（AI正在分析"当前分支还有什么问题"），但心跳检测系统错误地将其判定为 `interrupted`。

### 3.2 关键证据：DOM快照显示存在另一个任务

恢复诊断日志（第 5437 行）显示：
```json
{
  "lastTurns": [
    "PMCLI手动终止输出",
    "14:03用户34614810323现在几点",
    "PMCLI正在分析问题..."
  ]
}
```

**重大发现**: 
- `lastTurns[1]` 显示 `"14:03用户34614810323现在几点"`
- 这说明在 PMCLI 任务执行期间，Trae 界面上存在**另一个对话**（"现在几点"）
- 心跳检测扫描到了这个对话，导致状态判断错误

### 3.3 根因：多任务并发干扰

**核心问题**: 
1. PMCLI 任务正在 slot #3 执行
2. 但 Trae 界面上同时存在另一个对话（"现在几点"）
3. 心跳检测的 `scanTasks()` 扫描到了所有对话，包括不属于当前任务的对话
4. 系统误判当前任务状态为 `interrupted`

## 4. 系统行为分析

### 4.1 正常流程（预期）
```
用户发送消息 → 切换到PMCLI → 填充提示词 → 提交 → 等待响应 → 保存结果 → 回复群聊
```

### 4.2 实际流程（异常）
```
用户发送消息 → 切换到PMCLI → 填充提示词 → 提交 → 
  心跳检测到"interrupted" → 触发恢复 → 
  恢复过程中扫描到"几点"对话 → 
  系统混淆任务 → 最终回复"几点"的结果
```

### 4.3 回复内容错位原因

虽然保存到文件的结果是正确的（"当前分支..."任务的结果），但发送给飞书的回复内容被替换成了"几点"任务的结果。这可能是因为：
1. 恢复机制切换到了错误的对话
2. 或者 `waitResponse` 在恢复后捕获了错误的响应内容

| 影响项 | 严重程度 | 说明 |
|--------|----------|------|
| 任务结果错误 | 高 | 用户收到的是"几点"的回答，而非分支检查 |

| 数据完整性 | 中 | 本地保存的结果是正确的，但群聊回复错误 |
| 系统稳定性 | 中 | 心跳检测误判导致不必要的恢复操作 |

## 6. 修复建议

### 6.1 短期修复（立即实施）

1. **增强心跳检测的准确性**
   - 在 `scanTasks()` 中增加过滤逻辑，只扫描当前选中的任务
   - 避免扫描其他对话的内容

2. **恢复机制增加任务匹配验证**
   - 恢复前验证当前任务名称是否匹配
   - 如果不匹配，不要执行恢复操作

### 6.2 长期修复（后续迭代）

1. **重构任务隔离机制**
   - 每个任务应该有独立的执行上下文
   - 避免不同任务的对话互相干扰

2. **增强状态机逻辑**
   - 增加更多的状态校验点
   - 减少误判的可能性

3. **改进恢复策略**
   - 恢复前进行更全面的诊断
   - 如果检测到多个任务，应该暂停恢复并告警
