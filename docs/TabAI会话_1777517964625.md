## **二、技术修复方案的关键补强**

### **补强 1：必须加入 Cooldown 机制（报告完全遗漏）**

当前的三条修复建议（轮询查找、限制刷新、互斥锁）都是局部优化，无法根治死循环。必须在状态机层加入**恢复冷却期**：

```typescript
// 状态机层新增字段
private lastRecoveryAttemptAt: number = 0;
private readonly RECOVERY_COOLDOWN_MS = 30000; // 30秒冷却

shouldTriggerRecovery(state: HealthState): boolean {
  if (state !== 'frozen') return false;
  
  const elapsed = Date.now() - this.lastRecoveryAttemptAt;
  if (elapsed < this.RECOVERY_COOLDOWN_MS) {
    debug(`Recovery cooldown active, ${this.RECOVERY_COOLDOWN_MS - elapsed}ms remaining`);
    return false;
  }
  
  return true;
}
```

这个机制确保即使重试按钮查找失败，30 秒内不会再次触发恢复，给 UI 充分的渲染时间，也避免编辑器被反复重启。

### **补强 2：互斥锁方案存在竞态条件**

报告给的修复 3 使用了简单的 boolean 标志位：

```typescript
if (this.recoveryLock) { return; }
this.recoveryLock = true;  // ← 检查和赋值之间存在竞态窗口
```

在异步环境下，两个 Promise 可能同时通过 `if` 检查。正确做法是用 **Promise 门闩**：

```typescript
private recoveryPromise: Promise<void> | null = null;

async executeRecovery(state: HealthState): Promise<void> {
  if (this.recoveryPromise) {
    debug('Recovery already in progress, awaiting existing');
    return this.recoveryPromise;
  }
  
  this.recoveryPromise = this.doExecuteRecovery(state)
    .finally(() => { this.recoveryPromise = null; });
  
  return this.recoveryPromise;
}
```

这样多次调用会共享同一个 Promise，彻底消除竞态窗口。

### **补强 3：轮询查找需要结合 MutationObserver**

修复 1 的轮询实现是 5 次 × 1 秒的定时轮询，最坏情况需要 5 秒才能确认按钮存在。更优方案是**MutationObserver + 超时兜底**：

```typescript
private async clickRetryButtonWithObserver(timeoutMs = 5000): Promise<RetryResult> {
  const script = `
    new Promise((resolve) => {
      const tryClick = () => {
        const btn = document.querySelector('button[aria-label="重试"]');
        if (btn) {
          btn.click();
          resolve({ clicked: true, method: 'aria-label-observer' });
          return true;
        }
        return false;
      };
      
      if (tryClick()) return;
      
      const observer = new MutationObserver(() => {
        if (tryClick()) observer.disconnect();
      });
      observer.observe(document.body, { childList: true, subtree: true });
      
      setTimeout(() => {
        observer.disconnect();
        resolve({ clicked: false, method: 'timeout-after-observer' });
      }, ${timeoutMs});
    })
  `;
  return await this.cdp.evaluate(script);
}
```

响应时间从最差 5 秒降到 DOM 变更触发的毫秒级，同时避免无效轮询带来的 CDP 调用压力。

### **补强 4：刷新页面限制需要持久化**

修复 2 把 `refreshCount` 作为实例变量：

```typescript
private refreshCount = 0;  // ← 编辑器重启后归零
```

编辑器重启本身就是刷新的结果之一，重启后进程重新启动时 `refreshCount` 会被重置为 0，相当于限制失效。必须**持久化到文件或 localStorage**，并带时间窗口：

```typescript
// 使用持久化存储，1小时窗口
private async getRefreshCountInWindow(windowMs = 3600000): Promise<number> {
  const records = await this.storage.read('refresh-records.json') || [];
  const cutoff = Date.now() - windowMs;
  return records.filter(t => t > cutoff).length;
}
```

### **补强 5：缺少"完全放弃"的熔断策略**

报告的所有修复都假设"继续尝试恢复"，但某些场景下**放弃恢复并告警人工介入**才是更负责任的策略。建议加入三级熔断：

```
Level 1: 单次恢复失败 → 30 秒 cooldown 后重试
Level 2: 连续 3 次恢复失败 → 暂停该任务的心跳检测，飞书 @运维
Level 3: 编辑器重启累计 2 次 → 完全停止 daemon，输出人工介入指引
```

这比无休止尝试直到触发速率限制要优雅得多。
