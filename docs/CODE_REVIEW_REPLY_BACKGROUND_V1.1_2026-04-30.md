## 二、v1.1 关键变更评审

### 2.1 后台运行后等待：2秒 → 30秒 ✅ 同意

**变更理由**:
- 任务切后台的业务流程（停止输出流、移动队列、持久化状态、释放资源）P95耗时10~20秒
- UI动画与过渡（CSS动画、按钮状态切换、任务列表重排）累计3~5秒
- 异步状态同步（WebSocket/IPC消息队列积压）P99可达15~25秒

**三类耗时叠加P99总时长25~30秒**，因此30秒默认值合理。

**评审意见**: 同意。但强调这30秒是**上限**，实际通过动态等待可能只需5~15秒。

### 2.2 动态等待机制 ✅ 核心亮点

```typescript
private async waitForButton(
  buttonMatcher: string,
  maxWaitMs: number,      // 默认30000
  minWaitMs: number = 5000,  // 先等5秒
  pollIntervalMs: number = 1000  // 然后每秒轮询
): Promise<{ ready: boolean; actualWaitMs: number }> {
  // Step 1: 保守下限等待5秒
  await this.delay(minWaitMs);
  
  // Step 2: 轮询检测按钮是否就绪
  while (Date.now() - startTime < maxWaitMs) {
    const ready = await this.cdp.evaluate(`...`);
    if (ready) return { ready: true, actualWaitMs: Date.now() - startTime };
    await this.delay(pollIntervalMs);
  }
  
  return { ready: false, actualWaitMs: maxWaitMs };
}
```

**评审意见**: 这是v1.1最优秀的设计改进。

**优势**:
1. **快的环境**: 如果按钮5秒内就绪，实际只等5秒，不浪费25秒
2. **慢的环境**: 如果按钮需要25秒，轮询会等到它就绪，不误判
3. **数据收集**: `actualWaitMs`为后续调参提供真实数据

**建议补充**: 
- 轮询检测应检查按钮的`disabled`状态和可见性（`offsetParent !== null`）
- 建议添加`getBoundingClientRect().width > 0`确保按钮真实可点击

### 2.3 取消后等待：10秒 ✅ 同意

**作用**: 等待状态从`background`切换到`interrupted`，让重试按钮渲染。

**评审意见**: 10秒合理。这是给Trae状态机完成切换的时间，与之前的2秒相比更稳健。

## 三、核心逻辑评审

### 3.1 恢复流程

```
[background 状态]
      ↓
  超时判定（5分钟无DOM变化）
      ↓
  点击"后台运行"按钮
      ↓
  动态等待"取消"按钮就绪（5秒下限 + 轮询，上限30秒）
      ↓
  二次确认：DOM是否恢复活跃（若活跃则跳过取消）
      ↓
  点击"取消"按钮 → 任务进入interrupted
      ↓
  等待10秒（状态切换 + 重试按钮渲染）
      ↓
  复用现有executeFrozenRecovery
      ↓
  任务重新运行
```

**评审意见**: 流程完整，包含所有必要步骤。

**关键保护点**:
1. 动态等待 ✅ 不浪费快的环境时间
2. 二次确认 ✅ 防止等待期内任务自行恢复被误伤
3. 熔断机制 ✅ 同一任务最多2次
4. Cooldown ✅ 60秒防止连续触发

### 3.2 状态转换

```
normal
  └─→ background（检测到后台运行按钮）
        ├─→ normal（DOM重新活跃）
        ├─→ interrupted（触发超时恢复后）
        └─→ human-required（熔断后升级）

interrupted
  └─→ normal（复用现有frozen恢复路径）
```

**评审意见**: 状态机保持简洁，不新增`suspicious`、`probing`等中间状态。

### 4.2 并发控制

**评审意见**: 必须与现有互斥锁（recoveryLock）共享。

**注意点**: 动态等待期间持有锁会阻塞其他恢复动作30秒。这是可接受的，因为：
1. background恢复优先级最高
2. 30秒是上限，实际可能只需5~15秒
3. 其他恢复动作（如frozen）通常也在等待状态稳定
