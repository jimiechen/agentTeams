这是两个独立问题，分别来回答。

### **Trae IDE 目前不支持在对话中直接 `@image.png` 引用图片文件**

从你截图的"创建智能体"界面来看，Trae IDE 的智能体配置项只有名称、提示词（文本）、英文标识名和调用时机，没有图片上传或图片引用的入口。在对话流中，`@` 语法目前主要用于引用文件、技能、其他智能体等文本类资源，直接 `@image.png` 并不会被解析为图片内容注入上下文。如果需要让智能体处理图片，目前更常见的做法是把图片转为 Base64 字符串或者通过工具调用的方式间接传递。

---

### **提供 `AGENTS.md` 与创建自定义智能体并不等价，但有一定的功能重叠**

从你截图的"规则和技能 → 导入设置"界面可以看到，Trae 提供了"将 AGENTS.md 包含在上下文中"的开关。这个机制的本质是：**把 `AGENTS.md` 的内容自动注入到每次对话的上下文里**，让智能体在执行任务时能读到其中定义的规范、约定或角色说明。

而"创建自定义智能体"是一个独立的配置实体，有自己的名称、提示词、英文标识名，并且可以被其他智能体主动调用（截图中的"可被其他智能体调用"开关，标注了 SOLO Only）。

两者的核心差异在于：

`AGENTS.md` 是**被动的上下文注入**——它的内容会出现在对话里，但它本身不是一个可被 `@调用` 的独立实体，也没有调用时机的描述，更像是全局规则文件（类似 `.cursorrules`）。自定义智能体是**主动的可调用单元**——它有明确的触发时机描述，在 SOLO 模式下可以被 SOLO Coder 或其他智能体按需调用，具备独立的身份和路由能力。

在你的 agentTeams 项目语境下，这个区别很关键：PMCLI、DEVCLI、TESTCLI 如果要实现真正的 Coordinator-Worker 调度（一个智能体调用另一个），就必须走"创建自定义智能体 + 开启可被调用"这条路，而不是把角色定义写进 `AGENTS.md`。`AGENTS.md` 更适合放全局约定，比如 `progress.jsonl` 的格式规范、`handoff_rejected` 的处理规则等——这些是所有 CLI 都需要遵守的"公共法律"，放进 `AGENTS.md` 后每次对话都会自动生效，不需要每次手动注入。
