### **问题4：蒸馏失败处理**

评审文档的建议方向是对的，但需要更具体的分级处理策略：

**第一级（蒸馏服务不可用）**：跳过当天蒸馏，把当天的 Layer 0 文件标记为 `pending`，次日凌晨蒸馏时合并处理当天和昨天的记录，生成一份合并的 Layer 1 文件（命名为 `2026-04-28_2026-04-29.md`），压缩比会下降但不丢数据。

**第二级（蒸馏质量差）**：需要定义"质量差"的判断标准——输出字数 < 50 字（说明 AI 几乎没有提炼内容）或输出里缺少五维度中的三个以上（说明格式崩坏）。满足任一条件时，自动降级为"原始摘要模式"：直接截取当天所有 runs 文件的第一段（Prompt 行），拼成一份简版 Layer 1，标记为 `[低质量]`，至少保证有内容可注入。

**第三级（连续3天失败）**：发飞书告警，人工介入。

## **评审文档本身的几处遗漏**

**第一，集成点2的改动描述不完整。** 文档里写"在 fillPrompt 前插入 context building 步骤"，但没有说明这个步骤是同步还是异步。如果 `wiki-inject` 是同步读文件，它会阻塞 fillPrompt 的执行路径，在高频触发场景下会累积延迟。应该明确：`wiki-inject` 必须是同步内存读取（依赖上一轮提到的缓存机制），文件 I/O 只在缓存失效时发生，不在每次 fillPrompt 的关键路径上。

**第三，评审结论里"建议修改第1条"有歧义。** 文档写"增加'其他重要信息'作为第五维度的兜底"，但原设计已经有五个维度，加了之后是第六维度。上一轮评审建议增加的是"性能基线"维度，而不是一个模糊的兜底桶。"其他重要信息"这类开放式维度会导致蒸馏输出不可预测，AI 会把什么都往里塞，反而降低 Layer 1 的结构化质量。建议把这条修改意见改为"增加性能基线维度，记录关键指标均值和与历史基线的偏差"，有明确的信息类型约束。
