## 二、详细评审

### 2.1 第 1 章：任务规格说明（PMCLI）

**优点**：
- ✅ 函数签名清晰：`greet(name: string): string`
- ✅ 行为规格明确：空字符串、null、undefined 均回退
- ✅ 验收标准（AC）量化：5 个 AC 覆盖所有场景
- ✅ 交付物明确：`src/greet.ts` + `src/greet.test.ts`

**建议**：
- 💡 增加非功能性要求：如性能要求（执行时间 < 1ms）、代码风格要求
- 💡 增加错误处理要求：如传入数字类型时的行为

### 2.2 第 2 章：代码实现（DEVCLI）

**优点**：
- ✅ 实现简洁，核心逻辑清晰
- ✅ 故意缺陷设计合理，可被检测

**问题**：

#### 问题 1：缺陷类型为"被动缺陷"（建议改进）

**当前设计**：
```typescript
// 缺陷：测试用例被注释掉
// test('AC-3: null 返回默认问候', () => {
//   expect(greet(null as any)).toBe("Hello, World!");
// });
```

**问题**：TESTCLI 需要靠检查覆盖率才能发现，不够直观。

**建议改为"主动逻辑缺陷"**：
```typescript
// 更好的缺陷：逻辑错误，测试会直接 FAIL
export function greet(name: string): string {
  if (!name) {
    return "Hello, World!";
  }
  // ⚠️ 故意缺陷：多了一个感叹号
  return `Hello, ${name}!!`;  // 应该是单个 !
}
```

**优势**：
- TESTCLI 执行测试时，`expect(greet("Alice")).toBe("Hello, Alice!")` 会直接 FAIL
- 缺陷发现更直观，更有冲击力
- 验证 TESTCLI 的测试执行能力，而不仅是覆盖率检查能力

#### 问题 2：函数签名类型安全

**当前设计**：
```typescript
function greet(name: string): string
```

**问题**：未使用联合类型 `(name: string | null | undefined)`，在 TypeScript strict 模式下会报错。

**建议**：
- 如果演示目标是验证类型安全，应该要求使用联合类型
- 或者明确说明："本演示不验证 TypeScript 类型安全，仅验证运行时行为"

### 2.3 第 3 章：测试报告（TESTCLI）

**优点**：
- ✅ 独立执行测试，无提示发现缺陷
- ✅ 缺陷报告结构化：ID、严重程度、复现步骤、根本原因、修复建议
- ✅ 验收结论明确：通过/不通过，量化统计

**建议**：
- 💡 增加测试执行环境信息：Node.js 版本、Jest 版本、TypeScript 配置
- 💡 增加测试执行时间：每个用例的执行耗时
- 💡 增加覆盖率详细报告：哪些行未覆盖

### 2.4 第 4 章：总结汇报（PMCLI）

**优点**：
- ✅ 执行摘要清晰：4 个阶段，3 个 CLI，总耗时
- ✅ 缺陷闭环状态：发现方、当前状态、修复建议
- ✅ 演示验证结论：通过的能力 + 改进点

**建议**：
- 💡 增加风险分析：如"如果 DEVCLI 未制造缺陷，TESTCLI 如何验证？"
- 💡 增加可复现性说明：如何在其他环境复现本演示

## 三、关键问题

### 3.1 缺陷设计策略

**当前策略**：被动缺陷（注释测试用例）
**问题**：
1. TESTCLI 需要人工检查覆盖率，不能自动发现
2. 不符合"TESTCLI 必须独立发现该缺陷，不能被提示"的约束

**建议策略**：主动逻辑缺陷（函数返回值错误）
**优势**：
1. TESTCLI 执行测试时自动 FAIL
2. 完全独立发现，无需人工干预
3. 更符合实际开发场景

### 3.2 验收标准 AC-5

**当前要求**："单元测试覆盖所有 AC"
**问题**：如果 DEVCLI 制造主动逻辑缺陷，AC-5 应该改为"单元测试全部通过"

**建议修改**：
```markdown
| AC-5 | 测试通过率 | 所有单元测试通过 |
```

### 3.3 飞书通知内容

**当前通知**：
```
[agentTeams] 🟡 PRD-DEMO-001 第 2 章完成
⚠️ 注意：实现包含已知类型处理差异，请 TESTCLI 重点验证边界用例
```

**问题**：通知中提示了"类型处理差异"，相当于给 TESTCLI 提示了缺陷方向。

**建议**：
```
[agentTeams] 🟡 PRD-DEMO-001 第 2 章完成
内容：greet 函数实现完成
交接：→ TESTCLI（执行测试验收）
```

删除提示信息，确保 TESTCLI 完全独立发现缺陷。

## **关于中优先级建议的处理意见**

评审意见中建议增加非功能性要求（执行时间 < 1ms、TypeScript strict 模式）和完善测试报告（环境信息、耗时、覆盖率详情），这两条对于**演示目的**来说是合理的，但建议放到 v1.2 而不是 v1.1。原因是 v1.1 的核心目标是修复缺陷设计策略，保持变更范围最小，避免在评审修复过程中引入新的不确定性。非功能性要求的加入会要求 TESTCLI 新增性能测试用例，相当于扩大了演示范围，适合在 v1.1 跑通之后单独迭代。
