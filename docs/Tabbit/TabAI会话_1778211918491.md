## **评审意见审阅与采纳决策**

| 评审项 | 等级 | 采纳决策 | 处理方式 |
|-------|------|---------|---------|
| `docx.document.rawContent` 接口不存在 | 严重 | **完全采纳** | 改用 `drive.file.upload_all` 直接上传 .md 文件，不再走"创建文档 + 写入内容"两步 |
| 卡片按钮 url 多客户端兼容 | 中等 | **采纳** | 按钮新增 `multi_open_url` 字段 |
| 缺少 `rootFolderToken` 和 `uploadEnabled` 校验 | 低 | **采纳** | 启动时强校验配置，缺失则降级到 post 模式 |
| 新增 `reply_mode` 配置（post/card/hybrid） | 中等 | **采纳，强化** | 默认 `hybrid`，上传失败自动降级为 post |
| `p-queue` 限流并发 3 | 中等 | **采纳** | 在 `LarkDocUploader` 内部封装，对外透明 |
| 本地文件结构 `runs/{run-id}/response.md` + `meta.json` + `prompt.md` | 低 | **采纳** | 与原方案一致，确认保留 |

评审中**唯一需要补充**的是：评审建议"验证 `drive.file.upload_all` API 可用性，通过后即可批准实施"——这一步需要在 Step 0 完成，作为整个改造的前置门禁。

## **使用说明**

把上面三引号包裹的整段内容复制后，作为新一轮对话的首条消息发给下一个 Trae 实例（DEVCLI 角色）。提示词的关键设计有三点值得说明：

**第一**，Step 0 是新增的前置门禁。原方案 v1.0 直接进入开发，但评审已经证明"先写代码再发现接口不存在"会浪费大量时间。Step 0 用一个 30 分钟的小脚本验证三个核心 API（upload_all / create_folder / list），通过后才进入正式开发，相当于把风险前置。

**第二**，`reply_mode` 的 `hybrid` 默认模式吸收了评审建议中的"渐进式改造"思路。这意味着即使飞书侧出现任何问题（token 失效、API 限流、网络抖动），Trae 任务依然能用旧的 `replyPost` 完成回复，只是格式可能略乱——这比"完全不回复"好得多。

**第三**，Step 6 的五个端到端用例覆盖了之前出问题的所有场景：纯文本、代码块、表格、降级、深层路径。每个用例的预期结果都是可观测的（飞书可见、按钮可点、本地有文件），方便人工或后续 TESTCLI 验收。

下一个 Trae 拿到这份提示词后，理论上应该能在大约 4 小时内（含 Step 0 验证）完整产出可上线的飞书通知优化实现。
