### **核心优化方案：将飞书回复从"纯文本/富文本"改为"上传 Markdown 文件到飞书云文档 + 群消息附带文档卡片链接"，文档路径与本地仓库路径完全镜像**

当前的问题根源在 `replyPost()` 方法把所有响应内容塞进飞书 post 消息体——一旦内容包含代码块、表格、嵌套列表，飞书 post 的 schema 渲染就会错乱（因为 post 消息只支持有限的 tag 类型，不支持 Markdown 语法）。正确做法是把内容写成 Markdown 文件，先存本地、再上传飞书云文档，群里只发一条"卡片消息"指向云文档链接。

下面是给本地 Trae IDE 的完整改造指令，可以直接交给 DEVCLI 执行。

## **一、改造目标与不变项**

**改造目标**：

1. 飞书群消息不再承载长文本内容，只承载"任务完成 + 文档链接 + 摘要"三件事
2. 完整响应内容写入本地 `runs/{run-id}/response.md`，同步上传到飞书云文档 `agentTeams空间 > runs > {run-id} > response.md`
3. 飞书云文档路径与本地相对路径完全镜像，群聊消息中点击链接即可在飞书内阅读完整 Markdown

**保持不变**：

- `LarkBotClient` 的连接、认证、消息接收逻辑
- `MultiTaskRunner.handle()` 的整体流程（白名单、指令解析、任务调度、心跳检测）
- `withChatMutex` 互斥锁机制
- `persist()` 本地落盘逻辑（在此基础上扩展）

## **三、新增模块：`markdown-formatter.ts`**

负责把 Trae 返回的原始响应统一格式化为标准 Markdown。

```typescript
// src/markdown-formatter.ts

interface ResponseMetadata {
  runId: string;
  taskSlot: string;
  taskName?: string;
  senderId: string;
  promptText: string;
  responseText: string;
  durationMs: number;
  startedAt: string;
  finishedAt: string;
}

export function formatResponseToMarkdown(meta: ResponseMetadata): string {
  return `# Trae 任务执行报告

> **Run ID**: \`${meta.runId}\`  
> **任务槽**: \`${meta.taskSlot}\`  
> **任务名**: \`${meta.taskName ?? '未命名'}\`  
> **发起人**: \`${meta.senderId}\`  
> **开始时间**: ${meta.startedAt}  
> **结束时间**: ${meta.finishedAt}  
> **耗时**: ${(meta.durationMs / 1000).toFixed(1)}s

---

## 原始提示词

\`\`\`text
${meta.promptText}
\`\`\`

---

## AI 响应内容

${meta.responseText}

---

## 元数据

\`\`\`json
${JSON.stringify({ 
  runId: meta.runId, 
  taskSlot: meta.taskSlot,
  durationMs: meta.durationMs 
}, null, 2)}
\`\`\`
`;
}

export function extractSummary(responseText: string, maxChars = 200): string {
  // 提取响应内容的前 N 字作为群消息摘要
  const cleaned = responseText
    .replace(/```[\s\S]*?```/g, '[代码块]')
    .replace(/!\[.*?\]\(.*?\)/g, '[图片]')
    .replace(/\n{2,}/g, ' ')
    .trim();
  return cleaned.length > maxChars 
    ? cleaned.slice(0, maxChars) + '...' 
    : cleaned;
}
```

## **四、新增模块：`lark-doc-uploader.ts`**

封装飞书云文档上传逻辑，关键是**路径到 token 的映射**。

```typescript
// src/lark-doc-uploader.ts

import { Client } from '@larksuiteoapi/node-sdk';
import * as fs from 'fs/promises';
import * as path from 'path';

interface UploadResult {
  docToken: string;
  docUrl: string;
  relativePath: string;
}

export class LarkDocUploader {
  private client: Client;
  private rootFolderToken: string;
  private pathTokenCache = new Map<string, string>();

  constructor(client: Client, rootFolderToken: string) {
    this.client = client;
    this.rootFolderToken = rootFolderToken;
  }

  /**
   * 上传 Markdown 文件到飞书云文档
   * 自动按相对路径创建/查找父目录，与本地路径镜像
   */
  async uploadMarkdown(
    localFilePath: string,
    relativePath: string
  ): Promise<UploadResult> {
    const content = await fs.readFile(localFilePath, 'utf-8');
    
    // 1. 解析路径，逐级创建/查找文件夹
    const parts = relativePath.split('/').filter(Boolean);
    const fileName = parts.pop()!;
    const folderToken = await this.ensureFolderPath(parts);
    
    // 2. 创建飞书云文档
    const docResp = await this.client.docx.document.create({
      data: {
        folder_token: folderToken,
        title: fileName.replace(/\.md$/, ''),
      },
    });
    const docToken = docResp.data?.document?.document_id;
    if (!docToken) throw new Error('飞书云文档创建失败');
    
    // 3. 写入 Markdown 内容（通过 import 接口）
    await this.client.docx.document.rawContent({
      path: { document_id: docToken },
      data: { content },  // 飞书 docx API 支持直接写入 Markdown
    });
    
    return {
      docToken,
      docUrl: `https://feishu.cn/docx/${docToken}`,
      relativePath,
    };
  }

  /**
   * 按路径数组逐级查找或创建文件夹
   * 例如 ['runs', 'run-20260508-001'] 会在飞书中创建对应的目录结构
   */
  private async ensureFolderPath(parts: string[]): Promise<string> {
    let currentToken = this.rootFolderToken;
    let cumulativePath = '';
    
    for (const part of parts) {
      cumulativePath = cumulativePath ? `${cumulativePath}/${part}` : part;
      
      // 缓存命中
      if (this.pathTokenCache.has(cumulativePath)) {
        currentToken = this.pathTokenCache.get(cumulativePath)!;
        continue;
      }
      
      // 查询当前目录下是否已存在该子文件夹
      const listResp = await this.client.drive.file.listByFolder({
        params: { folder_token: currentToken },
      });
      const existing = listResp.data?.files?.find(
        f => f.name === part && f.type === 'folder'
      );
      
      if (existing) {
        currentToken = existing.token!;
      } else {
        // 创建新文件夹
        const createResp = await this.client.drive.file.createFolder({
          data: { name: part, folder_token: currentToken },
        });
        currentToken = createResp.data?.token!;
      }
      
      this.pathTokenCache.set(cumulativePath, currentToken);
    }
    
    return currentToken;
  }
}
```

## **五、改造 `lark-client.ts`：新增卡片消息发送**

在现有 `LarkBotClient` 上新增一个方法，用飞书"消息卡片"格式发送任务完成通知：

```typescript
// 新增方法：发送任务完成卡片
async sendTaskCompleteCard(
  messageId: string,
  params: {
    runId: string;
    taskName: string;
    durationSec: number;
    docUrl: string;
    relativePath: string;
    summary: string;
    status: 'success' | 'failed' | 'timeout';
  }
): Promise<void> {
  const statusEmoji = {
    success: '✅',
    failed: '❌',
    timeout: '⏱️',
  }[params.status];
  
  const card = {
    config: { wide_screen_mode: true },
    header: {
      title: { tag: 'plain_text', content: `${statusEmoji} 任务完成: ${params.taskName}` },
      template: params.status === 'success' ? 'green' : 'red',
    },
    elements: [
      {
        tag: 'div',
        fields: [
          { is_short: true, text: { tag: 'lark_md', content: `**Run ID**\n\`${params.runId}\`` } },
          { is_short: true, text: { tag: 'lark_md', content: `**耗时**\n${params.durationSec}s` } },
          { is_short: false, text: { tag: 'lark_md', content: `**相对路径**\n\`${params.relativePath}\`` } },
        ],
      },
      { tag: 'hr' },
      {
        tag: 'div',
        text: { tag: 'lark_md', content: `**摘要**\n${params.summary}` },
      },
      { tag: 'hr' },
      {
        tag: 'action',
        actions: [
          {
            tag: 'button',
            text: { tag: 'plain_text', content: '📄 查看完整报告' },
            url: params.docUrl,
            type: 'primary',
          },
        ],
      },
    ],
  };
  
  await this.client.im.message.reply({
    path: { message_id: messageId },
    data: {
      msg_type: 'interactive',
      content: JSON.stringify(card),
    },
  });
}
```

卡片消息的优势在于飞书原生支持渲染，按钮、字段、分隔线都是结构化元素，不会因为内容里有代码块或表格而错乱。

## **六、改造 `runner-multi.ts`：流程整合**

修改 `handle()` 方法的"4.5 持久化 + 4.6 回复"两步，串联 Markdown 生成、飞书上传、卡片发送：

```typescript
// runner-multi.ts 关键改动（仅展示新增/修改部分）

import { formatResponseToMarkdown, extractSummary } from './markdown-formatter';
import { LarkDocUploader } from './lark-doc-uploader';

export class MultiTaskRunner {
  private docUploader: LarkDocUploader;
  
  constructor(/* ... */) {
    // 初始化时构造 uploader
    this.docUploader = new LarkDocUploader(
      sharedLarkClient,
      this.cfg.lark.root_folder_token  // 配置项：飞书空间根目录 token
    );
  }
  
  handle = async (msg: LarkInbound, botKeyword: string): Promise<void> => {
    // ... 1-3 步保持不变 ...
    
    await withChatMutex(`run-${runId}`, async () => {
      // 4.1-4.4 保持不变
      await switchTask(this.cdp, targetSlot);
      await fillPrompt(this.cdp, parsed.prompt);
      await submit(this.cdp);
      await this.sendByKeyword(botKeyword, `📤 已提交，等待 AI 响应…`);
      
      const startedAt = new Date();
      const response = await waitResponse(this.cdp, {
        timeoutMs: this.cfg.pmbot.response_timeout_ms,
        taskName: targetTaskName || undefined,
      });
      const finishedAt = new Date();
      const duration = finishedAt.getTime() - startedAt.getTime();
      
      // ===== 4.5 新版持久化 =====
      const relativePath = `runs/${runId}/response.md`;
      const localPath = path.join(this.cfg.runs_dir, runId, 'response.md');
      
      // 4.5.1 生成标准 Markdown
      const markdownContent = formatResponseToMarkdown({
        runId,
        taskSlot: targetSlot,
        taskName: targetTaskName,
        senderId: msg.senderId,
        promptText: parsed.prompt,
        responseText: response.text,
        durationMs: duration,
        startedAt: startedAt.toISOString(),
        finishedAt: finishedAt.toISOString(),
      });
      
      // 4.5.2 写本地
      await fs.mkdir(path.dirname(localPath), { recursive: true });
      await fs.writeFile(localPath, markdownContent, 'utf-8');
      
      // 4.5.3 同步本地 meta.json + prompt.md（用于完整审计）
      await fs.writeFile(
        path.join(this.cfg.runs_dir, runId, 'meta.json'),
        JSON.stringify({ runId, taskSlot: targetSlot, durationMs: duration, status: 'success' }, null, 2)
      );
      await fs.writeFile(
        path.join(this.cfg.runs_dir, runId, 'prompt.md'),
        `# Prompt\n\n${parsed.prompt}\n`
      );
      
      // ===== 4.6 上传飞书 + 发卡片 =====
      let uploadResult;
      try {
        uploadResult = await this.docUploader.uploadMarkdown(localPath, relativePath);
      } catch (err) {
        // 上传失败时降级：仍然发卡片，但 url 改为本地路径提示
        console.error('[lark-upload] failed:', err);
        await this.replyByKeyword(botKeyword, msg.messageId,
          `⚠️ 任务完成但飞书上传失败，请查看本地: ${relativePath}`
        );
        return;
      }
      
      // 4.6.1 发送卡片消息
      const bot = this.bots.find(b => b.keyword === botKeyword);
      await bot?.sendTaskCompleteCard(msg.messageId, {
        runId,
        taskName: targetTaskName ?? targetSlot,
        durationSec: Math.round(duration / 1000),
        docUrl: uploadResult.docUrl,
        relativePath: uploadResult.relativePath,
        summary: extractSummary(response.text, 200),
        status: 'success',
      });
    });
  };
}
```

## **九、关键风险与应对**

**风险一：飞书 API 限流**。飞书云文档创建接口有 QPS 限制（通常每秒 5 次），高频任务会触发限流。应对方式是在 `LarkDocUploader` 里加一个简单的 `p-queue` 限流器，并发上限设为 3。

**风险二：飞书路径不支持深层嵌套**。如果 `runs/{run-id}/` 这种两级目录可以正常创建，更深的层级（如 `docs/prd/{prd-id}/chapters/`）需要先验证。建议在 Step 6 之前单独跑一个深度测试用例。

**风险三：上传超时阻塞主流程**。当前改造方案里上传失败会降级为"只发本地路径提示"，但上传成功路径是同步等待的。如果飞书 API 慢，会拖慢整个任务的完成响应。可选优化是把上传放到 `setImmediate` 异步执行，先发"任务完成（链接生成中）"消息，上传成功后再编辑消息追加链接——但这是 V2 优化，V1 不做。

按这个方案改造完成后，飞书群消息将彻底告别"格式紊乱"问题：群里只看到结构化卡片，详细内容全部在云文档中以原生 Markdown 渲染，本地仓库与飞书空间的相对路径完全镜像，任何人拿到一个路径都能在两地找到同一份文件。
