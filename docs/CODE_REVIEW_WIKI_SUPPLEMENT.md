## 一、文档A评审：WikiBot 任务接力协议

### 1.1 方案概述

在现有 WikiBot 架构基础上，增加**任务接力**能力：
- PMCLI/DEVCLI 任务完成后自动通知 WikiBot
- WikiBot 蒸馏完成后，分析任务结果生成**接力提示词**
- 将接力提示词发回群聊 @下一个 Bot，形成任务链

### 1.2 可行性评估

| 维度 | 评估 | 说明 |
|------|------|------|
| **技术可行性** | ✅ 高 | 复用现有飞书消息通道，无需新增基础设施 |
| **与现有架构兼容性** | ⚠️ 中 | 需要修改 runner-multi.ts 增加完成通知发送逻辑 |
| **实现复杂度** | 中等 | 核心难点在防循环机制，其他为流程编排 |
| **风险等级** | 中 | 存在消息风暴和无限循环风险，需严格防护 |

### 1.3 详细评审意见

#### ✅ 优势

1. **自然延伸现有架构**
   - 飞书群聊已经是通信总线，不需要新增消息队列或RPC
   - WikiBot 已经能接收 @mention，只需扩展 after_distill 阶段逻辑

2. **价值明确**
   - 将 WikiBot 从"被动蒸馏工具"升级为"主动协调中枢"
   - PMCLI/DEVCLI 专注执行，全局协调由 WikiBot 承担
   - 减少人工在群聊中手动发"下一步任务"的操作

3. **消息协议设计合理**
   - `#wiki-distill #task-relay` 标签机制清晰，便于路由
   - 接力提示词模板结构化，包含背景/任务/约束/预期产出

#### ⚠️ 问题与风险

| # | 问题 | 严重程度 | 建议 |
|---|------|---------|------|
| 1 | **防循环机制依赖 Prompt 约束** | 🔴 高 | 三道防护中两道依赖 LLM 正确理解 Prompt，不可靠。应增加代码级硬约束：在 WikiBotHandler 中检查消息是否含 "WikiBot 生成" 标记，如果是则强制不生成接力。 |
| 2 | **接力提示词质量不稳定** | 🟡 中 | LLM 生成的接力任务可能过于笼统或错误。建议增加"质量门"：接力提示词必须包含具体文件路径/模块名，否则降级为 `none`。 |
| 3 | **消息来源白名单未实现** | 🟡 中 | `project_rules.md` 中声明了白名单，但代码层未实现。应在 LarkBot 消息处理器中增加 sender 类型判断（Bot vs 用户）。 |
| 4 | **任务完成通知阻塞主流程** | 🟡 中 | runner-multi.ts 中发送完成通知是同步调用，如果飞书 API 延迟会影响任务响应时间。建议改为异步发送（fire-and-forget）。 |
| 5 | **接力目标 Bot 路由不明确** | 🟡 中 | 文档说"根据任务类型决定接力给 PMCLI 或 DEVCLI"，但判断逻辑依赖 LLM，可能出错。建议增加明确规则：含代码/修复/Bug 关键词 → DEVCLI，其他 → PMCLI。 |
| 6 | **未考虑接力任务失败场景** | 🟡 中 | 如果接力任务执行失败，是否需要再次接力？文档未说明。建议：接力任务失败不自动重试，只记录到 distill-errors.log。 |

#### 🔴 关键缺陷：防循环机制不够硬

当前方案的三道防护：
1. Prompt 约束（软）— LLM 可能不遵守
2. `#task-relay` 标签（软）— 可被伪造
3. `project_rules.md` 白名单（软）— 无代码 enforcement

**建议增加硬约束**：
```typescript
// 在 WikiBotHandler.shouldHandle 中
if (msg.text.includes('WikiBot 自动生成')) {
  return false; // 不处理接力消息，避免循环
}
```

### 1.4 实施建议

| 阶段 | 内容 | 优先级 |
|------|------|--------|
| Day 1 | 在 runner-multi.ts 中增加异步任务完成通知 | 🔴 高 |
| Day 2 | 在 WikiBotHandler 中增加硬防循环约束 | 🔴 高 |
| Day 3 | 实现 wiki-relay skill（生成接力提示词） | 🟡 中 |
| Day 4 | 端到端测试完整接力链 | 🟡 中 |
| Day 5 | 观察并调整接力提示词质量 | 🟢 低 |

### 1.5 评审结论

**有条件通过** ✅

必须修复：
1. 增加代码级硬防循环约束（不能仅靠 Prompt）
2. 任务完成通知改为异步发送
3. 在 LarkBot 层实现 Bot 消息白名单过滤

## 二、文档B评审：原生 Skills 钩子方案

### 2.1 方案概述

不依赖外部脚本和定时任务，直接利用 Trae IDE 的 `.trae/skills/` 机制：
- **钩子1（before_task）**: 任务开始前触发 `wiki-inject`，将记忆注入 prompt
- **钩子2（after_task）**: 任务完成后触发 `wiki-distill`，实时蒸馏为今日记忆
- **钩子3（周日条件）**: 额外触发 `wiki-merge`，合并本周记忆

### 2.2 可行性评估

| 维度 | 评估 | 说明 |
|------|------|------|
| **技术可行性** | ⚠️ 低 | Trae Skills 的 `trigger: before_task` / `after_task` 在当前版本中**未确认存在** |
| **与现有架构兼容性** | 🔴 低 | 与已实施的 Phase 1-4 架构冲突，需要废弃外部脚本方案 |
| **实现复杂度** | 低（如支持）| 纯 Prompt 指令，无需代码 |
| **风险等级** | 高 | 依赖 Trae 未公开/未稳定的 Skills 钩子机制 |

### 2.3 详细评审意见

#### ✅ 优势（假设 Skills 钩子存在）

1. **极简实现**
   - 3 个 `skill.md` 文件替代 3 个 TypeScript 文件 + 调度脚本
   - 无需 crontab、无需环境变量配置、无需独立 WikiBot 飞书应用

2. **实时蒸馏**
   - 每次任务完成后立即蒸馏，记忆更新频率从天级提升到任务级
   - 下一个任务开始时已经"记住"上一个任务的经验

3. **维护成本低**
   - 只改 Prompt 即可调整蒸馏策略
   - 不需要维护 TypeScript 代码和依赖

#### 🔴 关键问题：Skills 钩子机制未验证

| # | 问题 | 严重程度 | 说明 |
|---|------|---------|------|
| 1 | **`trigger: before_task` 是否存在** | 🔴 阻塞 | 文档假设 Trae Skills 支持 `before_task` / `after_task` 钩子，但当前代码中未找到相关实现。需要确认 Trae 版本是否支持。 |
| 2 | **与现有实现冲突** | 🔴 高 | 已投入实施的 Phase 1-4（TypeScript skills + WikiBot + 调度脚本）与此方案互斥。如果采用此方案，需要废弃已有代码。 |
| 3 | **蒸馏质量不可控** | 🟡 中 | 纯 Prompt 指令的蒸馏质量依赖 Trae 内置模型，无法像 TypeScript 方案那样做质量验证（字数检查、维度检查、降级摘要）。 |
| 4 | **无法发送飞书告警** | 🟡 中 | 纯 Skills 方案无法执行外部 API 调用（如飞书告警），连续失败告警功能丢失。 |
| 5 | **周日合并条件不可靠** | 🟡 中 | `condition: 今天是周日` 依赖 Trae 正确解析日期条件，不如代码中的 `getDay() === 0` 可靠。 |
| 6 | **多工作区共享困难** | 🟡 中 | 当前方案中每个工作区独立运行 skills，shared/ 目录的晋升机制难以在 Prompt 中实现。 |

#### 🔴 与现有架构的冲突

已实施的架构：
```
workspaces/
├── PMCLI/          ← 有 .trae/rules/，但 skills 在 mvp-runner/src/skills/
├── DEVCLI/         ← 同上
└── WikiBot/        ← 独立工作区，有飞书应用凭证
```

文档B建议的架构：
```
workspaces/PMCLI/
├── .trae/
│   ├── skills/     ← skill.md 放在这里
│   └── rules/      ← project_rules.md 声明钩子
```

**冲突点**：
1. `mvp-runner/src/skills/` 下的 TypeScript 文件需要废弃
2. WikiBot 独立飞书应用（刚配置的 `cli_a97a2dc1d0ba9bd3`）不再需要
3. `wiki-scheduler.ts` 调度脚本不再需要
4. `setup-wiki-scheduler.ps1/sh` 不再需要

### 2.4 验证建议

在决定是否采用此方案前，必须先验证：

```bash
# 验证 Trae 是否支持 trigger: before_task/after_task
# 在 workspaces/PMCLI/.trae/skills/test-trigger/skill.md 中创建测试 skill
# 观察是否在任务开始/完成时自动触发
```

如果 Trae **不支持**这些钩子，此方案完全不可行。

### 2.5 评审结论

**暂缓实施** ⏸️

原因：
1. **核心依赖未验证**：`trigger: before_task` / `after_task` 在 Trae 中的存在性未确认
2. **与已有投入冲突**：已完成的 Phase 1-4 需要大量废弃
3. **功能损失**：飞书告警、质量验证、多工作区协调等能力在纯 Skills 方案中难以实现

**建议**：
- 先验证 Trae Skills 是否支持生命周期钩子
- 如支持，可作为 Phase 5 优化方向（简化已有架构，而非替换）
- 如不支持，保持现有 TypeScript + 调度脚本方案
