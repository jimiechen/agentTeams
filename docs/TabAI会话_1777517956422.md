## **二、针对"重试按钮找不到"的五个具体修复方向**

### **方向 1：从"瞬时查找"改为"渲染等待 + MutationObserver"**

当前的 `delay(2000) + 一次性查找` 是最大的问题。任务切换后 React 重新挂载组件，`重试`按钮的渲染时机不可预测，可能 800ms 也可能 4 秒。

参考你脚本中 `waitForCompletion` 的轮询思路，但用 MutationObserver 替代定时轮询会更高效：

```javascript
async function clickRetryButtonRobust(trae, timeoutMs = 8000) {
  return await trae.evaluate(`
    new Promise((resolve) => {
      const tryFind = () => {
        // 复用你脚本里的多路查找思路
        const candidates = [
          document.querySelector('button[aria-label="重试"]'),
          document.querySelector('button[aria-label*="重试"]'),  // 模糊匹配
          document.querySelector('[role="button"][aria-label="重试"]'),  // role 兜底
          ...Array.from(document.querySelectorAll('button')).filter(b => 
            b.textContent?.trim() === '重试' && b.offsetParent !== null  // 必须可见
          )
        ];
        
        const btn = candidates.find(b => b && !b.disabled);
        if (btn) {
          btn.click();
          return { clicked: true, method: 'observer-found' };
        }
        return null;
      };
      
      // 立即尝试一次
      const immediate = tryFind();
      if (immediate) { resolve(immediate); return; }
      
      // MutationObserver 监听 DOM 变化
      const observer = new MutationObserver(() => {
        const result = tryFind();
        if (result) {
          observer.disconnect();
          resolve(result);
        }
      });
      observer.observe(document.body, { childList: true, subtree: true, attributes: true });
      
      // 超时兜底
      setTimeout(() => {
        observer.disconnect();
        resolve({ clicked: false, method: 'timeout', diagnosis: collectDiagnostics() });
      }, ${timeoutMs});
      
      function collectDiagnostics() {
        // 失败时返回诊断信息，方便定位
        return {
          totalButtons: document.querySelectorAll('button').length,
          allAriaLabels: Array.from(document.querySelectorAll('button[aria-label]'))
            .map(b => b.getAttribute('aria-label')).slice(0, 20),
          hasInterruptedText: document.body.textContent?.includes('手动终止输出'),
          chatTurnsCount: document.querySelectorAll('.chat-turn').length
        };
      }
    })
  `);
}
```

这个写法的关键改进有三点：**MutationObserver 替代定时轮询**让响应时间从最差 5 秒降到毫秒级；**`offsetParent !== null` 判断可见性**避免点击到隐藏的按钮；**失败时收集诊断信息**让你下次出问题时一眼看出 DOM 实际是什么样。

### **方向 2：补充"重试按钮还没渲染"的前置检测**

现在的问题是 recovery 不知道按钮"是没有"还是"还没出现"。你脚本里 `checkTaskStatus` 检测到 `interrupted` 后可以补一步检测：**任务确认 interrupted 后，等待"手动终止输出"文本出现，再去找按钮**。

```javascript
async function waitForInterruptedRendered(trae, timeoutMs = 5000) {
  const startTime = Date.now();
  while (Date.now() - startTime < timeoutMs) {
    const ready = await trae.evaluate(`
      (() => {
        // 三个就绪信号，任一满足即认为可以查找重试按钮
        const hasInterruptText = document.body.textContent?.includes('手动终止输出') 
          || document.body.textContent?.includes('已中断');
        const hasRetryButton = !!document.querySelector('button[aria-label*="重试"]');
        const stoppedThinking = !document.querySelector('[class*="thinking"]');
        return { ready: hasInterruptText && stoppedThinking, hasRetryButton };
      })()
    `);
    if (ready.ready) return ready;
    await new Promise(r => setTimeout(r, 300));
  }
  return { ready: false, hasRetryButton: false };
}
```

这个函数充当 `clickRetryButton` 的**前置守卫**——先确认 UI 进入了"可以点击重试"的稳定状态，再去查找按钮，避免对着半渲染的 DOM 做查找。

### **方向 3：用诊断脚本反向定位"按钮到底在哪"**

报告里连续 5 次说"找不到按钮"，但没有任何关于**当时 DOM 实际长什么样**的证据。这是排查盲区。建议立即增加一个诊断脚本，每次查找失败时自动输出 DOM 快照：

```javascript
async function diagnoseRetryButtonAbsence(trae) {
  return await trae.evaluate(`
    (() => {
      const allButtons = Array.from(document.querySelectorAll('button'));
      const allRoleButtons = Array.from(document.querySelectorAll('[role="button"]'));
      
      return {
        timestamp: new Date().toISOString(),
        totalButtonElements: allButtons.length,
        totalRoleButtons: allRoleButtons.length,
        
        // 所有按钮的 aria-label（排除空值）
        allAriaLabels: allButtons
          .map(b => b.getAttribute('aria-label'))
          .filter(Boolean),
        
        // 所有可见按钮的文本
        visibleButtonTexts: allButtons
          .filter(b => b.offsetParent !== null)
          .map(b => (b.textContent || '').trim())
          .filter(t => t.length > 0 && t.length < 30),
        
        // 检查关键容器是否存在
        hasChatContainer: !!document.querySelector('[class*="chat"]'),
        chatTurnsCount: document.querySelectorAll('.chat-turn').length,
        
        // 检查是否有遮罩层覆盖（可能阻止按钮渲染）
        hasModal: !!document.querySelector('[role="dialog"], [class*="modal"], [class*="mask"]'),
        
        // 当前 URL 和页面标题（确认没意外跳转）
        currentUrl: location.href,
        documentTitle: document.title,
        
        // 倒数三个 chat-turn 的文本（找到任务边界）
        lastTurns: Array.from(document.querySelectorAll('.chat-turn'))
          .slice(-3)
          .map(t => (t.textContent || '').slice(0, 100))
      };
    })()
  `);
}
```

把这个诊断脚本接入到 recovery-executor 的失败分支，每次查找失败时把诊断结果写入日志：

```typescript
if (!retryResult.clicked) {
  const diagnosis = await this.diagnoseRetryButtonAbsence();
  debug('Retry button missing, diagnosis: %j', diagnosis);
  // 这样下次复盘时能直接看到 DOM 实际状态
}
```

报告里之所以分析停留在"猜测 UI 渲染慢"的层面，就是因为没有这种诊断数据。加上之后下一次事故能在 1 分钟内定位到底是按钮真的不存在、还是 selector 写错了、还是被遮罩挡住了。

### **方向 4：参考 hover 触发模式增加备用路径**

你脚本里 `copy-with-hover-buttons.js` 处理了"按钮需要 hover 才显示"的场景。如果重试按钮也属于这类隐藏控件，三路查找永远会失败。建议在主查找失败后增加一个 hover 触发分支：

```javascript
async function tryHoverThenClick(trae) {
  return await trae.evaluate(`
    (() => {
      // 找到最后一个 chat-turn（中断消息所在位置）
      const turns = document.querySelectorAll('.chat-turn');
      const lastTurn = turns[turns.length - 1];
      if (!lastTurn) return { clicked: false, method: 'no-turn' };
      
      // 触发 hover 让按钮显示
      const hoverEvent = new MouseEvent('mouseenter', { bubbles: true });
      lastTurn.dispatchEvent(hoverEvent);
      
      // 短暂等待按钮出现（合成事件触发不需要等太久）
      return new Promise(resolve => {
        setTimeout(() => {
          const btn = lastTurn.querySelector('button[aria-label="重试"]');
          if (btn) {
            btn.click();
            resolve({ clicked: true, method: 'hover-then-click' });
          } else {
            resolve({ clicked: false, method: 'hover-no-button' });
          }
        }, 200);
      });
    })()
  `);
}
```

### **方向 5：把"任务切换 + 重试"拆成独立可观察的两步**

现在 `executeFrozenRecovery` 把切换任务、等待、查找按钮捏在一个流程里，任何一步失败都会进入"刷新页面"兜底。建议拆成两个独立动作，每步都有独立的成功判定：

```typescript
async executeFrozenRecovery() {
  // Step A: 切换任务（独立验证）
  const switched = await this.switchToTaskAndVerify('DEVCLI');
  if (!switched.success) {
    return { success: false, reason: 'switch-failed', stage: 'A' };
  }
  
  // Step B: 等待 UI 进入"可重试"状态（新增）
  const ready = await this.waitForInterruptedRendered();
  if (!ready.ready) {
    return { success: false, reason: 'ui-not-ready', stage: 'B', diagnosis: ready };
  }
  
  // Step C: 查找并点击（带 MutationObserver）
  const clicked = await this.clickRetryButtonRobust();
  if (!clicked.clicked) {
    const diagnosis = await this.diagnoseRetryButtonAbsence();
    return { success: false, reason: 'button-not-found', stage: 'C', diagnosis };
  }
  
  // Step D: 验证点击生效（任务状态变回 in_progress）
  const verified = await this.verifyTaskResumed('DEVCLI', 5000);
  if (!verified) {
    return { success: false, reason: 'click-no-effect', stage: 'D' };
  }
  
  return { success: true };
}
```

这样每次失败你都知道**卡在哪一步**，而不是笼统的"recovery failed"。报告里"刷新页面 3 次"的本质就是因为不知道是 B 失败、C 失败还是 D 失败，只好用最重的兜底。

## **三、立即可执行的三步行动**

第一步是**先加诊断脚本**（方向 3），不改任何业务逻辑，只是在失败分支记录 DOM 状态。这一步零风险，但下次事故能直接定位问题。

第二步是**用 MutationObserver 替换定时轮询**（方向 1），结合方向 2 的就绪等待，把响应时间和成功率同时提升。这一步可以本地测一个晚上后再合并。

第三步是**拆分恢复流程**（方向 5），让每个阶段有独立的成功判定和诊断数据。这一步是结构性改动，建议放在事故修复 PR 里和 Cooldown 机制（之前评审里提到的）一起做。

## **四、一个容易被忽略的关键点**

你的脚本文档里 `checkTaskStatus` 用了 `.index-module__task-item___zOpfg` 这种带 hash 的 className——**这种 className 在 Trae IDE 版本升级时极易变化**，是脚本失效的高风险点。建议在 recovery-executor 里所有依赖此类 className 的 selector 都加上**版本兼容声明**，例如：

```typescript
const TASK_ITEM_SELECTORS = [
  '.index-module__task-item___zOpfg',  // v1.x
  '[data-testid="task-item"]',          // v2.x（如果存在）
  '[class*="task-item"]'                // 模糊匹配兜底
];
```

并在启动时跑一次自检，记录当前 Trae IDE 版本下哪个 selector 生效，写入运行时日志。这是和"重试按钮找不到"同源的脆弱性，建议一并加固。
