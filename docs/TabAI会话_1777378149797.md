## **skill.md 实现（纯 Prompt 指令，无需 TypeScript）**

Trae Skills 的核心是 `skill.md`，它就是给内置模型的指令，不需要写代码。

### **`wiki-inject/skill.md`**

```markdown
---
name: wiki-inject
description: 在执行新任务前，读取项目记忆并组装到提示词中
trigger: before_task
---

## 你的任务

在开始处理用户请求之前，先执行以下步骤：

### Step 1：读取核心知识（Layer 2）

读取文件：`wiki/core/knowledge.md`

如果文件存在，将内容记为 `[核心知识]`。
如果文件不存在，`[核心知识]` 为空。

### Step 2：读取近期记忆（Layer 1）

读取以下文件（按日期从新到旧，最多3个）：
- `wiki/daily/今天日期.md`
- `wiki/daily/昨天日期.md`  
- `wiki/daily/前天日期.md`

将所有存在的文件内容合并，记为 `[近期记忆]`。

### Step 3：组装背景知识块

将以下内容插入到你处理用户请求之前：

```
## 项目背景知识（自动注入）

### 核心知识（长期有效）
{核心知识内容，如为空则写"暂无"}

### 近期动态（最近3天）
{近期记忆内容，如为空则写"暂无"}

---
```

### Step 4：基于背景知识处理用户请求

携带上述背景知识，继续执行用户的原始任务。

## 约束

- 背景知识注入总字数不超过 1800 字，超出时优先保留核心知识，截断近期记忆
- 如果 wiki/ 目录不存在，直接跳过注入，不报错
```

### **`wiki-distill/skill.md`**

```markdown
---
name: wiki-distill
description: 任务完成后，将本次执行记录蒸馏为今日记忆
trigger: after_task
---

## 你的任务

当前任务已执行完毕。现在执行知识蒸馏，将本次执行记录提炼为结构化记忆。

### Step 1：读取本次执行记录

读取 `runs/` 目录下今天（{today}）的所有 `.md` 文件。
将所有内容合并为原始记录。

### Step 2：读取今日已有记忆

读取 `wiki/daily/{today}.md`（如存在）。
记为 `[已有今日记忆]`，本次蒸馏需要与其合并，不是覆盖。

### Step 3：执行六维度蒸馏

基于原始记录，提炼以下六个维度的知识：

1. **新增知识**：本次任务学到了什么新信息（项目结构、API行为、环境特性）
2. **失败模式**：出现了什么类型的失败、如何修复、恢复率如何
3. **用户偏好**：用户表现出什么习惯或偏好
4. **协议变更**：DOM选择器、配置项、接口的变化
5. **性能基线**：关键指标（switchTask/fillPrompt/waitResponse）的均值和异常
6. **下一步关注**：明天需要跟进的具体可操作事项

### Step 4：写入今日记忆文件

将蒸馏结果写入 `wiki/daily/{today}.md`，格式如下：

```markdown
# {today} 今日记忆

## 新增知识
- ...

## 失败模式
- ...

## 用户偏好
- ...

## 协议变更
- ...

## 性能基线
- ...

## 下一步关注
- ...
```

**合并规则**：如果今日文件已存在，将新内容追加合并，不删除已有条目，相同内容去重。

### Step 5：更新 todo.md

将"下一步关注"中的条目追加到 `wiki/todo.md`，格式：
```
- [{today}] {条目内容}
```

## 约束

- 今日记忆总字数控制在 500 字以内
- 如果本次任务没有产生任何有价值的新信息，写"无变化"，不要省略维度
- 只在任务成功或部分成功时执行蒸馏，任务完全失败且无任何执行记录时跳过
```

### **`wiki-merge/skill.md`**

```markdown
---
name: wiki-merge
description: 每周日，将7天记忆合并为核心知识
trigger: after_task
condition: 今天是周日
---

## 你的任务

今天是周日，在完成任务蒸馏后，执行核心知识合并。

### Step 1：读取本周7天的 Layer 1

读取 `wiki/daily/` 目录下最近7天的所有 `.md` 文件。

### Step 2：备份现有 Layer 2

将 `wiki/core/knowledge.md` 复制到
`wiki/core/backups/knowledge-{today}.md`

### Step 3：执行合并与淘汰

将本周记忆与现有核心知识合并，遵循以下规则：

**保留**：
- 架构决策（长期有效的设计选择）
- 本周出现 ≥2 次的模式
- 稳定的用户偏好
- 已知陷阱（需要永久避免的问题）
- 标有 `[重要]` 的条目（无条件保留）

**淘汰**：
- 临时问题、一次性事件
- 已修复且不再复现的 Bug
- 细节过程（只保留结论）

**硬约束**：
- 总字数 ≤ 300 字
- 总条目 ≤ 50 条

### Step 4：写入核心知识文件

将合并结果写入 `wiki/core/knowledge.md`，格式：

```markdown
# 项目核心知识（更新时间：{today}）

## 架构决策
- [决策内容] - [原因/依据]

## 稳定模式
- [触发场景] → [处理方式] → [预期效果]

## 用户偏好
- [工作区] [偏好描述]

## 已知陷阱
- [陷阱描述] ⚠️ [避免方式]

## 性能基线
- [工作区] [指标]: [基准值]
```

## 任务生命周期规则

### 任务开始前
每次接收到新任务时，**必须先执行** `wiki-inject` skill：
- 读取项目记忆（wiki/core/ 和 wiki/daily/）
- 将背景知识注入到当前上下文
- 然后再处理用户的实际请求

### 任务完成后
每次任务执行完毕时，**必须执行** `wiki-distill` skill：
- 将本次执行记录蒸馏为今日记忆
- 更新 wiki/daily/{today}.md
- 更新 wiki/todo.md

如果今天是**周日**，额外执行 `wiki-merge` skill：
- 将本周7天记忆合并为核心知识
- 更新 wiki/core/knowledge.md

## Wiki 文件路径

- Layer 0（原始记录）：`runs/`
- Layer 1（每日记忆）：`wiki/daily/`
- Layer 2（核心知识）：`wiki/core/knowledge.md`
- 待办事项：`wiki/todo.md`

## 重要约束

- wiki/ 目录下的文件只由 wiki-* skills 维护，业务执行不修改这些文件
- 蒸馏和注入是**轻量操作**，不影响主任务的执行时间
- 如果 wiki/ 目录不存在，自动创建，不报错
---
```

## **与之前方案的对比**

| 维度 | 之前方案（外部脚本） | 现在方案（原生 Skills） |
|------|-------------------|----------------------|
| **触发方式** | crontab 定时 | 任务生命周期钩子 |
| **LLM 调用** | 外部 API / WikiBot | Trae 内置模型（同一个） |
| **代码量** | 3个 TypeScript 文件 | 3个 skill.md（纯 Prompt） |
| **部署** | 需要配置 cron + env | 放到 `.trae/skills/` 即生效 |
| **蒸馏时机** | 每天凌晨4点 | 每次任务完成后实时蒸馏 |
| **维护成本** | 高（脚本 + 调度） | 极低（只改 Prompt） |

**最大的差异**：蒸馏从"每天一次批处理"变成了"每次任务完成后实时蒸馏"，记忆更新频率从天级变成了任务级，AI 在下一个任务开始时就已经"记住"了上一个任务的经验。这比定时批处理更符合你们的使用场景——飞书消息触发的任务本来就是离散的，每次任务间隙正好是蒸馏的最佳时机。
