## 一、方案定位

### 1.1 解决的问题

当前心跳检测系统存在一个关键盲区：`background` 模式既不判断失败，也不触发自动恢复。当 DEVCLI/PMCLI 任务长时间处于"后台运行"状态但实际已经卡死时，系统无法主动介入，导致任务永久挂起。

### 1.2 方案本质

**用 Trae UI 已有的两个按钮（"后台运行" + "取消"）把未知的卡死状态转换为已知的可恢复状态**，不做复杂的存活探测，不做进程级干预，完全通过 UI 层操作完成恢复。

### 1.3 为什么选择简化方案

相比前期讨论的"分层诊断 + 主动探活 + 有限升级"完整方案（详见 `HEARTBEAT_OPTIMIZATION_GUIDE.md`），本方案有三个核心优势：

- **工程量小**：核心代码不超过 120 行，1~2 天可完成实现
- **风险可控**：只操作 UI 层按钮，不涉及进程管理、不发送系统信号
- **完美复用**：取消后的任务自动进入 `interrupted` 状态，直接走现有恢复链路

## 二、核心逻辑

### 2.1 三步恢复流程

```
[background 状态]
      ↓
  超时判定（5 分钟无 DOM 变化）
      ↓
  点击"后台运行"按钮 → UI 释放可操作状态
      ↓
  等待 30 秒（任务切后台 + UI 过渡 + 状态同步）
      ↓
  二次确认：DOM 是否恢复活跃（若活跃则跳过取消）
      ↓
  点击"取消"按钮 → 任务进入 interrupted
      ↓
  等待 10 秒（状态切换 + 重试按钮渲染）
      ↓
  复用现有 executeFrozenRecovery（切任务 → 找重试 → 点击）
      ↓
  任务重新运行
```

### 2.2 状态转换

```
normal
  └─→ background（检测到后台运行按钮）
        ├─→ normal（DOM 重新活跃）
        ├─→ interrupted（触发超时恢复后）
        └─→ human-required（熔断后升级）

interrupted
  └─→ normal（复用现有 frozen 恢复路径）
```

不新增 `suspicious`、`probing` 等中间状态，保持状态机简洁。

## 三、关键参数（v1.1 更新）

### 3.1 核心参数

| 参数 | 默认值 | 合理范围 | 说明 |
|------|--------|---------|------|
| 静默超时阈值 | 5 分钟 | 3~30 分钟 | DOM 无变化超过此时长触发恢复 |
| DOM 采样频率 | 10 秒 | 5~30 秒 | 定时采样而非 MutationObserver |
| **后台运行后等待** | **30 秒** | **10~120 秒** | **等待任务切后台完成** |
| **取消后等待** | **10 秒** | **3~30 秒** | **等待状态切换到 interrupted** |
| 单任务最大触发次数 | 2 次 | 1~5 次 | 熔断机制 |
| 恢复流程 Cooldown | 60 秒 | 30~300 秒 | 防止连续触发 |

### 3.2 为什么默认 30 秒

v1.0 推荐的 1.5~2 秒是基于"纯 UI 渲染等待"的经验值，在 Trae 环境下**严重低估**了以下三类真实耗时：

**耗时一：任务切后台的业务流程**。点击"后台运行"后，Trae 需要完成：
- 停止前台命令的标准输出流订阅
- 把任务从"前台活跃任务"移到"后台任务队列"
- 持久化任务状态到本地存储（IndexedDB / File）
- 通知主进程释放前台终端资源

这个流程在**卡死场景下尤其慢**——因为任务本身就已经不响应，后台切换会等待超时才能完成。实测 P95 耗时通常在 **10~20 秒**。

**耗时二：UI 动画与过渡**。Trae 使用了较多 CSS 动画（模态框淡出、按钮状态切换、任务列表重排），在低配机器或主进程繁忙时这些动画可能被延迟执行，累计耗时可达 **3~5 秒**。

**耗时三：异步状态同步**。前端状态（Vuex/Redux/Pinia）与后端状态同步通常通过 WebSocket 或 IPC 完成，在卡死场景下因为有消息队列积压，P99 可能达到 **15~25 秒**。

三类耗时叠加的 P99 总时长在 **25~30 秒**，因此 **30 秒作为默认值**能覆盖绝大多数场景而不出现"等待不足导致取消按钮点击失效"的问题。

### 3.3 不同环境的推荐配置

| 环境/场景 | 后台运行后等待 | 取消后等待 | 理由 |
|----------|---------------|-----------|------|
| 本地开发（高配机器） | 20 秒 | 5 秒 | 性能好，可缩短 |
| **生产环境（默认）** | **30 秒** | **10 秒** | **P99 覆盖 + 安全余量** |
| 低配机器 / 网络不佳 | 60 秒 | 20 秒 | 充分等待异步消息 |
| 首次引入 / 灰度期 | 45 秒 | 15 秒 | 保守值，观察后调整 |
| 任务特别大（如编译） | 90 秒 | 30 秒 | 后台切换本身耗时长 |

### 3.4 关于恢复总耗时的取舍

默认配置下，一次完整的恢复流程总耗时约为：

```
超时判定    : 5 分钟
后台点击等待: 30 秒
取消点击等待: 10 秒
frozen 恢复 : 10~20 秒
─────────────────────
总计        : 约 5 分 50 秒
```

相比 v1.0 的 5 分 4 秒，v1.1 多了约 40 秒。这 40 秒的代价换来的是**取消按钮点击成功率从约 70% 提升到 95%+**，属于合理的权衡。若需要进一步缩短总耗时，应该降低"静默超时阈值"（例如从 5 分钟降到 3 分钟），而不是缩短两步点击间隔。

// detector.ts
private async checkBackgroundTimeout(
  cdp: CDPClient,
  ctx: BackgroundStateContext
): Promise<{ timeout: boolean; silentDuration: number }> {
  const snapshot = await cdp.evaluate(`
    (() => {
      const term = document.querySelector('.terminal, .chat-container');
      if (!term) return '';
      const tail = term.textContent?.slice(-500) ?? '';
      return \`\-\-\\`;
    })()
  `);
  
  if (snapshot !== ctx.lastDomSnapshot) {
    ctx.lastActivityAt = Date.now();
    ctx.lastDomSnapshot = snapshot;
    return { timeout: false, silentDuration: 0 };
  }
  
  const silentDuration = Date.now() - ctx.lastActivityAt;
  return {
    timeout: silentDuration > this.config.silent_timeout_ms,
    silentDuration,
  };
}

### 4.4 RecoveryExecutor 层：动态等待 + 两步恢复（v1.1 核心）

```typescript
// recovery-executor.ts

/**
 * 动态等待指定按钮就绪
 * 先等最小下限，然后轮询检测目标按钮可用性
 */
private async waitForButton(
  buttonMatcher: string,  // 例如 "取消"
  maxWaitMs: number,
  minWaitMs: number = 5000,
  pollIntervalMs: number = 1000
): Promise<{ ready: boolean; actualWaitMs: number }> {
  const startTime = Date.now();
  
  // Step 1: 保守下限等待
  await this.delay(minWaitMs);
  
  // Step 2: 轮询检测
  while (Date.now() - startTime < maxWaitMs) {
    const ready = await this.cdp.evaluate(`
      (() => {
        const btns = Array.from(document.querySelectorAll('button'));
        const target = btns.find(b => 
          b.textContent?.trim() === '' &&
          b.offsetParent !== null &&
          !b.disabled &&
          b.getBoundingClientRect().width > 0
        );
        return !!target;
      })()
    `);
    
    if (ready) {
      return { ready: true, actualWaitMs: Date.now() - startTime };
    }
    
    await this.delay(pollIntervalMs);
  }
  
  return { ready: false, actualWaitMs: maxWaitMs };
}

/**
 * Background 模式超时恢复
 */
private async executeBackgroundTimeout(
  ctx: BackgroundStateContext
): Promise<RecoveryResult> {
  const taskId = this.getCurrentTaskId();
  const cfg = this.config.background_timeout_recovery;
  
  // 熔断检查
  if (ctx.triggerCount >= cfg.max_trigger_per_task) {
    await this.notifyHuman(`任务  background 超时恢复已达  次上限`);
    return { success: false, reason: 'human-required' };
  }
  
  // Cooldown 检查
  if (this.isInCooldown('background-timeout')) {
    return { success: false, reason: 'cooldown' };
  }
  
  // ===== Step 1: 点击"后台运行"按钮 =====
  const bgClicked = await this.cdp.evaluate(`
    (() => {
      const btns = Array.from(document.querySelectorAll('.icd-btn.icd-btn-tertiary'));
      const bgBtn = btns.find(b => b.textContent?.includes('后台运行'));
      if (bgBtn) { bgBtn.click(); return true; }
      return false;
    })()
  `);
  
  if (!bgClicked) {
    debug('❌ 后台运行按钮未找到');
    return { success: false, reason: 'bg-btn-not-found' };
  }
  
  debug('✅ 后台运行按钮已点击，开始等待任务切后台...');
  
  // ===== Step 2: 动态等待"取消"按钮就绪（默认 30 秒上限）=====
  const waitResult = cfg.dynamic_wait.enabled
    ? await this.waitForButton(
        '取消',
        cfg.click_intervals.after_background_click_ms,
        cfg.dynamic_wait.min_wait_before_poll_ms,
        cfg.click_intervals.poll_interval_ms
      )
    : await this.staticWait(cfg.click_intervals.after_background_click_ms);
  
  if (!waitResult.ready) {
    debug(`❌ 等待 ms 后取消按钮仍未就绪`);
    return { success: false, reason: 'cancel-btn-not-ready' };
  }
  
  debug(`✅ 取消按钮在 ms 内就绪`);
  
  // ===== Step 3: 二次确认（DOM 是否恢复活跃）=====
  const currentSnapshot = await this.computeDomSnapshot();
  if (currentSnapshot !== ctx.lastDomSnapshot) {
    debug('✅ 任务在等待期内恢复活跃，跳过取消操作');
    return { success: true, reason: 'self-recovered', actualWaitMs: waitResult.actualWaitMs };
  }
  
  // ===== Step 4: 点击"取消"按钮 =====
  const cancelClicked = await this.cdp.evaluate(`
    (() => {
      const btns = Array.from(document.querySelectorAll('button'));
      const cancelBtn = btns.find(b => 
        b.textContent?.trim() === '取消' && 
        b.offsetParent !== null
      );
      if (cancelBtn) { cancelBtn.click(); return true; }
      return false;
    })()
  `);
  
  if (!cancelClicked) {
    debug('❌ 取消按钮未找到（轮询后消失）');
    return { success: false, reason: 'cancel-btn-disappeared' };
  }
  
  debug('✅ 取消按钮已点击，等待状态切换...');
  
  // ===== Step 5: 等待状态切换到 interrupted（默认 10 秒）=====
  await this.delay(cfg.click_intervals.after_cancel_click_ms);
  
  // ===== Step 6: 更新计数、Cooldown、飞书通知 =====
  ctx.triggerCount += 1;
  this.setCooldown('background-timeout', cfg.cooldown_ms);
  
  if (cfg.feishu_notify) {
    await this.notifyFeishu({
      type: 'background-timeout-recovered',
      taskId,
      silentDuration: Date.now() - ctx.lastActivityAt,
      triggerCount: ctx.triggerCount,
      actualWaitMs: waitResult.actualWaitMs,
    });
  }
  
  debug(`✅ 后台超时恢复成功: 任务 ，第  次，等待 ms`);
  return { 
    success: true, 
    method: 'bg-then-cancel',
    actualWaitMs: waitResult.actualWaitMs 
  };
}
```

## 五、边界保护

### 5.1 四道保护机制（v1.1 新增第四道）

**保护一：按钮存在性校验**。必须在 DOM 里真实检测到"后台运行"按钮才启动计时。若按钮不存在，计时无意义。通过现有 `determineMode` 里的 `hasBackgroundBtn` 判断已满足。

**保护二：动态等待的"最后确认"**。动态等待结束后（按钮就绪或超时），再采样一次 DOM 快照，若发现任务已自行恢复则跳过取消动作。这是防止"正好在等待期内恢复"的任务被误伤。

**保护三：飞书通知留痕**。每次触发"后台运行 + 取消"恢复必须发飞书信息，字段包括：
- 任务 ID
- 静默时长
- 触发次数
- 动态等待的实际耗时（v1.1 新增）
- 恢复结果（成功/失败及原因）

**保护四（v1.1 新增）：等待上限保护**。动态等待的 `max_wait_ms` 参数不允许配置超过 120 秒（2 分钟）。如果某类任务确实需要等更久，应该走任务级覆盖配置并在代码审查时特殊评审，不能通过全局配置放宽上限。

### 5.2 与速率限制的协调

现有系统对 `refresh-page` 有速率限制（5 次/3600 秒）。新增的 `background-timeout` 恢复动作**不占用 refresh-page 配额**，建议独立配置为 **3 次/3600 秒**。

### 5.3 并发控制

必须与现有互斥锁（recoveryLock）共享。`executeBackgroundTimeout` 触发前必须获取锁，触发后必须释放锁。等待期间持有锁会阻塞其他恢复动作 30 秒，这是可接受的（因为 background 恢复优先级最高）。

### 6.1 功能验收

| 编号 | 验收项 | 标准 |
|------|--------|------|

| AC-6 | 飞书通知 | 每次恢复动作均有通知，字段完整 |

- 连续 24 小时运行无崩溃
- 至少成功恢复 3 次真实卡死场景
