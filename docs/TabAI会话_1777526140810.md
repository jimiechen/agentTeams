### **长时间终端执行无响应的核心处理思路是"分层诊断 + 主动探活 + 有限升级"，关键是把当前被动等待的 `background` 模式升级为主动探测模式，通过终端输出增量、进程状态、命令执行时长三个维度交叉判断真实状态**

你贴出的分析报告已经把问题定位得很清晰：`background` 模式在当前实现里是**检测盲区**——它既不判失败也不触发恢复，而 Layer 2 的网络活动检测又会对纯本地计算任务产生误判。这个问题的本质不是"检测不到",而是**当前的状态机设计缺少一个"长时间运行但不确定是否存活"的中间态处理逻辑**。以下从根因分析、处理策略、具体实现、兜底机制四个层面给出方案。

## **一、问题根因：状态机缺少"存活探测"环节**

先明确当前状态机的结构性缺陷，这决定了所有修复方案的方向。

当前状态机只有 `normal`、`background`、`frozen`、`crashed` 四个状态，中间缺少一个关键的**`suspicious`（可疑）状态**。`background` 被设计为"正常的延长执行"，但在终端场景下，用户看到的"长时间无响应"实际上可能是以下五种完全不同的情况：

**情况 A：命令仍在真实执行**——比如跑一个大型测试套件、下载大文件、编译大项目。终端虽然没有输出，但 CPU/IO 在活跃，进程状态正常。这是**真后台**，应该继续等待。

**情况 B：命令执行完成但 Shell 没有返回提示符**——常见于某些交互式命令（vim、less、ssh）用户按了某个键卡住。进程存在，但不再产生输出。这是**假后台**，需要用户介入或发送中断信号。

**情况 C：命令陷入死循环或死锁**——进程存在且 CPU 可能占用高，但不会产生有意义的输出。这是**真卡死**，需要强制中断并重试。

**情况 D：命令被操作系统挂起**——比如等待输入（stdin 阻塞）、等待网络响应超时、等待文件锁。进程存在但处于 sleep 状态。这是**可恢复卡死**，需要发送特定信号或输入。

**情况 E：终端连接层面出问题**——SSH 断开、WebSocket 超时、Trae 的终端 DOM 渲染卡死但命令已完成。这是**UI 层假象**，进程实际已经结束。

当前的 `background` 模式把这五种情况全部归为"正常"，这就是为什么会出现"任务卡在后台系统不知道"的盲区。正确的设计应该是把 `background` 细分，或在它之上叠加一个 `suspicious` 探测层。

## **二、核心处理策略：三阶段递进式诊断**

针对上述五种情况，采用**三阶段递进策略**——从最便宜的信号开始，逐步升级到最昂贵的干预手段。每个阶段有明确的触发条件和超时时间，避免过早或过晚介入。

### **阶段 1：被动观察期（0-3 分钟）**

进入 `background` 模式后的前 3 分钟，只做**零成本被动观察**，不发送任何干预指令。

核心监测信号包括三个维度：**终端输出增量**（过去 30 秒是否有任何字符新增，包括进度条的刷新）、**DOM 高度变化**（终端容器的 scrollHeight 是否增长）、**光标位置变化**（光标是否在移动，即使没有文本输出）。

只要三个信号中有任一个为"活跃"，就维持 `background` 状态，重置 3 分钟计时器。这个阶段能正确识别**情况 A 的长输出命令**（比如 `npm install` 的进度条刷新、测试用例的逐条输出）。

**关键实现点**：不能只看文本内容哈希，必须监听 DOM 的 `MutationObserver`，因为终端的进度条刷新通常使用 `\r` 回车符覆盖同一行文本，内容哈希看起来没变但 DOM 实际在变。

### **阶段 2：主动探测期（3-8 分钟）**

超过 3 分钟所有信号都静默，进入主动探测。**核心原则是"用无副作用的探测动作验证存活状态"**。

推荐的探测动作按优先级如下：

**探测 A：发送 Enter 键**。这是最安全的探测——在正常 shell 里按 Enter 只会产生一个新提示符，不会影响正在执行的命令（因为前台进程会拦截 Enter）。如果发送 Enter 后 1 秒内终端有任何响应（哪怕只是新的一行），说明**shell 存活但命令已完成**（对应情况 B），这时可以安全地继续下一步；如果 Enter 后完全没反应，说明命令仍在前台运行。

**探测 B：发送一个无害的查询命令**。例如在新的终端 tab 或分屏里执行 `ps aux | grep <task-name>` 或 `echo alive_probe_$(date +%s)`。这个探测不干扰原命令，能直接确认**进程是否真实存在**（对应情况 C/E 的区分）。如果能查到进程且 CPU > 5%，说明命令真实在跑；查不到进程说明是终端 UI 层假象。

**探测 C：CDP 层面的终端元素检查**。通过 CDP 读取终端 DOM 的 `data-status` 属性、prompt 元素是否存在、光标是否闪烁。这能识别 Trae 自己的终端渲染状态是否正常，区分"命令卡死"和"终端组件卡死"。

三个探测按顺序执行，每个探测间隔 30 秒，总共约 5 分钟的探测窗口。探测结果写入 `heartbeat.jsonl`，作为后续决策的依据。

### **阶段 3：有限干预期（8-15 分钟）**

探测期结束后仍无法确认存活，进入有限干预。**严禁直接 kill 进程**，必须按"温和 → 强制"的梯度进行。

**干预 A：发送 Ctrl+C**。这是标准的中断信号，大部分命令会优雅退出。发送后等待 30 秒观察终端是否出现新提示符。这一步能解决情况 C 和 D 的大部分场景。

**干预 B：发送 Ctrl+\\（SIGQUIT）**。比 Ctrl+C 更强的信号，会触发核心转储但保留进程清理机会。这是第二梯度。

**干预 C：点击"后台运行"按钮 + 新建终端继续**。如果前两步都无效，把当前卡死的终端转为后台（Trae UI 提供的能力），在新终端里执行 `cleanup` 命令（例如杀掉特定进程名）然后重试原任务。这是你当前报告里提到的"不点击后台运行按钮"需要补齐的恢复路径。

**干预 D：Session 级重启**。最后的兜底——关闭当前 Trae session，重启新 session 并从最近的 checkpoint 恢复任务。这一步应该触发告警和人工介入。

每次干预之间至少间隔 60 秒观察期，确保给命令足够的退出时间。**所有干预必须有 Cooldown 机制**（参考前期事故报告的修复建议），同一个命令 30 分钟内最多触发 3 次干预，超过则进入人工介入模式。

### **1. 状态机扩展**

在现有四状态基础上，新增 `suspicious` 和 `probing` 两个状态，状态转换图变为：

```
normal
  ├─→ background (检测到后台运行按钮)
  │     ├─→ normal (检测到前台运行)
  │     ├─→ suspicious (3 分钟无任何信号)
  │     └─→ frozen (检测到中断/弹窗)
  │
  └─→ frozen → ...

suspicious
  ├─→ normal (探测响应活跃)
  ├─→ probing (探测无响应，进入主动干预)
  └─→ background (探测发现真实活动)

probing
  ├─→ normal (干预后恢复)
  ├─→ frozen (干预失败，升级处理)
  └─→ human-intervention (所有干预失败)
```

新增一个 `human-intervention` 终态，表示需要人工介入，此时系统应该立即发送飞书通知并停止自动化。

### **2. Detector 改造**

在 `detector.ts` 里新增一个 `BackgroundActivityProbe` 子模块，专门负责阶段 1 的被动观察和阶段 2 的主动探测：

```typescript
class BackgroundActivityProbe {
  private lastDomHeight = 0;
  private lastContentHash = '';
  private lastActivityAt = Date.now();
  private readonly PASSIVE_TIMEOUT = 3 * 60 * 1000;  // 3 分钟
  private readonly PROBE_INTERVAL = 30 * 1000;       // 30 秒
  
  async checkActivity(cdp: CDPClient): Promise<ActivitySignal> {
    // 维度 1: DOM 高度变化
    const height = await cdp.evaluate(
      () => document.querySelector('.terminal')?.scrollHeight ?? 0
    );
    
    // 维度 2: 终端文本哈希（排除光标和时间戳）
    const hash = await this.computeTerminalHash(cdp);
    
    // 维度 3: 光标位置
    const cursor = await cdp.evaluate(
      () => document.querySelector('.xterm-cursor')?.getBoundingClientRect()
    );
    
    const isActive = height !== this.lastDomHeight 
                  || hash !== this.lastContentHash
                  || this.cursorMoved(cursor);
    
    if (isActive) {
      this.lastActivityAt = Date.now();
      this.lastDomHeight = height;
      this.lastContentHash = hash;
    }
    
    return {
      isActive,
      silentDuration: Date.now() - this.lastActivityAt,
      needsProbe: Date.now() - this.lastActivityAt > this.PASSIVE_TIMEOUT,
    };
  }
  
  async activeProbe(cdp: CDPClient): Promise<ProbeResult> {
    // 探测 A: 发送 Enter
    const enterResult = await this.probeEnter(cdp);
    if (enterResult.hasResponse) return { alive: true, method: 'enter' };
    
    // 探测 B: 进程查询（需要额外的 terminal session）
    const psResult = await this.probeProcess(cdp);
    if (psResult.processExists) return { alive: true, method: 'ps' };
    
    // 探测 C: DOM 状态检查
    const domResult = await this.probeDOM(cdp);
    return { 
      alive: domResult.terminalResponsive, 
      method: 'dom',
      uiStalled: !domResult.terminalResponsive,
    };
  }
}
```

### **3. RecoveryExecutor 改造**

```typescript
private async executeBackgroundRecovery(
  level: 'observe' | 'probe' | 'intervene'
): Promise<RecoveryResult[]> {
  // 弹窗处理保持现有逻辑
  await this.handleModals();
  
  if (level === 'observe') {
    return [{ type: 'observe', action: 'wait', result: 'ok' }];
  }
  
  if (level === 'probe') {
    const probeResult = await this.probe.activeProbe(this.cdp);
    return [{ type: 'probe', action: probeResult.method, result: probeResult.alive ? 'alive' : 'silent' }];
  }
  
  if (level === 'intervene') {
    // Cooldown 检查
    if (this.isInCooldown('intervene')) {
      return [{ type: 'intervene', action: 'skipped', result: 'cooldown' }];
    }
    
    // 梯度干预
    const interventionSteps = [
      () => this.sendCtrlC(),
      () => this.sendCtrlBackslash(),
      () => this.clickBackgroundButtonAndRestart(),
    ];
    
    for (const step of interventionSteps) {
      const result = await step();
      await this.wait(60 * 1000);  // 60 秒观察期
      const isRecovered = await this.checkRecovered();
      if (isRecovered) return [{ type: 'intervene', result: 'recovered' }];
    }
    
    // 所有干预失败，升级到人工
    await this.notifyHuman('background task unrecoverable');
    return [{ type: 'intervene', result: 'human-required' }];
  }
}
```

## **四、兜底机制：防止误伤的三道防线**

无论诊断多么精巧，都存在误判风险——可能把真实在跑的命令当作卡死强制中断，导致数据损坏或状态不一致。必须设计三道防线。

**第一道：命令类型白名单**。对已知长时间运行的命令（`npm install`、`yarn install`、`pip install`、`docker pull`、`git clone`、`mvn install` 等），被动观察超时延长到 30 分钟，且跳过阶段 3 的干预，只做阶段 1-2 的诊断。这个白名单通过扫描终端最近输出的命令行判断。

**第二道：用户可见的渐进告警**。进入 `suspicious` 状态时通过飞书发送"观察中"通知（不是告警，只是信息），进入 `probing` 状态时发送"正在探测"通知，进入 `intervene` 状态前发送"即将干预，30 秒内回复 CANCEL 可取消"通知。这让用户有机会在系统自动干预前介入，避免真实场景被误判破坏。

**第三道：全量回滚能力**。所有干预动作前必须保存完整的终端 snapshot（包括滚动历史、当前命令、环境变量），干预失败时能一键回滚到干预前状态。这个 snapshot 存储在 `runs/{task-id}/pre-intervention-{timestamp}.json`，保留 7 天。
