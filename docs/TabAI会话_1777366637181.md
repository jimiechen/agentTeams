## **评审意见**

### **问题一：压缩比设置（24:1 → 11:1）是否合理？**

压缩比本身合理，但文档里有一个**逻辑矛盾**需要修正。

Layer 0 → Layer 1 的 24:1 是基于"15 条记录 × 800 字 = 12000 字 → 500 字"计算的，这个数字会随每日执行量波动——如果某天只有 3 条记录，原始内容只有 2400 字，蒸馏后仍然要求 500 字，反而是"信息稀释"而不是压缩。建议把 Layer 1 的输出改为**动态上限**：原始内容 < 3000 字时上限 200 字，3000\~10000 字时上限 400 字，> 10000 字时上限 500 字。这样压缩比才是真实有效的，而不是固定字数填充。

Layer 1 → Layer 2 的 11:1（7 × 500 字 → 300 字）是合理的，但文档第 6.2 节的 30 天演进表格里有个数字问题：Day 7 的 Layer 2 写的是 400 字，Day 30 反而降回 300 字。这反映了一个真实现象——核心知识会先增长后收敛，但表格里 Day 14 是 450 字、Day 30 是 300 字，说明 Layer 2 有字数上限约束。这个约束应该在架构描述里明确：**Layer 2 设置硬上限 300 字，超出时强制淘汰优先级最低的条目**，而不是让它自然收缩，否则合并脚本的行为不可预测。

### **问题二：工作区完全隔离 vs 允许共享 common 知识？**

**建议：隔离为主，共享为辅，但 shared 层的优先级要明确定义。**

文档里的 `shared/wiki/core/common-patterns.md` 设计是对的，但缺少一个关键决策：**当 PMCLI 和 DEVCLI 的私有知识与 shared 知识冲突时，谁优先？**

实践中最常见的冲突场景是：shared 层记录了"删除弹窗选择器为 `.icd-delete-files-command-card-v2-actions-delete`"，但 PMCLI 私有层记录了"PMCLI 场景下该弹窗不出现，因为保守策略不触发删除操作"。如果 ContextBuilder 直接拼接两层，AI 会收到矛盾信息。

建议的优先级规则：**私有 Layer 2 > shared > 私有 Layer 1**。即工作区自己的核心知识优先级最高，shared 作为兜底补充，最近的私有每日记忆优先于 shared 但低于私有核心。同时，shared 层的更新应该有**门槛**：只有在两个工作区的 Layer 1 里连续 3 天都出现相同模式，才晋升到 shared 层，避免把工作区特有的知识错误地泛化。

### **问题三：五维度提炼是否覆盖所有关键信息类型？**

五维度设计（新增知识、失败模式、用户偏好、协议变更、待办）覆盖了大部分场景，但**缺少一个关键维度：性能基线**。

你们的执行记录里有 `switchTask: 150ms`、`waitResponse: 28500ms` 这类结构化指标，这些是判断"今天系统是否变慢了"的重要信号。如果某天 `waitResponse` 突然从 28s 涨到 45s，这是一个需要记录的事实，但它既不是"新增知识"，也不是"失败模式"（任务成功了），更不是"协议变更"。

建议把五维度扩展为**六维度**，增加"性能异常"：记录今天的关键指标均值和峰值，以及与历史基线的偏差。这一条在 Layer 1 里可能只有一行，但在 Layer 2 合并时能帮助识别"系统在某个时间段后持续变慢"这类趋势性问题。

另外，"待办"这个维度有一个设计风险：它是**跨日的状态**，但 Layer 1 是每日独立文件，如果今天的待办明天没有处理，后天蒸馏时就消失了。建议把"待办"从 Layer 1 的蒸馏维度里移出，改为独立的 `wiki/todo.md` 文件，由蒸馏脚本在生成 Layer 1 时同步更新这个文件，并在 Layer 2 合并时检查是否有超过 7 天未处理的待办需要升级或淘汰。

### **问题四：Trae Skills + Rules 的实现方式是否最优？**

架构方向是对的，但有两个实现细节需要调整。

**第一个问题**：`wiki-distill` skill 里的 `context.llm.generate()` 调用，在凌晨 2 点的定时任务场景下，如果 Trae IDE 没有打开或者用户没有登录，这个调用会失败。文档没有处理这个边界情况。建议蒸馏脚本不依赖 Trae IDE 的 LLM 接口，而是直接调用底层 API（OpenAI / Claude / 你们自己的模型服务），这样定时任务可以在后台独立运行，不依赖 IDE 状态。如果必须依赖 Trae，则需要增加失败重试机制：蒸馏失败时记录到 `wiki/distill-errors.log`，次日凌晨 2 点同时处理昨天和前天的未蒸馏记录。

**第二个问题**：`wiki-context-inject.md` 里的 `trigger: before_prompt` 这个触发时机，意味着**每次用户发送消息都会读文件**。对于高频场景，这会引入不必要的 I/O 延迟。建议 `ContextBuilder` 实现内存缓存：Layer 2 在进程启动时加载一次，Layer 1 最近 3 天在每日蒸馏完成后刷新一次，缓存失效时间设为当天结束（00:00 重置）。这样 `before_prompt` 触发时实际上是内存读取，延迟可忽略不计。

### **问题五：1800 字注入上限是否合适？**

**1800 字是合理的起点，但需要根据你们的基础 prompt 长度动态调整。**

这个上限的合理性取决于你们的 `fillPrompt` 里已有多少固定内容——性格前缀、对话历史、任务描述。如果基础 prompt 已经有 3000 字，加上 1800 字注入就是 4800 字，对于上下文窗口较小的模型可能已经接近限制。

建议把注入上限改为**相对配置**而不是绝对字数：在 `wiki_config` 里增加 `max_context_ratio: 0.15`，表示 Wiki 注入最多占总 context 窗口的 15%。这样当模型上下文窗口是 32K 时，注入上限自动扩展到 4800 字；当使用更小的模型时，上限自动收缩，不需要手动调整配置。

## **其他需要修正的细节**

**目录路径不一致**：文档第 1.3 节的架构图里写的是 `wiki/daily/`，但第 2.2 节的目录结构里也是 `wiki/daily/`，这两处一致。但第 4.1 节的 Rules 配置里 `layer1_path` 写的是 `./wiki/daily`，而 skill 代码里用的是 `join(workspacePath, 'wiki', 'daily', ...)`，这两者等价，没问题。唯一需要确认的是 `layer2_path` 在配置里写的是 `./wiki/core/knowledge.md`（文件路径），但 skill 代码里用 `join(workspacePath, config.layer2_path)` 直接拼接，如果 config 里的路径包含文件名，拼接后是正确的；如果只是目录名，就会出错。建议在 `wiki_config` 里把 `layer2_path` 和 `layer2_file` 分开定义，避免歧义。

**蒸馏 Prompt 里的模板变量未渲染**：第 3.1 节 `buildDistillPrompt` 里的 prompt 字符串有多处 `${date}`、`${rawContent.substring(0, 8000)}` 这类模板变量，但在文档里显示为空（因为是 Markdown 渲染问题）。实际代码里需要确认这些变量是否正确插值，特别是截断逻辑 `rawContent.substring(0, 8000)` 应该基于 token 数而不是字符数，中文字符的 token 消耗约是英文的 2 倍，8000 字符的中文内容可能超过 4000 tokens。

## **总体评审结论**

这份设计文档可以进入实施阶段，核心架构无需大改。需要在实施前处理的问题按优先级排序：

优先级最高的是**蒸馏脚本的 LLM 依赖解耦**——定时任务不能依赖 IDE 状态，否则第一次凌晨 2 点运行就可能失败，整个自动化闭环就断了。其次是**Layer 2 的硬上限约束**和**待办维度的独立文件化**，这两个如果不处理，Layer 2 的内容质量会在第 3\~4 周出现不可预测的波动。最后是**ContextBuilder 的内存缓存**，这个不影响正确性，但影响生产环境的响应延迟，可以在 Phase 2 自动化阶段再处理。

Phase 1 的三个 Skills 可以直接开始开发，目录结构和配置 schema 设计是合理的，不需要修改。
