## **三、第二层：三类失败场景模拟（90 分钟）**

目的：**用可控 prompt 诱导 Trae 真的进入三类卡死，验证协议的识别 → 恢复 → 验证闭环**。每类测试都有三个环节：诱导失败、观察识别、检查恢复。

### **测试 2.1：终端卡死（类型 A）**

**诱导 prompt**：
```
@DEVCLI 请在终端执行 `sleep 120` 命令，测试长时间运行的场景
```

或者更真实的：
```
@DEVCLI 帮我在当前项目目录跑一下 npm install，我想看看依赖安装过程
```

**预期观察**：
- 终端开始执行，AI 在 Chat 区输出终端内容
- 约 8 秒后（DEV 策略阈值），底部出现"后台运行 / 取消"按钮
- `hasTerminalBtn: true` 采到
- `mvp:probe` 打印诊断：`diagnoseStuck: terminal-hang`
- `mvp:recover` 打印恢复动作：`recoverTerminalHang action=background`
- 约 1 秒后按钮消失，`hasTerminalBtn: false`

**PASS 标准**：
- 识别延迟 < 10 秒（DEV 阈值 8s + 1 轮心跳 2s）
- 恢复动作成功点击"后台运行"
- 恢复后任务未被终止，最终有响应返回
- `metrics.jsonl` 写入一条 `stuck_kind='terminal-hang', recovery_action='bg_run', recovery_ok=true`

**FAIL 排查**：
- 如果识别延迟 >15s：阈值偏大或心跳间隔过长
- 如果恢复动作点击失败：按钮选择器可能变了，检查 `.icd-btn.icd-btn-tertiary`
- 如果恢复后任务被终止：点的是"取消"不是"后台运行"，检查 textContent 匹配

### **测试 2.3：模型停滞（类型 C）**

这个最难人为构造，因为模型停滞是客观网络/API 问题。有三种诱导方式：

**方式一：超大输出诱导自然停滞**
```
@DEVCLI 写一个完整的 Redis 客户端 SDK，支持 string/hash/list/set/zset 所有基础命令，每个命令都要有 TypeScript 类型定义、单元测试、JSDoc 注释、错误处理。
```
这类请求 token 数巨大，中间经常有自然停顿 > 30s。

**方式二：断网模拟**
开启 AI 回复后立刻断开本机网络 30 秒，Trae 会停留在已有输出状态。

**方式三：直接在 Console 强制冻结**
```javascript
// 冻结 fetch，模拟 Trae 内部网络阻塞
const _fetch = window.fetch;
window.fetch = () => new Promise(() => {});  // 永不 resolve
setTimeout(() => { window.fetch = _fetch; }, 45000);  // 45 秒后恢复
```

**预期观察**：
- AI 已经输出了一部分文本，`lastTurnTextLen` 稳定在某个值
- `btnFunction` 仍为 `stop`
- 30 秒内 textLen 不变、taskStatus 仍为 running
- 诊断打印：`diagnoseStuck: model-stalled, stalledSeconds=30`
- 恢复动作：`recoverModelStalled` 点击停止按钮
- 按钮变 `send`，任务标记 `stalled`
- 按 `max_retries=1` 自动重发同一 prompt 一次
- 如果重发成功，最终返回响应（带"**重试后结果**"前缀标记）

**PASS 标准**：
- 识别延迟 30s~35s（阈值 + 1 轮心跳）
- 恢复动作成功停止生成
- 重试机制触发一次
- `metrics.jsonl` 写入 `stuck_kind='model-stalled', retry_count=1`

**FAIL 排查**：
- 误判短暂停顿为停滞：阈值调到 45s 或改用滑动窗口方差判定
- 停止后按钮没变：可能 Trae 本身停止不生效，改用 `Page.reload()` 兜底

## **六、测试脚本自动化**

如果想把测试 1.1~1.5 做成自动化，别每次都手工在 Console 敲，写一个 `scripts/test-probe-signals.ts`：

```typescript
import { CDPClient } from '../src/cdp/client.js';
import { captureSnapshot } from '../src/actions/state-probe.js';
import { loadConfig } from '../src/config.js';

async function main() {
  const cfg = loadConfig();
  const cdp = new CDPClient(cfg.cdp.host, cfg.cdp.port);
  await cdp.connect();

  const cases = [
    {
      name: 'hasTerminalBtn',
      inject: `
        const el = document.createElement('button');
        el.className = 'icd-btn icd-btn-tertiary';
        el.textContent = '后台运行';
        el.id = '__test_terminal_btn';
        document.body.appendChild(el);
      `,
      cleanup: `document.getElementById('__test_terminal_btn')?.remove();`,
      expect: (s: any) => s.hasTerminalBtn === true,
    },
    {
      name: 'hasDeleteCard',
      inject: `
        const el = document.createElement('button');
        el.className = 'icd-delete-files-command-card-v2-actions-delete';
        el.id = '__test_delete_card';
        document.body.appendChild(el);
      `,
      cleanup: `document.getElementById('__test_delete_card')?.remove();`,
      expect: (s: any) => s.hasDeleteCard === true,
    },
  ];

  for (const c of cases) {
    console.log(`
=== testing  ===`);
    await cdp.evaluate(c.inject);
    await sleep(500);
    const snap = await captureSnapshot(cdp);
    const ok = c.expect(snap);
    console.log(ok ? '✅ PASS' : '❌ FAIL', snap);
    await cdp.evaluate(c.cleanup);
    await sleep(500);
    const snapAfter = await captureSnapshot(cdp);
    console.log('cleanup check:', !c.expect(snapAfter) ? '✅' : '❌', snapAfter);
  }

  await cdp.dispose();
}

function sleep(ms: number) { return new Promise(r => setTimeout(r, ms)); }

main().catch(console.error);
```

运行 `DEBUG=mvp:probe npx tsx scripts/test-probe-signals.ts`，一次性验证所有注入型用例。模型停滞和真实 prompt 类测试没法自动化，只能手工做。
