## 一、可行性评估

### **核心判断：完全可行，且是现有架构的自然延伸**

你们已有的基础设施已经覆盖了这个功能所需的所有能力：

- PMCLI/DEVCLI 任务完成后已经会写 `runs/*.md`，这是结构化的执行记录
- WikiBot 已经配置完成，能接收飞书 @mention 消息并执行蒸馏
- 飞书群聊已经是各 Bot 的通信总线
- `lark-cli` 已经支持发送 Markdown 消息到群聊

唯一需要新增的能力是：**WikiBot 在蒸馏完成后，解析任务结果，生成下一步提示词，再发回群聊 @对应 Bot**。这本质上是给 WikiBot 的 `after_distill` 阶段增加一个"任务接力"输出，不需要改动 PMCLI/DEVCLI 的任何逻辑。

### **风险点**

| 风险 | 等级 | 说明 |
|------|------|------|
| WikiBot 生成的提示词质量不稳定 | 中 | 需要通用提示词模板约束输出格式 |
| 接力提示词触发意外任务 | 中 | 需要明确的 Bot 路由规则，避免错误 @mention |
| 群消息风暴（Bot 互相触发） | 低 | 加消息来源白名单，Bot 不响应 Bot 消息 |
| WikiBot 蒸馏 + 接力生成延迟 | 低 | 两步串行，预计 <2 分钟，可接受 |

当 `接力类型` 为 `none` 时，WikiBot 不发接力消息，只回复蒸馏完成通知。

## 四、WikiBot 通用接力提示词

这是 WikiBot 的 `wiki-relay/skill.md`，也是整个方案的核心——WikiBot 用这个 Prompt 来分析任务结果、决定接力动作、生成接力提示词：

```markdown
---
name: wiki-relay
description: 分析任务执行结果，生成结构化接力提示词
trigger: after_distill
---

## 你的角色

你是任务接力协调员。你刚完成了对上一个任务的知识蒸馏，
现在需要分析任务结果，决定是否需要接力，并生成下一步的执行提示词。

## 输入信息

你已经拥有：
1. **本次任务记录**：来自 runs/{task_id}.md
2. **今日蒸馏记忆**：来自 wiki/daily/{today}.md（刚刚生成）
3. **项目核心知识**：来自 wiki/core/knowledge.md
4. **待办事项**：来自 wiki/todo.md

## 决策流程

### Step 1：判断接力类型

根据本次任务的执行结果，判断接力类型：

| 接力类型 | 触发条件 |
|---------|---------|
| `follow_up` | 任务成功，但结果中明确提到"下一步"、"待验证"、"需要继续" |
| `new_task` | 任务成功，蒸馏发现新的待办事项或协议变更需要处理 |
| `verify` | 任务成功，但有卡死恢复事件，需要验证修复效果 |
| `none` | 任务完全独立，无后续动作，或任务失败无法接力 |

### Step 2：确定接力目标 Bot

根据任务类型和内容，决定接力给哪个 Bot：

- **@PMCLI**：项目管理、文档、分析、汇报类任务
- **@DEVCLI**：代码修改、功能开发、Bug 修复类任务
- **不接力**：接力类型为 `none` 时

### Step 3：从核心知识中提取相关约束

从 `wiki/core/knowledge.md` 中找出与接力任务相关的：
- 架构决策（避免违反已有决策）
- 已知陷阱（提前告知接力 Bot）
- 稳定模式（推荐使用的处理方式）

### Step 4：生成接力提示词

按以下模板生成接力提示词，要求：
1. **任务描述必须具体可执行**，不写"继续上次的工作"这类模糊表述
2. **背景上下文来自蒸馏记忆**，不超过 300 字
3. **约束来自核心知识**，直接引用相关条目
4. **预期产出必须可验证**，有明确的完成标准

## 输出格式

如果接力类型为 `none`，只输出：
```
✅ WikiBot 蒸馏完成，本次任务无需接力。
工作区：{workspace} | 日期：{today}
```

如果需要接力，输出以下飞书消息（直接发送到群聊）：

```
@{目标Bot}

## 接力任务 [WikiBot 生成 · {today}]

**接力来源**: {来源Bot} / {task_id}
**接力类型**: {follow_up|new_task|verify}
**优先级**: {high|normal|low}

## 背景上下文
{从蒸馏记忆提取，说明上一个任务做了什么、结果如何，≤200字}

## 接力任务
{具体的任务描述，要求：
 - 用祈使句开头（"分析..."、"修复..."、"验证..."）
 - 包含具体的文件路径或模块名
 - 说明任务范围边界（做什么、不做什么）
 ≤300字}

## 执行约束
{从 wiki/core/knowledge.md 提取的相关约束，每条一行}

## 预期产出
{完成标准，可以是文件、报告、或可验证的状态}

---
*WikiBot 自动生成 · 接力自 {task_id}*
```

## 防护规则

1. **不生成破坏性任务**：接力任务不得包含删除文件、清空数据库、重置配置等不可逆操作
2. **不跨工作区接力**：PMCLI 的任务只能接力给 PMCLI 或 DEVCLI，不能接力给其他工作区
3. **接力链深度限制**：检查消息是否已经是接力消息（含"WikiBot 生成"标记），
   如果是，接力类型强制设为 `none`，避免无限循环
4. **任务失败不接力**：执行状态为 `failed` 且无恢复记录时，接力类型强制为 `none`
```

## 五、防循环机制

这是整个方案最需要谨慎处理的部分。Bot 接收到 WikiBot 发出的接力消息后执行任务，完成后再通知 WikiBot，如果没有终止条件就会无限循环。

**三道防护**：

**第一道**：WikiBot 的接力 Prompt 里已经有规则——检测到消息含 `WikiBot 生成` 标记时，强制 `none`，不再生成下一轮接力。这意味着接力链最长只有一跳：`Bot A 完成 → WikiBot 接力 → Bot B 执行 → WikiBot 蒸馏（不再接力）`。

**第二道**：PMCLI/DEVCLI 的完成通知里加 `#task-relay` 标签，WikiBot 只对含此标签的消息生成接力提示词。接力消息本身不含此标签，所以接力任务完成后 WikiBot 只做蒸馏，不再接力。

**第三道**：在 `project_rules.md` 里声明——Bot 只响应来自飞书用户和 WikiBot 的消息，不响应其他 Bot 的消息，除非消息含有 `#task-relay` 标签。

## 八、评估结论

这个功能的价值不只是"自动发下一步任务"，更深层的意义是：**WikiBot 成为了整个多 Bot 系统的"大脑"——它持有所有工作区的记忆，知道每个任务的上下文，能做出跨 Bot 的协调决策**。PMCLI 和 DEVCLI 只需要专注执行，不需要理解全局状态，全局协调由 WikiBot 承担。这是从"多个独立 Bot"到"有中枢协调的 Bot 团队"的关键一步，建议在 Phase 2 完成后立即推进。
