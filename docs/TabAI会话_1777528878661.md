## 一、方案定位

### 1.1 解决的问题

当前心跳检测系统存在一个关键盲区：`background` 模式既不判断失败，也不触发自动恢复。当 DEVCLI/PMCLI 任务长时间处于"后台运行"状态但实际已经卡死时，系统无法主动介入，导致任务永久挂起。

### 1.2 方案本质

**用 Trae UI 已有的两个按钮（"后台运行" + "取消"）把未知的卡死状态转换为已知的可恢复状态**，不做复杂的存活探测，不做进程级干预，完全通过 UI 层操作完成恢复。

### 1.3 为什么选择简化方案

相比前期讨论的"分层诊断 + 主动探活 + 有限升级"完整方案（详见 `HEARTBEAT_OPTIMIZATION_GUIDE.md`），本方案有三个核心优势：

- **工程量小**：核心代码不超过 100 行，1 天可完成实现
- **风险可控**：只操作 UI 层按钮，不涉及进程管理、不发送系统信号
- **完美复用**：取消后的任务自动进入 `interrupted` 状态，直接走现有恢复链路

## 二、核心逻辑

### 2.1 三步恢复流程

```
[background 状态]
      ↓
  超时判定（5 分钟无 DOM 变化）
      ↓
  点击"后台运行"按钮 → UI 释放可操作状态
      ↓
  等待 2 秒（让 UI 过渡）
      ↓
  点击"取消"按钮 → 任务进入 interrupted
      ↓
  复用现有 executeFrozenRecovery（切任务 → 找重试 → 点击）
      ↓
  任务重新运行
```

### 2.2 状态转换

```
normal
  └─→ background（检测到后台运行按钮）
        ├─→ normal（DOM 重新活跃）
        ├─→ interrupted（触发超时恢复后）
        └─→ human-required（熔断后升级）

interrupted
  └─→ normal（复用现有 frozen 恢复路径）
```

不新增 `suspicious`、`probing` 等中间状态，保持状态机简洁。

## 三、关键参数

| 参数 | 推荐值 | 说明 |
|------|--------|------|
| 超时阈值 | 5 分钟 | DOM 无变化超过此时长触发恢复 |
| DOM 采样频率 | 10 秒 | 定时采样而非 MutationObserver，降低复杂度 |
| 两步点击间隔 | 2 秒 | 等待 UI 过渡到后台状态 |
| 单任务最大触发次数 | 2 次 | 熔断机制，超过则人工介入 |
| 恢复流程 Cooldown | 60 秒 | 防止连续触发 |

### 3.1 参数调整原则

**超时阈值**需要根据实际数据调整。初始 5 分钟是保守值，若观察到：
- 正常长任务（如 `npm install`）被误伤 → 延长至 10 分钟
- 卡死任务恢复不及时 → 缩短至 3 分钟
- 不同任务类型表现差异大 → 按任务类型差异化配置

// state-machine.ts
interface BackgroundStateContext {
  enteredAt: number;
  lastActivityAt: number;
  lastDomSnapshot: string;
  triggerCount: number;  // 熔断计数
}

// detector.ts
private async checkBackgroundTimeout(
  cdp: CDPClient,
  ctx: BackgroundStateContext
): Promise<{ timeout: boolean; silentDuration: number }> {
  // 采样当前 DOM 快照
  const snapshot = await cdp.evaluate(`
    (() => {
      const term = document.querySelector('.terminal, .chat-container');
      if (!term) return '';
      const tail = term.textContent?.slice(-500) ?? '';
      return \`\${term.scrollHeight}-\${tail.length}-\${hashString(tail)}\`;
    })()
  `);
  
  // 对比快照，变化则重置计时器
  if (snapshot !== ctx.lastDomSnapshot) {
    ctx.lastActivityAt = Date.now();
    ctx.lastDomSnapshot = snapshot;
    return { timeout: false, silentDuration: 0 };
  }
  
  const silentDuration = Date.now() - ctx.lastActivityAt;
  return {
    timeout: silentDuration > 5 * 60 * 1000,
    silentDuration,
  };
}
```

### 4.3 RecoveryExecutor 层：两步恢复

```typescript
// recovery-executor.ts
private async executeBackgroundTimeout(
  ctx: BackgroundStateContext
): Promise<RecoveryResult> {
  const taskId = this.getCurrentTaskId();
  
  // 熔断检查
  if (ctx.triggerCount >= 2) {
    await this.notifyHuman(`任务 ${taskId} background 超时恢复已达 2 次上限`);
    return { success: false, reason: 'human-required' };
  }
  
  // Cooldown 检查
  if (this.isInCooldown('background-timeout')) {
    return { success: false, reason: 'cooldown' };
  }
  
  // Step 1: 点击"后台运行"按钮
  const bgClicked = await this.cdp.evaluate(`
    (() => {
      const btns = Array.from(document.querySelectorAll('.icd-btn.icd-btn-tertiary'));
      const bgBtn = btns.find(b => b.textContent?.includes('后台运行'));
      if (bgBtn) { bgBtn.click(); return true; }
      return false;
    })()
  `);
  
  if (!bgClicked) {
    debug('❌ 后台运行按钮未找到');
    return { success: false, reason: 'bg-btn-not-found' };
  }
  
  await this.delay(2000);
  
  // 最后确认：检查 DOM 是否突然活跃
  const currentSnapshot = await this.computeDomSnapshot();
  if (currentSnapshot !== ctx.lastDomSnapshot) {
    debug('✅ 任务在 2 秒等待期内恢复活跃，跳过取消');
    return { success: true, reason: 'self-recovered' };
  }
  
  // Step 2: 点击"取消"按钮
  const cancelClicked = await this.cdp.evaluate(`
    (() => {
      const btns = Array.from(document.querySelectorAll('button'));
      const cancelBtn = btns.find(b => 
        b.textContent?.trim() === '取消' && 
        b.offsetParent !== null  // 可见性检查
      );
      if (cancelBtn) { cancelBtn.click(); return true; }
      return false;
    })()
  `);
  
  if (!cancelClicked) {
    debug('❌ 取消按钮未找到');
    return { success: false, reason: 'cancel-btn-not-found' };
  }
  
  // 记录熔断计数和 Cooldown
  ctx.triggerCount += 1;
  this.setCooldown('background-timeout', 60 * 1000);
  
  // 飞书通知
  await this.notifyFeishu({
    type: 'background-timeout-recovered',
    taskId,
    silentDuration: Date.now() - ctx.lastActivityAt,
    triggerCount: ctx.triggerCount,
  });
  
  debug(`✅ 后台超时恢复成功: 任务 ${taskId}，第 ${ctx.triggerCount} 次`);
  return { success: true, method: 'bg-then-cancel' };
}
```

### 4.4 状态衔接

取消后任务自动进入 `interrupted` 状态，由现有 `executeFrozenRecovery` 处理，**无需新增任何恢复逻辑**。这是方案最优雅的部分——新增代码只负责"翻译状态"，恢复动作复用已有能力。

## 五、边界保护

### 5.1 三道必须的保护

**保护一：按钮存在性校验**。必须在 DOM 里真实检测到"后台运行"按钮才启动计时。若按钮不存在，说明任务未进入后台模式，计时无意义。通过现有 `determineMode` 里的 `hasBackgroundBtn` 判断已满足，但需确保状态切出时清零计时器。

**保护二：两秒等待期的"最后确认"**。点击"后台运行"后等待 2 秒期间，若 DOM 突然变化，说明任务自行恢复，立即中止后续"取消"动作。这能避免在"正好超时那一刻恢复"的任务上误伤。实现见 4.3 节的 `currentSnapshot` 对比逻辑。

**保护三：飞书通知留痕**。每次触发"后台运行 + 取消"恢复必须发飞书信息，包含：
- 任务 ID
- 静默时长
- 触发次数（第几次）
- 恢复结果（成功/失败及原因）

这是事后复盘的唯一依据，不可省略。

### 5.2 与速率限制的协调

现有系统对 `refresh-page` 有速率限制（5 次/3600 秒）。新增的 `background-timeout` 恢复动作**不占用 refresh-page 配额**，因为它不刷新页面，只点击两个按钮。但应独立纳入速率限制体系，建议配置为 **3 次/3600 秒**。

### 5.3 并发控制

必须与现有互斥锁（recoveryLock）共享。`executeBackgroundTimeout` 触发前必须获取锁，触发后必须释放锁，避免与 `executeFrozenRecovery` 并发冲突。
